ES Correlation Searches

Correlation searches are the detection engine of Splunk Enterprise Security. They run scheduled SPL queries against your data to identify threats and create notable events.

Choosing the Right Approach

What type of detection are you building?
Single Event Pattern

→ Standard Correlation Search

Detect specific events like malware alerts or policy violations.

Aggregation/Threshold

→ Stats-based Correlation

Detect patterns like brute force (>10 failures in 5 min).

How do you want to handle risk?
Immediate Alert Needed

→ Notable Event Output

Creates incident for SOC investigation immediately.

Contribute to Risk Score

→ Risk-Based Alerting

Add risk points; alert when threshold exceeded.

Anatomy of a Correlation Search

Search Configuration
Basic search settings
  • Name: Descriptive search name
  • App: Usually SA-ThreatIntelligence or SplunkEnterpriseSecuritySuite
  • Schedule: Cron expression (e.g., */5 * * * *)
  • Time Range: Earliest/latest relative to scheduled time
SPL Query
Detection logic
  • Uses data model or raw search
  • Filters for threat indicators
  • Aggregates if needed (stats, timechart)
  • Returns events to create notables
Notable Event Fields
Fields to include in notable
  • rule_name: Name of detection
  • rule_title: Human-readable title
  • security_domain: Access, Endpoint, Network, etc.
  • severity: critical, high, medium, low, informational
Adaptive Response
Automated actions
  • Create Notable: Primary action for ES
  • Risk Modifier: Add risk score to entity
  • Send Email: Alert notification
  • Run Script: Custom automation

Correlation Search Examples

Example 1: Brute Force Attack Detection

Authentication - Brute Force
| `datamodel("Authentication", "Authentication")`
| search action=failure
| stats 
    count as failure_count,
    dc(user) as unique_users,
    values(user) as targeted_users,
    earliest(_time) as firstTime,
    latest(_time) as lastTime
  by src
| where failure_count > 10
| eval attack_duration = lastTime - firstTime
| eval rule_name = "Brute Force Attack Detected"
| eval rule_title = "Brute Force from " . src
| eval security_domain = "access"
| eval severity = case(
    failure_count >= 100, "critical",
    failure_count >= 50, "high",
    failure_count >= 20, "medium",
    true(), "low"
)

Example 2: Malware Detection

Endpoint - Malware Alert
| `datamodel("Malware", "Malware_Attacks")`
| search action!=allowed
| eval rule_name = "Malware Detected"
| eval rule_title = signature . " on " . dest
| eval security_domain = "endpoint"
| eval severity = case(
    match(category, "(?i)ransomware"), "critical",
    match(category, "(?i)trojan|backdoor"), "high",
    match(category, "(?i)adware|pup"), "low",
    true(), "medium"
)
| table _time, dest, user, signature, file_name, file_hash, 
       action, severity, rule_name, rule_title, security_domain

Example 3: Data Exfiltration Detection

Network - Large Data Transfer
| `datamodel("Network_Traffic", "All_Traffic")`
| stats 
    sum(bytes_out) as total_bytes_out,
    dc(dest) as unique_destinations,
    values(dest) as destinations
  by src, user
| eval total_GB = round(total_bytes_out/1024/1024/1024, 2)
| where total_GB > 5 // More than 5GB outbound
| eval rule_name = "Large Outbound Data Transfer"
| eval rule_title = user . " transferred " . total_GB . "GB"
| eval security_domain = "network"
| eval severity = case(
    total_GB >= 50, "critical",
    total_GB >= 20, "high",
    total_GB >= 10, "medium",
    true(), "low"
)

Using CIM Data Models

Why Use Data Models?

Data models provide normalized field names across vendors, enabling:

  • Portability: Same search works for any vendor
  • Performance: Accelerated data models are faster
  • Consistency: Standard fields like src, dest, user
  • Built-in content: ES rules use data models

Authenticationuser, src, dest, action, appLogin success/failure
Network_Trafficsrc, dest, src_port, dest_port, bytesFirewall, flow data

Malwaredest, file_name, file_hash, signatureAV alerts

Data Model Query Syntax
// Using datamodel command (standard)
| `datamodel("Authentication", "Authentication")`
| search action=failure

// Using tstats for accelerated models (faster)
| tstats count from datamodel=Authentication.Authentication 
  where Authentication.action=failure 
  by Authentication.src, Authentication.user

// Converting tstats field names
| rename Authentication.* as *

Interview Questions

Correlation Search:

  • Part of Enterprise Security app
  • Creates notable events (ES incidents)
  • Supports adaptive response actions
  • Integrates with ES incident workflow
  • Has security-specific fields (security_domain, severity)

Regular Saved Search:

  • Core Splunk feature
  • Creates alerts (not notable events)
  • Basic actions (email, webhook)
  • No ES integration

Tuning strategies:

  • Adjust threshold: Increase count/value thresholds
  • Add exclusions: NOT src IN (known_good_ips)
  • Whitelist lookups: | lookup whitelist src OUTPUT is_whitelist | where isnull(is_whitelist)
  • Time-based: Exclude maintenance windows
  • Add context: Require additional conditions
  • Throttling: Limit alerts per entity per time
