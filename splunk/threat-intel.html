Threat Intelligence Framework

Complete guide to the Splunk Enterprise Security Threat Intelligence Framework. Configure feeds, manage IOCs, and create threat-matched detections.

TI Framework Overview

The ES Threat Intelligence Framework:

  • Ingests threat feeds from multiple sources (TAXII, STIX, CSV, lookup)
  • Normalizes indicators into KV Store collections
  • Correlates IOCs against incoming events in real-time
  • Generates notable events when matches occur

Supported IOC Types

IP Address Domain File Hash URL Email Certificate Registry

Configure Threat Feeds

Add TAXII Feed

Enterprise Security → Configure → Data Enrichment → Threat Intelligence Downloads
1
Click "New" to add new threat feed
2
Configure TAXII 2.x feed:
Type:taxii2
Name:AlienVault_OTX
URL:https://otx.alienvault.com/taxii/root/
Collection:[Select collection]
Username:[API key as username]
Interval:3600 (hourly)
Weight:2 (confidence level)
3
Save and enable the feed

Add CSV/Lookup Feed

For local IOC lists or custom feeds:

# Create CSV file with required fields # $SPLUNK_HOME/etc/apps/SA-ThreatIntelligence/lookups/local_iocs.csv ip,description,threat_key,weight 192.168.100.50,Known C2 server,apt_group_x,3 10.0.0.99,Suspected compromised host,internal_threat,2 203.0.113.50,Botnet controller,botnet_mirai,3

Configure in local/inputs.conf:

[threatlist://local_iocs] type = lookup description = Local IOC List url = lookup://local_iocs interval = 86400 weight = 3 fields = ip:ip,description:description,threat_key:threat_key

Query IOC Collections

# View all IP IOCs | inputlookup ip_intel | table ip, description, threat_key, weight, _time # Search for specific IOC | inputlookup ip_intel where ip="192.168.100.50" # Count IOCs by source | inputlookup ip_intel | stats count by threat_key | sort - count

Threat Matching & Correlation

How Matching Works

  1. Events ingested with CIM-compliant field names (src_ip, dest_ip, url, file_hash)
  2. Threat Gen searches compare events against KV Store collections
  3. Matches generate "Threat Activity Detected" notable events
  4. Risk score added to associated assets/identities

Threat Gen Correlation Search

Enterprise Security → Configure → Content → Content Management → Threat - Threat Activity Detected
# Default Threat Activity Detected search (simplified) | tstats summariesonly=true count from datamodel=Network_Traffic by All_Traffic.src_ip, All_Traffic.dest_ip | rename All_Traffic.* as * | lookup ip_intel ip as src_ip OUTPUT threat_key as src_threat | lookup ip_intel ip as dest_ip OUTPUT threat_key as dest_threat | where isnotnull(src_threat) OR isnotnull(dest_threat)

Manual IOC Hunting

# Hunt for IP IOC matches in firewall logs index=firewall | lookup ip_intel ip as src_ip OUTPUT threat_key, description, weight | where isnotnull(threat_key) | table _time, src_ip, dest_ip, threat_key, description, weight # Hunt for domain IOC matches in proxy logs index=proxy | lookup domain_intel domain as url_domain OUTPUT threat_key, description | where isnotnull(threat_key) | table _time, src_ip, url, url_domain, threat_key # Hunt for file hash matches in endpoint logs index=endpoint sourcetype=*endpoint* | lookup file_intel file_hash as hash OUTPUT threat_key, file_name as ti_filename | where isnotnull(threat_key) | table _time, host, file_path, hash, threat_key, ti_filename

Managing IOCs

Add IOC Manually

Enterprise Security → Configure → Data Enrichment → Threat Intelligence Management
1
Select IOC type (IP, Domain, File Hash, etc.)
2
Click "New" and enter IOC details
3
Set threat_key (campaign/source name), weight, expiration

Add via SPL

# Add IP IOC via outputlookup | makeresults | eval ip="203.0.113.100", description="New C2 server", threat_key="incident_2024_001", weight=3 | outputlookup append=true ip_intel

Remove/Expire IOCs

# Remove specific IOC | inputlookup ip_intel | where ip!="192.168.100.50" | outputlookup ip_intel # Expire old IOCs (keep last 90 days) | inputlookup ip_intel | where _time > relative_time(now(), "-90d") | outputlookup ip_intel

Tuning & Best Practices

Weight/Confidence Scoring

WeightConfidenceUse For
1LowUnvetted community feeds, informational
2MediumReputable open feeds, older IOCs
3HighCommercial feeds, recent campaigns
4CriticalConfirmed active threats, incident-specific

Reducing False Positives

  • Whitelist: Add legitimate IPs/domains to threat_whitelist lookup
  • Weight threshold: Only alert on weight >= 2
  • Expiration: Auto-expire IOCs older than 90 days
  • Source quality: Prioritize commercial/vetted feeds

Interview Questions

  1. Ingest: Configure threat feeds (TAXII, CSV, lookup) in Data Enrichment settings
  2. Normalize: IOCs stored in KV Store collections (ip_intel, domain_intel, file_intel, etc.)
  3. Correlate: Threat Gen searches compare CIM-normalized events against IOC collections
  4. Alert: Matches generate "Threat Activity Detected" notable events
  5. Enrich: Risk scores added to assets/identities for Risk-Based Alerting

Key benefit: Automated correlation without writing custom searches for each feed.

  • Whitelist: Add known-good IPs/domains to threat_whitelist lookup
  • Weight threshold: Filter alerts to only high-confidence (weight >= 3) matches
  • Feed quality: Prioritize commercial/vetted feeds over unvetted community sources
  • Expiration: Auto-expire IOCs after 90 days (stale IOCs cause FPs)
  • Contextual tuning: Suppress matches on internal scanning tools, CDNs, etc.

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.

Comprehensive Troubleshooting

Systematic Troubleshooting Methodology

  1. Define the Problem: Clearly articulate what is not working as expected. Gather specific error messages, timestamps, and affected components.
  2. Gather Information: Collect relevant logs, metrics, and configuration details. Document recent changes that might be related.
  3. Analyze Data: Review collected information for patterns and anomalies. Compare against baseline behavior and known-good configurations.
  4. Develop Hypothesis: Form theories about potential root causes. Prioritize based on likelihood and impact.
  5. Test Solutions: Implement fixes in a controlled manner. Test one change at a time to isolate the effective solution.
  6. Document Resolution: Record the root cause and solution. Update runbooks and knowledge base for future reference.
