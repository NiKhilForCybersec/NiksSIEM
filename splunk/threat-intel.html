Threat Intelligence Framework

Complete guide to the Splunk Enterprise Security Threat Intelligence Framework. Configure feeds, manage IOCs, and create threat-matched detections.

TI Framework Overview

The ES Threat Intelligence Framework:

  • Ingests threat feeds from multiple sources (TAXII, STIX, CSV, lookup)
  • Normalizes indicators into KV Store collections
  • Correlates IOCs against incoming events in real-time
  • Generates notable events when matches occur

Supported IOC Types

IP Address Domain File Hash URL Email Certificate Registry

For local IOC lists or custom feeds:

# Create CSV file with required fields # $SPLUNK_HOME/etc/apps/SA-ThreatIntelligence/lookups/local_iocs.csv ip,description,threat_key,weight 192.168.100.50,Known C2 server,apt_group_x,3 10.0.0.99,Suspected compromised host,internal_threat,2 203.0.113.50,Botnet controller,botnet_mirai,3

Configure in local/inputs.conf:

[threatlist://local_iocs] type = lookup description = Local IOC List url = lookup://local_iocs interval = 86400 weight = 3 fields = ip:ip,description:description,threat_key:threat_key

# View all IP IOCs | inputlookup ip_intel | table ip, description, threat_key, weight, _time # Search for specific IOC | inputlookup ip_intel where ip="192.168.100.50" # Count IOCs by source | inputlookup ip_intel | stats count by threat_key | sort - count

Threat Matching & Correlation

How Matching Works

  1. Events ingested with CIM-compliant field names (src_ip, dest_ip, url, file_hash)
  2. Threat Gen searches compare events against KV Store collections
  3. Matches generate "Threat Activity Detected" notable events
  4. Risk score added to associated assets/identities

Threat Gen Correlation Search

Enterprise Security â†’ Configure â†’ Content â†’ Content Management â†’ Threat - Threat Activity Detected
# Default Threat Activity Detected search (simplified) | tstats summariesonly=true count from datamodel=Network_Traffic by All_Traffic.src_ip, All_Traffic.dest_ip | rename All_Traffic.* as * | lookup ip_intel ip as src_ip OUTPUT threat_key as src_threat | lookup ip_intel ip as dest_ip OUTPUT threat_key as dest_threat | where isnotnull(src_threat) OR isnotnull(dest_threat)

Manual IOC Hunting

# Hunt for IP IOC matches in firewall logs index=firewall | lookup ip_intel ip as src_ip OUTPUT threat_key, description, weight | where isnotnull(threat_key) | table _time, src_ip, dest_ip, threat_key, description, weight # Hunt for domain IOC matches in proxy logs index=proxy | lookup domain_intel domain as url_domain OUTPUT threat_key, description | where isnotnull(threat_key) | table _time, src_ip, url, url_domain, threat_key # Hunt for file hash matches in endpoint logs index=endpoint sourcetype=*endpoint* | lookup file_intel file_hash as hash OUTPUT threat_key, file_name as ti_filename | where isnotnull(threat_key) | table _time, host, file_path, hash, threat_key, ti_filename

Tuning & Best Practices

Weight/Confidence Scoring

WeightConfidenceUse For
1LowUnvetted community feeds, informational
2MediumReputable open feeds, older IOCs
3HighCommercial feeds, recent campaigns
4CriticalConfirmed active threats, incident-specific

Reducing False Positives

  • Whitelist: Add legitimate IPs/domains to threat_whitelist lookup
  • Weight threshold: Only alert on weight >= 2
  • Expiration: Auto-expire IOCs older than 90 days
  • Source quality: Prioritize commercial/vetted feeds

What Enterprises Usually Miss

ðŸ”´ Critical Gaps

  • No IOC expiration: Ancient IOCs causing false positives on legitimate CDNs/IPs
  • No whitelist management: Legitimate services (CDNs, cloud providers) triggering alerts
  • Unconfigured threat gen: ES Threat Intel framework installed but not enabled
  • No feed validation: Using feeds without measuring FP/TP rates
  • Missing CIM normalization: Network events not mapped, threat matching fails silently

Operational Gaps

  • No feed health monitoring: Feeds silently fail, no new IOCs for weeks
  • Weight not configured: All IOCs treated equally regardless of confidence
  • No investigation playbook: Analysts don't know how to handle TI matches
  • Silo'd intelligence: TI not integrated with broader detection engineering
  • No feedback loop: Investigation findings not improving IOC quality

Efficiency Issues

  • Too many feeds: Duplicate IOCs across 10+ feeds
  • No deduplication: Same IOC creating multiple alerts
  • Missing context: IOC matches without malware family or actor info
  • No prioritization: All TI alerts same urgency regardless of source
  • Manual processes: IOC updates requiring manual intervention

Best Practices Often Ignored

  • Monthly feed review: Evaluate each feed's FP rate and value
  • IOC lifecycle management: Auto-expire IOCs after 90-180 days
  • Whitelist maintenance: Regular review and updates of known-good entities
  • TI platform integration: Centralize IOC management before pushing to Splunk
  • Threat hunting integration: Use TI for hypothesis-driven hunting, not just alerting

Interview Questions

  1. Ingest: Configure threat feeds (TAXII, CSV, lookup) in Data Enrichment settings
  2. Normalize: IOCs stored in KV Store collections (ip_intel, domain_intel, file_intel, etc.)
  3. Correlate: Threat Gen searches compare CIM-normalized events against IOC collections
  4. Alert: Matches generate "Threat Activity Detected" notable events
  5. Enrich: Risk scores added to assets/identities for Risk-Based Alerting

Key benefit: Automated correlation without writing custom searches for each feed.

  • Whitelist: Add known-good IPs/domains to threat_whitelist lookup
  • Weight threshold: Filter alerts to only high-confidence (weight >= 3) matches
  • Feed quality: Prioritize commercial/vetted feeds over unvetted community sources
  • Expiration: Auto-expire IOCs after 90 days (stale IOCs cause FPs)
  • Contextual tuning: Suppress matches on internal scanning tools, CDNs, etc.

Additional Interview Questions

Systematic troubleshooting approach:

  1. Identify the symptom: What exactly is not working as expected?
  2. Check connectivity: Verify network paths and authentication
  3. Review logs: Check internal/audit logs for errors
  4. Validate configuration: Compare against documented standards
  5. Test incrementally: Isolate the problem component
  6. Document findings: Record what worked and what didn't

Key performance indicators:

  • Availability: Uptime percentage, service availability
  • Performance: Query response time, processing latency
  • Volume: Events per second, data ingestion rate
  • Quality: Parse success rate, enrichment coverage
  • Efficiency: Resource utilization, cost per GB

High availability strategies:

  • Redundancy: Multiple instances of critical components
  • Load balancing: Distribute traffic across instances
  • Failover: Automatic switchover on failure
  • Monitoring: Health checks and alerting
  • DR Planning: Documented recovery procedures
