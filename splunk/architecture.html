Splunk Architecture

Understanding Splunk's distributed architecture is essential for deployment planning, scaling, troubleshooting, and optimizing performance. Learn how forwarders, indexers, and search heads work together.

Choosing Your Deployment Model

What is your expected daily ingestion volume?
< 50 GB/day

→ Single Instance or Splunk Cloud

All-in-one server or cloud deployment. Simplest to manage.

> 50 GB/day

→ Continue to distributed options

Do you require high availability and disaster recovery?
Yes - HA Required

→ Clustered Architecture

Indexer cluster + Search Head cluster. Automatic failover, data replication.

No - Can tolerate downtime

→ Basic Distributed

Separate indexers and search heads without clustering.

Core Components

Forwarders
  • Collect data from sources
  • Universal Forwarder (UF) - lightweight
  • Heavy Forwarder (HF) - can parse
  • Deploy on every endpoint/server
  • Does NOT index data
Indexers
  • Store and index data
  • Process search requests
  • Scale horizontally for volume
  • Can cluster for HA
  • Main storage component
Search Heads
  • User interface (Web UI)
  • Coordinate searches across indexers
  • Run dashboards and alerts
  • Can cluster for availability
  • Does NOT store data
Management
  • Deployment Server - push configs
  • Cluster Manager - manage indexers
  • License Manager - track usage
  • Monitoring Console - health
  • Usually combined on one server

Search Factor (SF) Number of searchable copies (with index files) SF=2 (two copies are searchable)

Why This Architecture?
• RF=3: PCI requires data resilience - 3 copies survives 2 indexer failures
• Multi-site: DR capability across data centers
• SH Cluster: SOC needs 24/7 access, no single point of failure
• Heavy Forwarders: Centralize syslog from network devices, apply parsing

Interview Questions

Replication Factor (RF):

  • How many copies of raw data are stored
  • RF=3 means each bucket exists on 3 different indexers
  • Protects against data loss if indexers fail

Search Factor (SF):

  • How many of those copies are searchable (have index files)
  • SF=2 means 2 copies have .tsidx files built
  • Other copies are raw-only (can be made searchable)

Key relationship: SF must be ≤ RF. Common setting: RF=3, SF=2

Universal Forwarder:

  • Lightweight (~50MB footprint)
  • Collect and forward only - no parsing
  • Deploy on endpoints, servers
  • Best for: Most data collection scenarios

Heavy Forwarder:

  • Full Splunk instance (~1GB+)
  • Can parse, filter, route, transform
  • Can receive syslog directly
  • Best for: Syslog aggregation, data routing, parsing at edge
  • Searches continue: Indexers keep serving searches
  • Ingestion continues: Data still indexed
  • Replication paused: No new bucket replication decisions
  • Recovery blocked: Can't recover failed indexers
  • Config changes blocked: Can't push new cluster configs

Best practice: Set up Cluster Manager HA using external tools like Pacemaker/Corosync for automatic failover.
