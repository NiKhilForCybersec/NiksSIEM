Dashboards & Visualizations

Complete guide to creating dashboards and visualizations in Splunk. Covers Dashboard Studio, Classic Dashboards, and security-specific examples.

Dashboard Types

TypeBest ForFormat
Dashboard Studio Modern, pixel-perfect layouts JSON definition
Classic Dashboard Quick creation, SimpleXML XML definition
ES Security Dashboards Pre-built ES views Built-in

Creating Dashboards

Search → Save As → Dashboard Panel

Method 1: From Search Results

  1. Run your SPL query
  2. Click "Save As" → "Dashboard Panel"
  3. Choose "New Dashboard" or add to existing
  4. Select visualization type

Method 2: Dashboard Studio

Dashboards → Create New Dashboard → Dashboard Studio
  1. Drag visualizations onto canvas
  2. Configure data source (SPL query)
  3. Style and position elements
  4. Add inputs (dropdowns, time picker)

Visualization Types

Single Value

KPI metrics, counts, totals

| stats count

Line/Area Chart

Trends over time

| timechart count

Bar/Column

Comparisons by category

| stats count by field

Pie Chart

Proportions

| stats count by field

Table

Detailed data display

| table field1, field2

Choropleth Map

Geographic data

| geostats count

SOC Dashboard Examples

Sobeys Dashboard Strategy

For a retail enterprise with 1,500+ stores, dashboards should be organized by:

  • Executive: Enterprise-wide KPIs, compliance status, risk posture
  • Regional: Atlantic, Quebec, Ontario, West performance comparison
  • Operational: SOC analyst triage, real-time alerts
  • Loss Prevention: Fraud detection, shrinkage monitoring

<form version="1.1" theme="dark">

  <!-- Row 1: KPIs -->
  <row>
    <panel>
      <title>High-Risk Transactions</title>
      <single>
        <search>

| eval variance = actual_amount - expected_amount
| stats sum(variance) as total_variance
| eval display = "$" + tostring(round(abs(total_variance), 2), "commas")
| eval color = if(total_variance < 0, "red", "green")</query>

  <!-- Row 2: Fraud Trends -->
  <row>
    <panel>
      <title>Fraud Indicators by Type (Trend)</title>
      <chart>
        <search>
          <query>index=pos_transactions risk_score>=50 earliest=$time_range.earliest$ latest=$time_range.latest$
| lookup store_info.csv store_id OUTPUT region, brand
| search region=$region$ brand=$brand$
| eval fraud_type = case(
    transaction_type="VOID" AND payment_method="CASH", "Post-Tender Void",
    transaction_type="RETURN" AND receipt_present="NO", "No-Receipt Return",
    discount_amount > 20, "Excessive Discount",
    override_used="YES", "Override Abuse",
    1=1, "Other"

| timechart span=1d count by fraud_type</query>
        </search>

        <option name="charting.chart.nullValueMode">zero</option>

      <chart>
        <search>
          <query>index=pos_transactions earliest=$time_range.earliest$ latest=$time_range.latest$
| lookup store_info.csv store_id OUTPUT region, brand, store_type
| search region=$region$ brand=$brand$
| stats count(eval(risk_score>=70)) as exceptions, count as total by store_type
| eval exception_rate = round(exceptions / total * 100, 3)
| fields store_type, exception_rate</query>
        </search>
        <option name="charting.chart">bar</option>
      </chart>
    </panel>
  </row>

  <!-- Row 3: High Risk Employees -->
  <row>
    <panel>
      <title>Highest Risk Employees</title>
      <table>
        <search>
          <query>index=pos_transactions earliest=$time_range.earliest$ latest=$time_range.latest$
| lookup store_info.csv store_id OUTPUT region, brand, store_name
| search region=$region$ brand=$brand$
| stats 
    sum(risk_score) as total_risk,
    count as total_transactions,
    sum(eval(if(transaction_type="VOID",1,0))) as voids,
    sum(eval(if(transaction_type="RETURN" AND receipt_present="NO",1,0))) as no_receipt_returns,
    sum(eval(if(override_used="YES",1,0))) as overrides
    by cashier_id, store_id, store_name
| eval void_rate = round(voids / total_transactions * 100, 2)
| eval avg_risk = round(total_risk / total_transactions, 1)
| lookup employee_info.csv employee_id as cashier_id OUTPUT employee_name, hire_date
| eval tenure_days = round((now() - strptime(hire_date, "%Y-%m-%d")) / 86400)
| sort -total_risk
| head 20
| table store_name, cashier_id, employee_name, tenure_days, total_transactions, voids, void_rate, no_receipt_returns, overrides, total_risk, avg_risk</query>
        </search>
        <option name="drilldown">row</option>
        <option name="count">20</option>
        <format type="color" field="total_risk">
          <colorPalette type="minMidMax" maxColor="#DC4E41" midColor="#F8BE34" minColor="#FFFFFF"></colorPalette>
          <scale type="minMidMax" midType="percentile" midValue="50"></scale>
        </format>
        <format type="color" field="void_rate">
          <colorPalette type="minMidMax" maxColor="#DC4E41" minColor="#FFFFFF"></colorPalette>
          <scale type="minMidMax" minValue="0" maxValue="5"></scale>
        </format>
        <drilldown>
          <link target="_blank">search?q=index=pos_transactions cashier_id="$row.cashier_id$" | table _time, transaction_type, amount, risk_score</link>
        </drilldown>
      </table>
    </panel>
  </row>

  <!-- Row 4: Regional Comparison -->
  <row>
    <panel>
      <title>Exception Rate by Region</title>
      <chart>
        <search>

      </chart>
    </panel>
    <panel>
      <title>Shrinkage by Category (Top 10)</title>
      <chart>

| search region=$region$ brand=$brand$
| lookup category_info.csv category_id OUTPUT category_name, avg_unit_value
| eval shrinkage_value = quantity * avg_unit_value
| stats sum(shrinkage_value) as total_shrinkage by category_name
| sort -total_shrinkage
| head 10</query>
        </search>
        <option name="charting.chart">bar</option>
        <option name="charting.chart.showDataLabels">all</option>
      </chart>
    </panel>
  </row>
</form>

<dashboard version="1.1" theme="dark">

  <row>
    <panel>
      <title>Failed Logins (24h)</title>
      <single>
        <search>
          <query>| tstats count FROM datamodel=Authentication.Failed_Authentication 
  WHERE Authentication.user="*$user_filter$*" 
  BY _time span=1h

  WHERE Authentication.user="*$user_filter$*"</query>
          <earliest>$time_range.earliest$</earliest>
          <latest>$time_range.latest$</latest>
        </search>
        <option name="colorMode">block</option>
        <option name="useColors">0</option>
      </single>
    </panel>
    <panel>
      <title>Unique Users</title>
      <single>
        <search>
          <query>| tstats dc(Authentication.user) as unique_users 
  FROM datamodel=Authentication 
  WHERE Authentication.user="*$user_filter$*"</query>
          <earliest>$time_range.earliest$</earliest>
          <latest>$time_range.latest$</latest>
        </search>
      </single>
    </panel>
    <panel>
      <title>Unique Sources</title>
      <single>
        <search>
          <query>| tstats dc(Authentication.src) as unique_sources 
  FROM datamodel=Authentication 
  WHERE Authentication.user="*$user_filter$*"</query>
          <earliest>$time_range.earliest$</earliest>
          <latest>$time_range.latest$</latest>
        </search>
      </single>
    </panel>
  </row>

  <row>
    <panel>
      <title>Authentication Activity Over Time</title>
      <chart>
        <search>
          <query>| tstats count FROM datamodel=Authentication 
  WHERE Authentication.user="*$user_filter$*" 
  BY _time, Authentication.action span=1h
| timechart span=1h sum(count) by Authentication.action</query>
          <earliest>$time_range.earliest$</earliest>
          <latest>$time_range.latest$</latest>
        </search>
        <option name="charting.chart">area</option>
        <option name="charting.chart.stackMode">stacked</option>
        <option name="charting.legend.placement">bottom</option>
      </chart>
    </panel>
    <panel>
      <title>Failed Login Sources (Map)</title>
      <map>
        <search>
          <query>| tstats count FROM datamodel=Authentication.Failed_Authentication 
  WHERE Authentication.user="*$user_filter$*" 
  BY Authentication.src
| iplocation Authentication.src
| geostats count by Country latfield=lat longfield=lon</query>
          <earliest>$time_range.earliest$</earliest>
          <latest>$time_range.latest$</latest>
        </search>
        <option name="mapping.type">marker</option>
      </map>
    </panel>
  </row>

  <row>
    <panel>
      <title>Users with Most Failed Logins</title>

  BY Authentication.user, Authentication.src, Authentication.app
| sort -count
| head 20
| rename Authentication.user as User, Authentication.src as Source, Authentication.app as Application, count as "Failed Attempts"</query>
          <earliest>$time_range.earliest$</earliest>
          <latest>$time_range.latest$</latest>
        </search>
        <option name="drilldown">row</option>
        <option name="count">20</option>
        <format type="color" field="Failed Attempts">
          <colorPalette type="minMidMax" maxColor="#DC4E41" minColor="#FFFFFF"></colorPalette>
          <scale type="minMidMax" minValue="0" maxValue="50"></scale>
        </format>
        <drilldown>
          <link target="_blank">search?q=| tstats count FROM datamodel=Authentication WHERE Authentication.user="$row.User$" BY _time, Authentication.action span=1h</link>
        </drilldown>
      </table>
    </panel>
  </row>

What Enterprises Usually Miss

🔴 Critical Gaps

  • No role-based access: Analysts see same dashboards as executives, information overload
  • Stale dashboards: Dashboards created for old use cases, no longer relevant
  • Real-time abuse: Every dashboard using real-time search, crushing search head
  • No drilldown: Pretty visualizations with no ability to investigate underlying data
  • Missing critical metrics: Dashboard shows activity but not coverage gaps or data quality

Operational Gaps

  • Dashboard sprawl: Hundreds of dashboards, no governance or catalog
  • No version control: Dashboard XML not tracked, changes lost
  • Hardcoded time ranges: No time picker, analysts can't adjust
  • Orphaned dashboards: Creator left, no one maintains or understands
  • No mobile consideration: Dashboards unreadable on tablets/phones

Performance Issues

  • Raw searches in panels: Not using tstats or accelerated data models
  • All-time searches: Panels searching months of data for simple counts
  • Too many panels: 20+ panels each running expensive queries
  • No base search: Same query repeated in multiple panels
  • Excessive refresh: Auto-refresh every 30 seconds not needed

Best Practices Often Ignored

  • Use base searches: One query, multiple visualizations
  • tstats for data model queries: 10-100x faster than raw searches
  • Document dashboard purpose: Description panel explaining what/why
  • Test with realistic data: Verify performance with production volumes
  • Regular cleanup: Archive unused dashboards quarterly

Interview Questions

  1. Identify audience: SOC analysts need different views than executives
  2. Focus on actionable metrics: Notable counts, failed logins, risky users
  3. Use appropriate visualizations: Single values for KPIs, timecharts for trends, tables for details
  4. Add filters: Time picker, severity dropdown, user search for interactivity
  5. Drilldown: Link panels to detailed searches or investigation pages
  6. Performance: Use tstats for data model queries, limit time ranges

Advanced Interview Questions

Systematic optimization approach:

  1. Identify slow panels: Check Job Inspector for each panel's execution time
  2. Use tstats instead of raw searches:
    # Slow: index=security sourcetype=authentication | stats count by user # Fast (10-100x improvement): | tstats count from datamodel=Authentication by Authentication.user
  3. Implement base searches: One query feeding multiple panels
    <search id="base_auth"> | tstats count from datamodel=Authentication by user action </search> <panel depends="$base_auth.done$"> <search base="base_auth"> | stats sum(count) by action </search> </panel>
  4. Reduce time ranges: Default to 24h, not 7d
  5. Use summary indexes: Pre-compute heavy aggregations

Validation: Dashboard should load in <10 seconds with all panels.

Drilldown types:

  1. Search drilldown: Click opens new search with context
    <drilldown> <link target="_blank"> search?q=index=security user=$click.value$&earliest=$earliest$ </link> </drilldown>
  2. Dashboard drilldown: Navigate to detail dashboard
    <drilldown> <link>user_investigation?user=$click.value$</link> </drilldown>
  3. Token-based drilldown: Update other panels
    <drilldown> <set token="selected_user">$click.value$</set> </drilldown>

Best practice: Always preserve time context (earliest/latest tokens) in drilldowns.

Classic (SimpleXML):

  • XML-based, can edit code directly
  • Full ES integration
  • Wide TA/app support
  • Simpler but less flexible layout

Dashboard Studio:

  • Modern drag-and-drop interface
  • Absolute positioning, better visual design
  • New visualization options
  • JSON-based (DSL)
  • Limited ES integration currently

When to use which:

  • Classic: ES dashboards, complex token logic, need TA compatibility
  • Studio: Executive dashboards, visual-first design, new deployments
