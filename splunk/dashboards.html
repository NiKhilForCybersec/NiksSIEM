SOC Dashboard Examples

Security Overview Dashboard

# Panel 1: Notable Events Count (Single Value) index=notable earliest=-24h | stats count | rename count as "Active Notables" # Panel 2: Notable Events by Severity (Pie) index=notable earliest=-24h | stats count by urgency # Panel 3: Notable Events Over Time (Timechart) index=notable earliest=-7d | timechart span=1h count by urgency # Panel 4: Top 10 Notable Types (Bar) index=notable earliest=-24h | stats count by search_name | sort - count | head 10

Authentication Dashboard

# Panel 1: Failed Logins (Timechart) | tstats count from datamodel=Authentication.Failed_Authentication where * by _time span=1h | timechart span=1h count # Panel 2: Top Users with Failed Logins (Table) | tstats count from datamodel=Authentication.Failed_Authentication where * by Authentication.user | sort - count | head 10 | rename Authentication.user as User # Panel 3: Failed Login Sources (Map) | tstats count from datamodel=Authentication.Failed_Authentication where * by Authentication.src | iplocation Authentication.src | geostats count by Country

Network Traffic Dashboard

# Panel 1: Traffic Volume (Timechart) | tstats sum(All_Traffic.bytes) as bytes from datamodel=Network_Traffic by _time span=1h | eval GB = round(bytes/1024/1024/1024, 2) | timechart span=1h sum(GB) as "Traffic (GB)" # Panel 2: Top Talkers (Bar) | tstats sum(All_Traffic.bytes) as bytes from datamodel=Network_Traffic by All_Traffic.src_ip | sort - bytes | head 10 | eval MB = round(bytes/1024/1024, 0) # Panel 3: Top Destinations (Table) | tstats count, sum(All_Traffic.bytes) as bytes from datamodel=Network_Traffic by All_Traffic.dest_ip, All_Traffic.dest_port | sort - count | head 20

What Enterprises Usually Miss

ðŸ”´ Critical Gaps

  • No role-based access: Analysts see same dashboards as executives, information overload
  • Stale dashboards: Dashboards created for old use cases, no longer relevant
  • Real-time abuse: Every dashboard using real-time search, crushing search head
  • No drilldown: Pretty visualizations with no ability to investigate underlying data
  • Missing critical metrics: Dashboard shows activity but not coverage gaps or data quality

Operational Gaps

  • Dashboard sprawl: Hundreds of dashboards, no governance or catalog
  • No version control: Dashboard XML not tracked, changes lost
  • Hardcoded time ranges: No time picker, analysts can't adjust
  • Orphaned dashboards: Creator left, no one maintains or understands
  • No mobile consideration: Dashboards unreadable on tablets/phones

Performance Issues

  • Raw searches in panels: Not using tstats or accelerated data models
  • All-time searches: Panels searching months of data for simple counts
  • Too many panels: 20+ panels each running expensive queries
  • No base search: Same query repeated in multiple panels
  • Excessive refresh: Auto-refresh every 30 seconds not needed

Best Practices Often Ignored

  • Use base searches: One query, multiple visualizations
  • tstats for data model queries: 10-100x faster than raw searches
  • Document dashboard purpose: Description panel explaining what/why
  • Test with realistic data: Verify performance with production volumes
  • Regular cleanup: Archive unused dashboards quarterly

Interview Questions

  1. Identify audience: SOC analysts need different views than executives
  2. Focus on actionable metrics: Notable counts, failed logins, risky users
  3. Use appropriate visualizations: Single values for KPIs, timecharts for trends, tables for details
  4. Add filters: Time picker, severity dropdown, user search for interactivity
  5. Drilldown: Link panels to detailed searches or investigation pages
  6. Performance: Use tstats for data model queries, limit time ranges

Advanced Interview Questions

Systematic optimization approach:

  1. Identify slow panels: Check Job Inspector for each panel's execution time
  2. Use tstats instead of raw searches:
    # Slow: index=security sourcetype=authentication | stats count by user # Fast (10-100x improvement): | tstats count from datamodel=Authentication by Authentication.user
  3. Implement base searches: One query feeding multiple panels
    <search id="base_auth"> | tstats count from datamodel=Authentication by user action </search> <panel depends="$base_auth.done$"> <search base="base_auth"> | stats sum(count) by action </search> </panel>
  4. Reduce time ranges: Default to 24h, not 7d
  5. Use summary indexes: Pre-compute heavy aggregations

Validation: Dashboard should load in <10 seconds with all panels.

Drilldown types:

  1. Search drilldown: Click opens new search with context
    <drilldown> <link target="_blank"> search?q=index=security user=$click.value$&earliest=$earliest$ </link> </drilldown>
  2. Dashboard drilldown: Navigate to detail dashboard
    <drilldown> <link>user_investigation?user=$click.value$</link> </drilldown>
  3. Token-based drilldown: Update other panels
    <drilldown> <set token="selected_user">$click.value$</set> </drilldown>

Best practice: Always preserve time context (earliest/latest tokens) in drilldowns.

Classic (SimpleXML):

  • XML-based, can edit code directly
  • Full ES integration
  • Wide TA/app support
  • Simpler but less flexible layout

Dashboard Studio:

  • Modern drag-and-drop interface
  • Absolute positioning, better visual design
  • New visualization options
  • JSON-based (DSL)
  • Limited ES integration currently

When to use which:

  • Classic: ES dashboards, complex token logic, need TA compatibility
  • Studio: Executive dashboards, visual-first design, new deployments

SOC Operations Dashboard structure:

  1. Top row - KPIs: Single values for critical metrics
    • Open notables (by urgency)
    • MTTR (Mean Time to Respond)
    • Events/sec ingestion rate
    • Active incidents
  2. Middle - Trend and Queue:
    • Notable event trend (timechart)
    • Queue breakdown by status/owner
  3. Bottom - Detail table:
    • Recent notables with drilldown
    • Columns: Time, Rule, Urgency, Status, Owner

Key features:

  • Time picker affecting all panels
  • Urgency filter dropdown
  • Auto-refresh every 5 minutes
  • Drilldown to investigation view

Permission model:

  • Private: Only creator can view/edit
  • App: Users with app access can view
  • Global: All users can view (discouraged)

Implementation:

# In local.meta or via UI [views/mysoc_dashboard] export = none owner = admin access = read : [ soc_analysts, soc_managers ], write : [ soc_managers ]

Role-based patterns:

  • Create dashboard app for each role (SOC, Exec, Hunt)
  • Assign roles to apps via authorize.conf
  • Use depends= to show/hide panels by role

Security note: Dashboard permissions don't restrict data access - users still see data they have index permissions for.

Troubleshooting Guide

Common Issues and Solutions

IssuePossible CauseSolution
Data not appearing Collection or parsing issue Check collector status, verify parser configuration
Slow performance Resource constraints or inefficient queries Review resource utilization, optimize queries
Authentication failures Credential or permission issues Verify credentials, check RBAC configuration
Missing fields Parser not extracting fields Review parsing rules, update field extraction
Duplicate events Multiple collection paths Audit data sources, implement deduplication

Diagnostic Steps

  1. Check Status: Verify component health in console
  2. Review Logs: Examine internal logs for errors
  3. Test Connectivity: Verify network paths and authentication
  4. Validate Config: Compare configuration against documentation
  5. Isolate Issue: Narrow down to specific component
  6. Escalate: Contact support if needed with diagnostics
