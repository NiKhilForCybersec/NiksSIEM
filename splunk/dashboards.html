Dashboards & Visualizations

Complete guide to creating dashboards and visualizations in Splunk. Covers Dashboard Studio, Classic Dashboards, and security-specific examples.

Dashboard Types

TypeBest ForFormat
Dashboard Studio Modern, pixel-perfect layouts JSON definition
Classic Dashboard Quick creation, SimpleXML XML definition
ES Security Dashboards Pre-built ES views Built-in

Creating Dashboards

Search → Save As → Dashboard Panel

Method 1: From Search Results

  1. Run your SPL query
  2. Click "Save As" → "Dashboard Panel"
  3. Choose "New Dashboard" or add to existing
  4. Select visualization type

Method 2: Dashboard Studio

Dashboards → Create New Dashboard → Dashboard Studio
  1. Drag visualizations onto canvas
  2. Configure data source (SPL query)
  3. Style and position elements
  4. Add inputs (dropdowns, time picker)

Visualization Types

Single Value

KPI metrics, counts, totals

| stats count

Line/Area Chart

Trends over time

| timechart count

Bar/Column

Comparisons by category

| stats count by field

Pie Chart

Proportions

| stats count by field

Table

Detailed data display

| table field1, field2

Choropleth Map

Geographic data

| geostats count

SOC Dashboard Examples

Security Overview Dashboard

# Panel 1: Notable Events Count (Single Value) index=notable earliest=-24h | stats count | rename count as "Active Notables" # Panel 2: Notable Events by Severity (Pie) index=notable earliest=-24h | stats count by urgency # Panel 3: Notable Events Over Time (Timechart) index=notable earliest=-7d | timechart span=1h count by urgency # Panel 4: Top 10 Notable Types (Bar) index=notable earliest=-24h | stats count by search_name | sort - count | head 10

Authentication Dashboard

# Panel 1: Failed Logins (Timechart) | tstats count from datamodel=Authentication.Failed_Authentication where * by _time span=1h | timechart span=1h count # Panel 2: Top Users with Failed Logins (Table) | tstats count from datamodel=Authentication.Failed_Authentication where * by Authentication.user | sort - count | head 10 | rename Authentication.user as User # Panel 3: Failed Login Sources (Map) | tstats count from datamodel=Authentication.Failed_Authentication where * by Authentication.src | iplocation Authentication.src | geostats count by Country

Network Traffic Dashboard

# Panel 1: Traffic Volume (Timechart) | tstats sum(All_Traffic.bytes) as bytes from datamodel=Network_Traffic by _time span=1h | eval GB = round(bytes/1024/1024/1024, 2) | timechart span=1h sum(GB) as "Traffic (GB)" # Panel 2: Top Talkers (Bar) | tstats sum(All_Traffic.bytes) as bytes from datamodel=Network_Traffic by All_Traffic.src_ip | sort - bytes | head 10 | eval MB = round(bytes/1024/1024, 0) # Panel 3: Top Destinations (Table) | tstats count, sum(All_Traffic.bytes) as bytes from datamodel=Network_Traffic by All_Traffic.dest_ip, All_Traffic.dest_port | sort - count | head 20

Dashboard Inputs (Filters)

Time Picker

<input type="time" token="time_range"> <label>Time Range</label> <default> <earliest>-24h</earliest> <latest>now</latest> </default> </input>

Dropdown

<input type="dropdown" token="severity"> <label>Severity</label> <choice value="*">All</choice> <choice value="critical">Critical</choice> <choice value="high">High</choice> <choice value="medium">Medium</choice> <default>*</default> </input> # Use in search: index=notable urgency=$severity$

Text Input

<input type="text" token="user_search"> <label>User</label> <default>*</default> </input> # Use in search: index=auth user=*$user_search$*

Interview Questions

  1. Identify audience: SOC analysts need different views than executives
  2. Focus on actionable metrics: Notable counts, failed logins, risky users
  3. Use appropriate visualizations: Single values for KPIs, timecharts for trends, tables for details
  4. Add filters: Time picker, severity dropdown, user search for interactivity
  5. Drilldown: Link panels to detailed searches or investigation pages
  6. Performance: Use tstats for data model queries, limit time ranges

Additional Interview Questions

Systematic troubleshooting approach:

  1. Identify the symptom: What exactly is not working as expected?
  2. Check connectivity: Verify network paths and authentication
  3. Review logs: Check internal/audit logs for errors
  4. Validate configuration: Compare against documented standards
  5. Test incrementally: Isolate the problem component
  6. Document findings: Record what worked and what didn't

Key performance indicators:

  • Availability: Uptime percentage, service availability
  • Performance: Query response time, processing latency
  • Volume: Events per second, data ingestion rate
  • Quality: Parse success rate, enrichment coverage
  • Efficiency: Resource utilization, cost per GB

High availability strategies:

  • Redundancy: Multiple instances of critical components
  • Load balancing: Distribute traffic across instances
  • Failover: Automatic switchover on failure
  • Monitoring: Health checks and alerting
  • DR Planning: Documented recovery procedures

Common Issues and Solutions

IssuePossible CauseSolution
Data not appearing Collection or parsing issue Check collector status, verify parser configuration
Slow performance Resource constraints or inefficient queries Review resource utilization, optimize queries
Authentication failures Credential or permission issues Verify credentials, check RBAC configuration
Missing fields Parser not extracting fields Review parsing rules, update field extraction
Duplicate events Multiple collection paths Audit data sources, implement deduplication

Diagnostic Steps

  3. Test Connectivity: Verify network paths and authentication
  4. Validate Config: Compare configuration against documentation
  5. Isolate Issue: Narrow down to specific component
  6. Escalate: Contact support if needed with diagnostics

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.

Comprehensive Troubleshooting

Systematic Troubleshooting Methodology

  1. Define the Problem: Clearly articulate what is not working as expected. Gather specific error messages, timestamps, and affected components.
  2. Gather Information: Collect relevant logs, metrics, and configuration details. Document recent changes that might be related.
  3. Analyze Data: Review collected information for patterns and anomalies. Compare against baseline behavior and known-good configurations.
  4. Develop Hypothesis: Form theories about potential root causes. Prioritize based on likelihood and impact.
  5. Test Solutions: Implement fixes in a controlled manner. Test one change at a time to isolate the effective solution.
  6. Document Resolution: Record the root cause and solution. Update runbooks and knowledge base for future reference.

Common Issues and Resolutions

CategoryIssueDiagnostic StepsResolution
Connectivity Agent not reporting Check agent service status, network connectivity, firewall rules Restart service, verify network path, update firewall rules
API timeouts Test API endpoint, check rate limits, verify credentials Retry with backoff, request limit increase, refresh credentials
Data Quality Missing fields Review parsing rules, check source format, validate schema Update parser, fix source configuration, adjust schema mapping
Duplicate events Audit data sources, check collection paths, review timestamps Consolidate sources, implement deduplication, fix time sync
Performance Slow queries Analyze query execution plan, check resource utilization Optimize query, add indexes, scale resources
High latency Monitor queue depths, check processing bottlenecks Scale processing capacity, optimize data flow
Authentication Login failures Verify credentials, check account status, review permissions Reset password, unlock account, grant required permissions
Token expiration Check token validity, review refresh mechanism Refresh token, update token management configuration

Diagnostic Commands and Queries

# Health Check Queries (adjust syntax for your platform) # Check system health status | filter log_type = "system_health" | sort _time desc | limit 50 # Monitor error rates over time | filter severity in ("error", "critical") | timechart count by component span=1h # Identify slow operations | filter operation_time > 30000 | stats avg(operation_time), max(operation_time), count by operation_name | sort avg(operation_time) desc # Check connectivity status | filter event_type = "connectivity" | stats latest(status), latest(_time) by source_name | where status != "connected" # Review authentication failures | filter event_type = "authentication" and result = "failure" | stats count by user, source_ip, failure_reason | sort count desc
