search vs where - When to Use Which?

15 Questions

Decision Tree: search vs where

Do you need WILDCARDS in your filter?
YES → wildcards needed
Use: search
NO → exact match
Continue ↓
Do you need to compare TWO FIELDS to each other?
YES → field1 vs field2
Use: where
NO → field vs value
Continue ↓
Do you need FUNCTIONS in your condition?
YES → len(), lower(), etc.
Use: where
NO → simple comparison
Either works

Key Differences

Featuresearchwhere
Wildcards✓ Yes✗ No
Field-to-field comparison✗ No✓ Yes
Equality operator=== (double)
Functions✗ No✓ Yes
Case sensitivityCase-insensitiveCase-sensitive
Treats unquoted asString literalField name
Question 1 of 15
You need to find all events where the status field starts with "fail". Which command should you use?
  • A| search status=fail*
  • B| where status=fail*
  • C| where status=="fail*"
  • D| where like(status, "fail")
✓ Correct: A) | search status=fail*
Why: search supports wildcards (*) directly. where does NOT support wildcards - you'd need where like(status, "fail%") with % instead of *. Option D is missing the wildcard character.
Question 2 of 15
You need to find events where bytes_sent is GREATER than bytes_received. Which is correct?
  • A| search bytes_sent > bytes_received
  • B| where bytes_sent > bytes_received
  • C| search bytes_sent > "bytes_received"
  • D| eval diff=bytes_sent-bytes_received | search diff>0
✓ Correct: B) | where bytes_sent > bytes_received
Why: where can compare two fields directly. search CANNOT compare fields to each other - it only compares fields to literal values. Option D would work but is unnecessarily complex.
Question 3 of 15
You want to filter for usernames longer than 10 characters. Which is correct?
  • A| search len(username) > 10
  • B| where len(username) > 10
  • C| search username_length > 10
  • D| eval x=len(username) | search x > 10
✓ Correct: B) | where len(username) > 10
Why: where supports functions like len(), lower(), isnull(), etc. search does NOT support any functions. Option D would work but is unnecessarily verbose.
Question 4 of 15
What does this search return? index=main | where user=admin
  • AEvents where user field equals "admin"
  • BEvents where user field equals admin field (field-to-field)
  • CSyntax error - where requires ==
  • DAll events regardless of user value
✓ Correct: B) Events where user field equals admin field
Why: In where, unquoted strings are treated as FIELD NAMES. So user=admin compares the user field to the admin field. To match the literal string "admin", use where user="admin" or where user=='admin'.
Question 5 of 15
Which correctly finds events where status is exactly "SUCCESS" (case-sensitive)?
  • A| search status=SUCCESS
  • B| search status="SUCCESS"
  • C| where status="SUCCESS"
  • DBoth A and C
✓ Correct: C) | where status="SUCCESS"
Why: search is CASE-INSENSITIVE (status=SUCCESS matches "success", "Success", "SUCCESS"). where is CASE-SENSITIVE and will only match exact case "SUCCESS".
Question 6 of 15
You want to find events where the error_code field EXISTS (is not null). Which is most efficient?
  • A| search error_code=*
  • B| where error_code=*
  • C| where isnotnull(error_code)
  • D| where error_code!=""
✓ Correct: A) | search error_code=*
Why: error_code=* with search is the fastest way to check field existence. Option B is invalid (where doesn't support wildcards). Option C works but is slower. Option D checks for empty string, not null.
Question 7 of 15
Find events where the error field contains "timeout" anywhere in the value:
  • A| search error=*timeout*
  • B| where error=*timeout*
  • C| where like(error, "timeout")
  • D| where like(error, "%timeout%")
✓ Correct: A and D are both correct
Why: A uses search wildcards (*). D uses the like() function with SQL-style wildcards (%). Note: B is invalid syntax, C would only match exact "timeout".
Question 8 of 15
What's wrong with this search? index=main | where status=200 OR status=404
  • ANothing - it's correct
  • BShould use == instead of = for equality in where
  • COR should be lowercase
  • DNumbers need quotes
✓ Correct: B) Should use == for equality
Why: In where, single = is assignment, double == is comparison. The correct syntax is: | where status==200 OR status==404 or use | where status IN (200, 404).
Question 9 of 15
Which is the MOST EFFICIENT way to filter for status=200?
  • Aindex=main status=200
  • Bindex=main | search status=200
  • Cindex=main | where status==200
  • DAll are equally efficient
✓ Correct: A) index=main status=200
Why: Filtering in the base search (before the first pipe) uses the INDEX and is fastest. Both | search and | where filter AFTER data is retrieved. Always put filters as early as possible!
Question 10 of 15
You need to find NULL values in the email field. Which works?
  • A| search email=null
  • B| search NOT email=*
  • C| where isnull(email)
  • DBoth B and C
✓ Correct: D) Both B and C work
Why: NOT email=* finds events where email doesn't exist. isnull(email) does the same with a function. Option A searches for the literal string "null", not actual null values.
Question 11 of 15
Which finds events where response_time is between 100 and 500 (inclusive)?
  • A| search response_time>=100 AND response_time<=500
  • B| where response_time>=100 AND response_time<=500
  • C| search response_time BETWEEN 100 AND 500
  • D| where response_time BETWEEN 100 AND 500
✓ Correct: B) | where response_time>=100 AND response_time<=500
Why: Splunk doesn't have a BETWEEN operator. Use where with compound conditions. Option A doesn't work because search can't do numeric comparisons reliably.
Question 12 of 15

  • BFinds URLs containing "https://" anywhere
  • CFinds URLs ending with "https://"
  • DSyntax error
✓ Correct: A) Finds URLs starting with "https://"
Why: match() uses regex. The ^ anchor means "starts with". This is a powerful alternative to like() for complex patterns.
Question 13 of 15
Find events where user is "admin" OR "root" OR "system". Most elegant solution?
  • A| search user=admin OR user=root OR user=system
  • B| where user IN ("admin", "root", "system")
  • C| search user IN (admin, root, system)
  • D| where match(user, "admin|root|system")
✓ Correct: B) | where user IN ("admin", "root", "system")
Why: IN operator in where is the cleanest solution for multiple values. Option A works but is verbose. Option C is invalid (search doesn't support IN). Option D works but is overkill.
Question 14 of 15
Which correctly filters for events where src_ip does NOT equal dest_ip?
  • A| search src_ip!=dest_ip
  • B| where src_ip!=dest_ip
  • C| search NOT src_ip=dest_ip
  • DAll of the above
✓ Correct: B) | where src_ip!=dest_ip
Why: Only where can compare two fields. search cannot do field-to-field comparisons - options A and C would compare src_ip to the literal strings "dest_ip".
Question 15 of 15
You have a calculated field ratio created by eval. You want events where ratio > 0.5. Which works?
  • A| eval ratio=sent/received | search ratio>0.5
  • B| eval ratio=sent/received | where ratio>0.5
  • C| search sent/received > 0.5
  • DBoth A and B work
✓ Correct: B) | eval ratio=sent/received | where ratio>0.5
Why: where handles numeric comparisons correctly. search treats values as strings, so "0.5" might not compare correctly with numbers. Always use where for numeric filtering on calculated fields.

Question 16 of 40
You need to count how many UNIQUE users logged in today. Which function?
  • A| stats count(user)
  • B| stats dc(user)
  • C| stats values(user)
  • D| stats count by user | count
✓ Correct: B) | stats dc(user)
Why: dc() = distinct count = unique count. count(user) counts all non-null user values including duplicates. values(user) lists the unique users but doesn't count them.
Question 17 of 40
What's the difference between first(status) and earliest(status)?
  • AThey are identical
  • Bfirst() = first encountered; earliest() = from earliest _time
  • Cfirst() = alphabetically first; earliest() = first encountered
  • Dearliest() is deprecated, use first()
✓ Correct: B) first() = first encountered; earliest() = from earliest _time
Why: first() returns the first value encountered during processing (arbitrary order). earliest() returns the value from the event with the EARLIEST _time timestamp. For time-accurate results, use earliest/latest!
Question 18 of 40
You need a line chart showing errors per hour over the last 24 hours. Which command?
  • A| stats count by _time span=1h
  • B| chart count by _time span=1h
  • C| timechart span=1h count
  • D| stats count | chart over _time
✓ Correct: C) | timechart span=1h count
Why: timechart is designed specifically for time-series data. It automatically handles _time bucketing and creates visualization-ready output. Options A and B have syntax errors (span goes with timechart, not stats/chart).
Question 19 of 40
Show count of events per user per status (pivot table style). Which is correct?
  • A| stats count by user by status
  • B| stats count by user, status
  • C| chart count over user by status
  • DBoth B and C
✓ Correct: D) Both B and C work
Why: stats count by user, status creates a table with rows for each user/status combo. chart count over user by status creates a pivot-style output with users as rows and statuses as columns. Option A has invalid syntax.
Question 20 of 40

✓ Correct: B) Multivalue field listing all unique apps per user

Question 21 of 40

✓ Correct: A) | stats sum(bytes) as total_bytes, avg(response_time) as avg_rt, count
Why: Multiple stats functions can be in one command, separated by commas. Option C is missing commas. Option B would lose data with each stats command.
Question 22 of 40
Find the top 5 users by total bytes downloaded:
  • A| top limit=5 user by bytes
  • B| stats sum(bytes) as total by user | sort -total | head 5
  • C| stats sum(bytes) as total by user | top 5 total
  • D| stats sum(bytes) as total by user | sort 5 -total
✓ Correct: B and D both work
Why: First aggregate with stats, then sort descending. sort 5 -total is a shortcut that sorts and limits in one command. top doesn't work with aggregated values this way.
Question 23 of 40

  • DThey produce identical output

Question 24 of 40

✓ Correct: B) | timechart count by status
Why: by in timechart splits the data into multiple series (lines/bars). Each unique status value becomes a separate line. count(status) would count non-null status values, not split by status.
Question 25 of 40
What does eventstats do differently than stats?
  • Aeventstats is faster
  • Beventstats adds stats to each event without collapsing results
  • Ceventstats works with events, stats works with fields
  • Deventstats only works in real-time searches
✓ Correct: B) eventstats adds stats to each event without collapsing
Why: stats collapses events into aggregated rows. eventstats calculates the same statistics but KEEPS all original events, adding the calculated field to each. Useful for comparing event values to totals/averages.
Question 26 of 40

  • D| accum bytes as running_total
✓ Correct: C) | streamstats sum(bytes) as running_total
Why: streamstats calculates CUMULATIVE statistics as it processes each event. eventstats calculates the TOTAL across all events and adds it to each (not cumulative). accum is deprecated.

Question 31 of 60
Categorize HTTP status: 2xx=Success, 4xx=Client Error, 5xx=Server Error. Best approach?
  • A| eval category=if(status<300,"Success",if(status<500,"Client Error","Server Error"))
  • B| eval category=case(status<300,"Success",status<500,"Client Error",1=1,"Server Error")
  • C| eval category=coalesce(status<300,"Success",status<500,"Client Error","Server Error")
  • DA and B are equally good
✓ Correct: B) case() function
Why: case() is cleaner for multiple conditions. 1=1 at the end acts as the "else" clause. Nested if() (Option A) works but is harder to read. coalesce() is for null handling, not conditionals.

  • D"N/A" if both email and phone exist
✓ Correct: B) First non-null value from the list

Question 33 of 60
Concatenate first_name and last_name with a space:
