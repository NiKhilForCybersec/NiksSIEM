Retention & Storage Tiers - Complete Guide

Complete guide to Splunk's index lifecycle, storage tiers, SmartStore, and data retention configuration with exact settings and CLI commands.

Storage Tier Overview

Hot Buckets

What: Actively receiving new data. Searchable with fastest performance.

Storage: High-performance SSD/NVMe

Default: Data stays until bucket rolls to warm (size or time based)

Config: maxDataSize, maxHotBuckets, maxHotSpanSecs

Warm Buckets

What: No longer receiving data but fully searchable. Good performance.

Storage: SSD or fast HDD

Default: Rolls to cold when warm DB size limit reached

Config: maxWarmDBCount, warmToColdScript

Cold Buckets

What: Older data, still searchable but slower. Often on cheaper storage.

Storage: HDD, NAS, or remote storage

Default: Rolls to frozen (deleted or archived) based on retention

Config: coldPath, coldToFrozenScript, coldToFrozenDir

Frozen Buckets

What: Archived data. Not searchable until restored (thawed).

Storage: S3, Azure Blob, Glacier, tape

Default: Deleted unless coldToFrozenDir or coldToFrozenScript set

Config: frozenTimePeriodInSecs, coldToFrozenDir

Data Lifecycle Flow

HOT → (size/time) → WARM → (count) → COLD → (retention) → FROZEN/DELETE

Index Configuration

indexes.conf - Complete Reference

$SPLUNK_HOME/etc/system/local/indexes.conf or in an app

# indexes.conf - Index definitions with retention [default] # Default settings for all indexes homePath = $SPLUNK_DB/$_index_name/db coldPath = $SPLUNK_DB/$_index_name/colddb thawedPath = $SPLUNK_DB/$_index_name/thaweddb # Global retention (7 days default) frozenTimePeriodInSecs = 604800 [main] # Main index - 90 day retention homePath = $SPLUNK_DB/main/db coldPath = $SPLUNK_DB/main/colddb thawedPath = $SPLUNK_DB/main/thaweddb frozenTimePeriodInSecs = 7776000 # 90 days = 90 * 24 * 60 * 60 = 7,776,000 seconds # Hot bucket settings maxHotBuckets = 10 maxHotSpanSecs = 7776000 maxDataSize = auto_high_volume # Warm bucket settings maxWarmDBCount = 300 # Size limits (optional) maxTotalDataSizeMB = 500000 [security] # Security logs - 1 year retention, archive to frozen homePath = $SPLUNK_DB/security/db coldPath = $SPLUNK_DB/security/colddb thawedPath = $SPLUNK_DB/security/thaweddb frozenTimePeriodInSecs = 31536000 # 365 days = 31,536,000 seconds # Archive instead of delete coldToFrozenDir = /archive/splunk/security [firewall] # Firewall logs - 30 day hot/warm, 90 day total homePath = /fast-storage/splunk/firewall/db coldPath = /slow-storage/splunk/firewall/colddb thawedPath = $SPLUNK_DB/firewall/thaweddb frozenTimePeriodInSecs = 7776000 # High volume optimizations maxHotBuckets = 20 maxDataSize = 10240 # 10GB max bucket size [compliance] # Compliance logs - 7 year retention homePath = $SPLUNK_DB/compliance/db coldPath = /archive-storage/splunk/compliance/colddb thawedPath = $SPLUNK_DB/compliance/thaweddb frozenTimePeriodInSecs = 220752000 # 7 years = 7 * 365 * 24 * 60 * 60 # Archive to S3 via script coldToFrozenScript = /opt/splunk/bin/archive_to_s3.sh

SmartStore (Remote Storage)

What is SmartStore?

SmartStore moves warm buckets to remote object storage (S3, GCS, Azure Blob) while keeping hot data local. Reduces indexer storage costs dramatically.

SmartStore Configuration

$SPLUNK_HOME/etc/system/local/indexes.conf

# indexes.conf - SmartStore with S3 [volume:remote_store] storageType = remote path = s3://my-splunk-bucket/indexes remote.s3.endpoint = https://s3.us-east-1.amazonaws.com remote.s3.access_key = AKIAIOSFODNN7EXAMPLE remote.s3.secret_key = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY [main] remotePath = volume:remote_store/$_index_name homePath = $SPLUNK_DB/main/db coldPath = $SPLUNK_DB/main/colddb thawedPath = $SPLUNK_DB/main/thaweddb # Local cache size maxGlobalDataSizeMB = 500000 # SmartStore settings hotlist_recency_secs = 86400 hotlist_bloom_filter_recency_hours = 360

Azure Blob Configuration

[volume:azure_store] storageType = remote path = https://mystorageaccount.blob.core.windows.net/splunk-indexes remote.azure.connection_string = DefaultEndpointsProtocol=https;AccountName=mystorageaccount;AccountKey=...

GCS Configuration

[volume:gcs_store] storageType = remote path = gs://my-splunk-bucket/indexes remote.gcs.credential_file = /opt/splunk/etc/gcs-credentials.json

CLI Commands

# List all indexes with retention splunk list index # Show index details splunk show index main # Check bucket status splunk _internal call /services/data/indexes/main # Force bucket roll (hot to warm) splunk _internal call /services/data/indexes/main -POST -action=roll-hot-buckets # Rebuild bucket (after restore) splunk rebuild /path/to/bucket # Check SmartStore cache splunk _internal call /services/data/indexes/main/buckets # Thaw frozen bucket splunk thaw main /archive/splunk/main/db_1234567890_1234567800_0

Monitoring Storage

# Index storage usage | rest /services/data/indexes | table title, currentDBSizeMB, maxTotalDataSizeMB, totalEventCount, frozenTimePeriodInSecs | eval retention_days = frozenTimePeriodInSecs / 86400 | eval pct_used = round((currentDBSizeMB / maxTotalDataSizeMB) * 100, 1) # Bucket counts by tier | dbinspect index=* | stats count by index, state | xyseries index, state, count # Daily ingestion by index index=_internal source=*license_usage.log type=Usage | stats sum(b) as bytes by idx | eval GB = round(bytes/1024/1024/1024, 2) | sort - GB # SmartStore cache hit ratio index=_internal sourcetype=splunkd component=CacheManager | stats avg(cache_hit_ratio) by index

What Enterprises Usually Miss

Critical Gaps

  • No retention monitoring: Data frozen before compliance period met, audit failures
  • Disk space crisis: No alerts on storage capacity, indexers stop ingesting
  • Frozen data lost: coldToFrozenDir not set, data deleted instead of archived
  • Compliance violations: Different indexes with different retention, no documentation
  • SmartStore cache exhaustion: Cache too small, every search fetches from S3

Operational Gaps

  • No capacity planning: Volume growth exceeds storage procurement timeline
  • Index sprawl: Dozens of indexes with inconsistent settings
  • Hot bucket contention: Too many concurrent writers to same index
  • Warm storage full: Cold path not on separate volume
  • No freeze verification: Data assumed archived but never validated

Performance Issues

  • Bucket too large: maxDataSize too high, search scans entire large buckets
  • Too many hot buckets: Excessive parallelism wastes file handles
  • Cold storage latency: Cold path on slow NAS, historical searches crawl
  • SmartStore thrashing: Cache eviction rate too high, constant S3 fetches
  • Bloom filter miss: Older buckets not in hotlist, expensive searches

Best Practices Often Ignored

  • Weekly storage review: | rest /services/data/indexes | table title currentDBSizeMB maxTotalDataSizeMB
  • Document retention policy: Map indexes to compliance requirements
  • Test freeze/thaw process: Verify archived data is recoverable before you need it
  • Separate storage paths: Hot on SSD, warm/cold on HDD, different volumes
  • Alert on 80% capacity: Proactive notification before crisis

Interview Questions

Four tiers:

  1. Hot: Actively receiving data, fastest search, SSD storage. Rolls when size (maxDataSize) or time (maxHotSpanSecs) limit reached.
  2. Warm: Not receiving data but searchable, good performance. Rolls when count (maxWarmDBCount) exceeded.
  3. Cold: Older searchable data, can be on slower/cheaper storage. Rolls based on frozenTimePeriodInSecs.
  4. Frozen: Archived or deleted. Not searchable until thawed. Controlled by coldToFrozenDir or coldToFrozenScript.

Key config: indexes.conf defines paths, retention (frozenTimePeriodInSecs), and bucket limits.

SmartStore offloads warm buckets to remote object storage (S3, GCS, Azure Blob) while keeping hot data local.

Benefits:

  • Dramatically reduces indexer storage costs (80%+ savings)
  • Enables longer retention without massive local storage
  • Separates compute (indexers) from storage (object store)

Use when: High data volumes, cloud deployments, need long retention, want storage cost optimization.

Tradeoff: Slightly slower searches for data not in local cache.

Advanced Interview Questions

Calculation approach:

  1. Total storage needed:
    • Raw: 500GB × 365 = 182.5TB
    • With compression (typical 50%): ~91TB
    • With tsidx overhead (15-20%): ~109TB total
  2. Index configuration:
    [myindex] homePath = $SPLUNK_DB/myindex/db coldPath = $SPLUNK_DB/myindex/colddb thawedPath = $SPLUNK_DB/myindex/thaweddb frozenTimePeriodInSecs = 31536000 # 365 days maxDataSize = auto_high_volume maxTotalDataSizeMB = 112000000 # ~109TB limit
  3. Storage tier split:
    • Hot (7 days): 3.5TB on SSD
    • Warm (30 days): 15TB on fast storage
    • Cold (remainder): ~90TB on cheaper storage

Pro tip: With SmartStore, you'd only need ~20TB local (hot + cache), rest in S3.

Bucket lifecycle:

  • Hot → Warm: Triggered by maxDataSize OR maxHotSpanSecs
  • Warm → Cold: Triggered by maxWarmDBCount
  • Cold → Frozen: Triggered by frozenTimePeriodInSecs OR maxTotalDataSizeMB

Tuning for high-volume:

# indexes.conf [high_volume_index] # Roll hot buckets at 10GB (default auto = ~750MB) maxDataSize = 10000 # Keep hot for max 24 hours maxHotSpanSecs = 86400 # Allow 300 warm buckets before rolling to cold maxWarmDBCount = 300 # Freeze data older than 90 days frozenTimePeriodInSecs = 7776000

Monitoring: | dbinspect index=myindex | stats count by state

Thaw process:

  1. Locate frozen buckets: Find in coldToFrozenDir or archive location
  2. Copy to thawed path:
    cp -r /archive/myindex/db_1234567890_1234567800_0 \ $SPLUNK_DB/myindex/thaweddb/
  3. Rebuild bucket manifest:
    ./splunk rebuild /opt/splunk/var/lib/splunk/myindex/thaweddb/db_xxx
  4. Refresh indexer: ./splunk refresh index myindex

Key points:

  • Thawed data is searchable but won't participate in clustering
  • No automatic rolling - stays in thawed until manually removed
  • Document chain of custody for legal/compliance

Age-based (frozenTimePeriodInSecs):

  • Data older than X seconds gets frozen
  • Predictable - "always keep 90 days"
  • Storage costs can vary with volume
  • Best for: Compliance requirements with specific timeframes

Size-based (maxTotalDataSizeMB):

  • Oldest data frozen when index exceeds size limit
  • Predictable storage costs
  • Retention period varies with volume
  • Best for: Budget-constrained environments

Combination: Both can be set - whichever triggers first wins. Common pattern: size limit as safety net, age as primary.

Key monitoring searches:

# Index sizes across cluster | rest /services/data/indexes | stats sum(currentDBSizeMB) as totalMB by title | sort - totalMB # Bucket state distribution | dbinspect index=* | stats count by index state # Storage capacity index=_internal source=*metrics.log group=per_index_thruput | timechart span=1d sum(kb) by series # Cluster bucket status | rest /services/cluster/master/indexes | table title buckets_searchable replication_factor search_factor

Alerting thresholds:

  • Alert at 80% capacity: Time to add storage or adjust retention
  • Alert on replication behind: Cluster health issue
  • Alert on freeze rate spike: Possible config issue or unexpected volume

How SmartStore caching works:

  • Hot buckets: Always local (not in remote store)
  • Warm buckets: Uploaded to S3, cached locally
  • Cache eviction: LRU-based when cache fills

Optimization settings:

# server.conf [cachemanager] max_cache_size = 500 # GB of local cache per index hotlist_recency_secs = 604800 # Keep last 7 days hot hotlist_bloom_filter_recency_hours = 168 # 7 days bloom filters

Monitoring cache efficiency:

index=_internal source=*metrics.log group=cachemanager | timechart avg(cache_hit_ratio) avg(download_bytes)

Target: Cache hit ratio >90% for typical searches. If lower, increase cache size or improve search habits.
