Querying Notable Events

SPL Notable Event Queries 
// View all open notables
\`notable\`
| search status_label!="Resolved" AND status_label!="Closed"
| table _time, rule_name, urgency, owner, status_label, src, dest, user

// SOC metrics - alert volume by rule
\`notable\`
| stats count by rule_name
| sort -count

// MTTR calculation
\`notable\`
| where status_label="Resolved"
| eval resolve_time = status_end - _time
| stats avg(resolve_time) as avg_mttr by rule_name
| eval avg_mttr_hours = round(avg_mttr/3600, 2)

Interview Preparation

Describe the Notable Event workflow in Splunk ES.

Notable Events are the core alert mechanism in ES:

Creation: Correlation searches detect threats and create notables via action.notable=1. Each notable includes the detection context, affected entities (src, dest, user), severity, and drilldown search.

Incident Review: Analysts work from the Incident Review dashboard. They can filter by severity, security domain, status, and owner. The swimlane view shows notable trends over time.

Workflow: New → In Progress (assigned) → Pending (if waiting) → Resolved (investigated) → Closed. Analysts add comments, disposition (true/false positive), and document findings.

Metrics: The `notable` macro in searches lets you track MTTA (time to assign), MTTR (time to resolve), and false positive rates.
