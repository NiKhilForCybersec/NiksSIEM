Notable Events - Investigation Workflow

Complete guide to working with Notable Events in Splunk Enterprise Security. Covers the investigation workflow, status management, and analyst procedures.

What are Notable Events?

Notable Events are security alerts generated by Correlation Searches in Enterprise Security. They represent potential security incidents that require analyst review.

Notable Event Lifecycle

New → In Progress → Pending → Resolved → Closed

Incident Review Dashboard

Enterprise Security → Security Intelligence → Incident Review

Key Features

  • Timeline view: Notable events over time with urgency color coding
  • Filters: Filter by status, urgency, owner, security domain, search name
  • Quick actions: Change status, assign owner, add comments
  • Drilldown: Click to open full investigation view

Investigation Workflow

Step 1: Triage

1
Review notable details: Rule name, urgency, time, key fields (user, host, IP)
2
Set status to "In Progress" and assign to yourself
3
Quick validation: Is this a known false positive? Check suppression lists.

Step 2: Investigate with Adaptive Response Actions

Click "Actions" on a notable to run investigation enrichments:

ActionPurposeData Source
Asset InvestigatorHost details, software, vulnerabilitiesAsset table
Identity InvestigatorUser details, manager, departmentIdentity table
Threat ActivityIOC matches for this host/userThreat Intel
Risk AnalysisRisk score history for entityRisk framework
User ActivityAuthentication, access patternsAuth data model
Network ActivityConnection history for IP/hostNetwork data model

Step 3: Pivot Searches

Drill down into raw events for deeper investigation:

# View all events that triggered this notable index=notable event_id="{event_id}" | table _time, search_name, src, dest, user, urgency # Investigate user activity around notable time index=* user="{user from notable}" earliest=-1h latest=+1h | stats count by index, sourcetype | sort - count # Check all authentication for affected user | tstats count from datamodel=Authentication where Authentication.user="{user}" by Authentication.action, Authentication.src, Authentication.app | sort - count # Network activity from affected host | tstats count from datamodel=Network_Traffic where All_Traffic.src="{host}" by All_Traffic.dest, All_Traffic.dest_port | sort - count

Step 4: Document & Disposition

4
Add investigation notes as comments on the notable
5
Set final status:
  • Resolved - True Positive: Confirmed security incident → Escalate/Create ticket
  • Resolved - False Positive: Not a real threat → Consider suppression rule
  • Closed - No Action: Benign/expected activity

Managing Notables via SPL

# View all open notables index=notable | where status_label!="Closed" | table _time, search_name, urgency, status_label, owner, src, dest, user # View notables assigned to me index=notable owner="{your_username}" | where status_label="In Progress" | table _time, search_name, urgency, src, dest # Count notables by rule index=notable earliest=-7d | stats count by search_name | sort - count # Update notable status (via sendalert) | sendalert update_notable_status status="2" comment="Investigated - confirmed FP"

Suppression Rules

Enterprise Security → Configure → Content → Content Management → [Rule] → Create Suppression

Suppress known false positives to reduce noise:

# Suppression rule example Search: search_name="Brute Force Access Behavior Detected" Fields to suppress: src_ip="10.0.1.50" # Known scanning tool Expiration: 30 days

Best Practices

  • Always set expiration dates on suppressions
  • Document reason in suppression description
  • Review suppressions quarterly
  • Prefer tuning the rule over blanket suppression

Interview Questions

  1. Triage: Open Incident Review, review notable details (rule, urgency, entities). Set status to "In Progress" and assign to yourself.
  2. Enrich: Use Adaptive Response Actions - Asset Investigator, Identity Investigator, Threat Activity to gather context.
  3. Pivot: Drill down to raw events. Search for user/host activity before and after the notable time.
  4. Correlate: Check if same entity appears in other notables. Review risk score history.
  5. Document: Add comments with investigation findings.
  6. Disposition: Set final status (True Positive → escalate, False Positive → consider suppression, No Action → close).
  • Tune correlation search: Add exclusions to the search logic (exclude known safe IPs, service accounts)
  • Suppression rules: Create time-limited suppressions for specific field values
  • Adjust thresholds: Increase count thresholds or time windows
  • Whitelist lookups: Maintain lookup tables of known-good entities
  • Asset/Identity context: Exclude notables where asset priority is low

Best practice: Track FP rate per rule and prioritize tuning high-volume, low-value rules.

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.

Comprehensive Troubleshooting

Systematic Troubleshooting Methodology

  1. Define the Problem: Clearly articulate what is not working as expected. Gather specific error messages, timestamps, and affected components.
  2. Gather Information: Collect relevant logs, metrics, and configuration details. Document recent changes that might be related.
  3. Analyze Data: Review collected information for patterns and anomalies. Compare against baseline behavior and known-good configurations.
  4. Develop Hypothesis: Form theories about potential root causes. Prioritize based on likelihood and impact.
  5. Test Solutions: Implement fixes in a controlled manner. Test one change at a time to isolate the effective solution.
  6. Document Resolution: Record the root cause and solution. Update runbooks and knowledge base for future reference.
