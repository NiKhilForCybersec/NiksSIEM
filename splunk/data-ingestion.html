Splunk Data Ingestion

Getting data into Splunk is the foundation of your security monitoring. Choose the right input method based on your data source, volume, and transformation requirements.

Choosing the Right Input Method

What type of data source are you onboarding?
Windows/Linux Endpoint

→ Deploy Universal Forwarder

Install UF on endpoint. Configure inputs.conf for event logs, files, metrics.

Network Device / Appliance

→ Continue to syslog options

Does the device support syslog output?
Yes - Syslog Supported

→ Use Heavy Forwarder or Indexer

Configure syslog input on HF/Indexer. Point device to send syslog.

No Syslog / Cloud Service

→ Use Add-on or HEC

Install vendor TA from Splunkbase or use HTTP Event Collector API.

Do you need to parse/filter data before indexing?
Yes - Need Transformation

→ Route through Heavy Forwarder

HF can parse, filter, route, and transform before sending to indexers.

No - Raw Data OK

→ Send Direct to Indexer

Parse at search time. Simpler architecture but higher storage.

Method 1: Universal Forwarder

Universal Forwarder (UF)
Lightweight agent deployed on endpoints to collect and forward data
Best For
Windows/Linux/macOS endpoints, servers
Footprint
~50MB install, minimal CPU/RAM
Capabilities
File monitor, WinEventLog, scripted inputs
Limitations
Cannot parse or transform (forward only)

Windows Event Log Collection

inputs.conf (on Universal Forwarder)
# Security Event Log - Critical for SOC
[WinEventLog://Security]
disabled = 0
index = wineventlog
sourcetype = WinEventLog:Security
# Blacklist noisy events (optional)
blacklist1 = EventCode="4689" Message="Process Exited"
blacklist2 = EventCode="5156" # WFP allowed connection (very noisy)

# System Event Log
[WinEventLog://System]
disabled = 0
index = wineventlog
sourcetype = WinEventLog:System

# PowerShell Script Block Logging (CRITICAL for threat detection)
[WinEventLog://Microsoft-Windows-PowerShell/Operational]
disabled = 0
index = wineventlog
sourcetype = WinEventLog:PowerShell

# Sysmon (if deployed)
[WinEventLog://Microsoft-Windows-Sysmon/Operational]
disabled = 0
index = wineventlog
sourcetype = XmlWinEventLog:Microsoft-Windows-Sysmon/Operational

Syslog Input Configuration

inputs.conf (on Heavy Forwarder)
# UDP Syslog - Most network devices use this
[udp://514]
connection_host = ip
sourcetype = syslog
index = network

# TCP Syslog - More reliable, use for critical sources
[tcp://514]
connection_host = ip
sourcetype = syslog
index = network

# Dedicated port for Palo Alto (recommended)
[udp://5514]
connection_host = ip
sourcetype = pan:traffic
index = pan_logs

# TLS Syslog for sensitive sources
[tcp-ssl://6514]
connection_host = ip
sourcetype = syslog
index = network

Method 3: HTTP Event Collector (HEC)

HTTP Event Collector (HEC)
RESTful API for sending events and metrics over HTTPS
Best For
Applications, containers, serverless, IoT
Authentication
Token-based (no Splunk credentials)
Port
8088 (HTTPS) or 8088 (HTTP for internal)
Format
JSON events or raw text
HEC API Example (curl)
# Send single event
curl -k https://splunk-hec.company.com:8088/services/collector/event \
  -H "Authorization: Splunk YOUR-HEC-TOKEN-HERE" \
  -d '{"event": {"user": "jsmith", "action": "login", "status": "success"}, 
       "sourcetype": "myapp:auth", 
       "index": "main"}'

# Send batch of events (more efficient)
curl -k https://splunk-hec.company.com:8088/services/collector/event \
  -H "Authorization: Splunk YOUR-HEC-TOKEN-HERE" \
  -d '{"event": {"user": "jsmith", "action": "login"}, "time": 1705312800}
      {"event": {"user": "mjones", "action": "logout"}, "time": 1705312801}'

# Send raw event (let Splunk parse)
curl -k https://splunk-hec.company.com:8088/services/collector/raw \
  -H "Authorization: Splunk YOUR-HEC-TOKEN-HERE" \
  -d 'Jan 15 10:30:00 webserver01 nginx: 192.168.1.100 - GET /api/users 200'

Why HEC? No sidecars or forwarders in containers. Fluent Bit collects stdout/stderr and forwards via HEC. Token-based auth is perfect for CI/CD pipelines.

Best Practice: Use separate HEC tokens per application/team. Set index and sourcetype in token config for governance.

Method Comparison

Method Best For Pros Cons
Universal Forwarder Endpoints, servers Lightweight, secure, managed Requires agent install
Heavy Forwarder Syslog aggregation Parse at edge, filter, route More resources, licensing
HEC Apps, containers, APIs No agent, token auth, JSON Requires app changes
Modular Inputs (TAs) Cloud APIs, DBs Pre-built integrations May need credentials
Direct Syslog to Indexer Simple deployments No HF needed No pre-indexing transform

Interview Questions

License calculation steps:

  1. Sample collection: Collect 24 hours of raw logs from the source
  2. Measure volume: Check file sizes or event counts
  3. Account for growth: Add 20-30% buffer for growth
  4. Consider filtering: Can you drop non-security events?

Common estimates:

  • Windows Security logs: 100-500 MB/day per server (varies greatly)
  • Palo Alto firewall: 1-10 GB/day depending on traffic
  • Proxy logs: 2-20 GB/day (very chatty)

Use HEC when:

  • Can't install agents (containers, serverless, IoT)
  • Application can send HTTP/JSON natively
  • Need token-based auth (CI/CD, multi-tenant)
  • Ephemeral workloads (containers spinning up/down)

Use Forwarder when:

  • Collecting OS-level logs (Event Log, syslog files)
  • Need reliable delivery with acknowledgment
  • Want centralized config management
  • Collecting from persistent servers/endpoints
