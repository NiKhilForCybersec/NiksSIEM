Splunk Forwarders - Complete Deployment Guide

Comprehensive guide to Universal Forwarders and Heavy Forwarders. Covers deployment methods, configuration, management at scale, and troubleshooting.

Forwarder Types

Feature Universal Forwarder (UF) Heavy Forwarder (HF)
Purpose Lightweight data collection Data collection + parsing + routing
Resource Usage ~20 MB RAM, minimal CPU 1-4 GB RAM, moderate CPU
Parsing No (raw data only) Yes (full props.conf/transforms.conf)
Searching No Yes (can search locally)
Use Cases Endpoints, servers, workstations Syslog aggregation, data routing, edge parsing
License Free Requires Splunk Enterprise license

Universal Forwarder Deployment

Installation Methods

Windows - MSI Silent Install

# Basic installation msiexec.exe /i splunkforwarder-9.1.0-x64-release.msi AGREETOLICENSE=yes /quiet # Full deployment with configuration msiexec.exe /i splunkforwarder-9.1.0-x64-release.msi ^ AGREETOLICENSE=yes ^ RECEIVING_INDEXER="indexer1.company.com:9997,indexer2.company.com:9997" ^ DEPLOYMENT_SERVER="deployment.company.com:8089" ^ WINEVENTLOG_APP_ENABLE=1 ^ WINEVENTLOG_SEC_ENABLE=1 ^ WINEVENTLOG_SYS_ENABLE=1 ^ WINEVENTLOG_FWD_ENABLE=1 ^ WINEVENTLOG_SET_ENABLE=1 ^ SPLUNKUSERNAME=admin ^ SPLUNKPASSWORD=changeme ^ /quiet

Common MSI Parameters:

  • RECEIVING_INDEXER - Comma-separated indexer:port list
  • DEPLOYMENT_SERVER - Deployment server for central management
  • WINEVENTLOG_*_ENABLE - Enable Windows Event Log collection
  • LAUNCHSPLUNK=1 - Start after install
  • INSTALLDIR - Custom install path

Linux - CLI Installation

# Download and extract wget -O splunkforwarder.tgz "https://download.splunk.com/products/universalforwarder/releases/9.1.0/linux/splunkforwarder-9.1.0-Linux-x86_64.tgz" tar -xzf splunkforwarder.tgz -C /opt # Accept license and set admin credentials /opt/splunkforwarder/bin/splunk start --accept-license --answer-yes --seed-passwd 'YourPassword123!' # Configure deployment server /opt/splunkforwarder/bin/splunk set deploy-poll deployment.company.com:8089 -auth admin:YourPassword123! # Enable boot start /opt/splunkforwarder/bin/splunk enable boot-start -user splunk # Restart /opt/splunkforwarder/bin/splunk restart

Enterprise Deployment - SCCM/GPO

# PowerShell deployment script for SCCM $splunkMSI = "\\fileserver\software\splunkforwarder-9.1.0-x64-release.msi" $deploymentServer = "deployment.company.com:8089" $indexers = "indexer1.company.com:9997,indexer2.company.com:9997" $arguments = @( "/i `"$splunkMSI`"", "AGREETOLICENSE=yes", "DEPLOYMENT_SERVER=`"$deploymentServer`"", "RECEIVING_INDEXER=`"$indexers`"", "WINEVENTLOG_APP_ENABLE=1", "WINEVENTLOG_SEC_ENABLE=1", "WINEVENTLOG_SYS_ENABLE=1", "LAUNCHSPLUNK=1", "/quiet" ) -join " " Start-Process msiexec.exe -ArgumentList $arguments -Wait -NoNewWindow

Essential Configuration Files

outputs.conf - Where to Send Data

$SPLUNK_HOME/etc/system/local/outputs.conf

# outputs.conf - Send to indexer cluster [tcpout] defaultGroup = indexers [tcpout:indexers] server = indexer1.company.com:9997, indexer2.company.com:9997, indexer3.company.com:9997 # Load balancing autoLBFrequency = 30 # SSL encryption sslCertPath = $SPLUNK_HOME/etc/auth/server.pem sslRootCAPath = $SPLUNK_HOME/etc/auth/ca.pem sslPassword = password sslVerifyServerCert = true # Queuing for reliability [tcpout:indexers] useACK = true maxQueueSize = 500MB

inputs.conf - What to Collect

$SPLUNK_HOME/etc/apps/myapp_inputs/local/inputs.conf

# inputs.conf - Common input configurations # Monitor a directory of log files [monitor:///var/log/application/*.log] disabled = false index = main sourcetype = app:logs whitelist = \.log$ ignoreOlderThan = 7d # Windows Event Logs [WinEventLog://Security] disabled = false index = wineventlog evt_resolve_ad_obj = 1 checkpointInterval = 5 current_only = 0 [WinEventLog://System] disabled = false index = wineventlog [WinEventLog://Application] disabled = false index = wineventlog # Windows Security Events - specific event codes [WinEventLog://Security] disabled = false index = wineventlog whitelist = 4624,4625,4634,4648,4672,4688,4720,4726,4728,4732,4756 # Scripted input [script://$SPLUNK_HOME/etc/apps/myapp/bin/custom_metrics.sh] disabled = false index = metrics sourcetype = custom:metrics interval = 60

deploymentclient.conf - Central Management

$SPLUNK_HOME/etc/system/local/deploymentclient.conf

# deploymentclient.conf - Connect to Deployment Server [deployment-client] phoneHomeIntervalInSecs = 60 [target-broker:deploymentServer] targetUri = deployment.company.com:8089

Heavy Forwarder Configuration

Syslog Collection

$SPLUNK_HOME/etc/apps/syslog_inputs/local/inputs.conf

# inputs.conf - Syslog listeners # UDP Syslog (standard) [udp://514] connection_host = ip sourcetype = syslog index = network # TCP Syslog (reliable) [tcp://514] connection_host = ip sourcetype = syslog index = network # Dedicated ports per device type [tcp://1514] connection_host = ip sourcetype = pan:traffic index = firewall [tcp://1515] connection_host = ip sourcetype = cisco:asa index = firewall # TLS Syslog [tcp-ssl://6514] connection_host = ip sourcetype = syslog_secure index = network

Data Routing (props.conf + transforms.conf)

# transforms.conf - Route based on content [route_to_firewall_index] REGEX = (TRAFFIC|SYSTEM|THREAT) DEST_KEY = _MetaData:Index FORMAT = firewall [route_to_auth_index] REGEX = (LOGIN|LOGOUT|AUTH) DEST_KEY = _MetaData:Index FORMAT = authentication # props.conf - Apply routing [syslog] TRANSFORMS-routing = route_to_firewall_index, route_to_auth_index

Deployment Server Management

Settings → Forwarder Management

Server Classes

Server classes group forwarders and define which apps they receive.

# serverclass.conf on Deployment Server # $SPLUNK_HOME/etc/system/local/serverclass.conf [serverClass:Windows_Servers] whitelist.0 = *.company.com blacklist.0 = linux* machineTypesFilter = windows-x64 [serverClass:Windows_Servers:app:Splunk_TA_windows] restartSplunkWeb = false restartSplunkd = true stateOnClient = enabled [serverClass:Linux_Servers] whitelist.0 = linux*.company.com machineTypesFilter = linux-x86_64 [serverClass:Linux_Servers:app:Splunk_TA_nix] restartSplunkWeb = false restartSplunkd = true stateOnClient = enabled

Deploy Apps to Forwarders

1
Place app in $SPLUNK_HOME/etc/deployment-apps/
2
Create/edit serverclass.conf to assign app to server class
3
Reload: $SPLUNK_HOME/bin/splunk reload deploy-server
4
Forwarders check in and receive apps (within phoneHomeInterval)

Useful SPL Queries

# List all forwarders and last check-in index=_internal sourcetype=splunkd group=tcpin_connections | stats latest(_time) as last_seen by hostname, version, arch | eval status=if(last_seen > relative_time(now(), "-15m"), "Active", "Stale") | sort - last_seen # Forwarder data volume by host index=_internal sourcetype=splunkd group=per_host_thruput | stats sum(kb) as total_kb by series | eval MB = round(total_kb/1024, 2) | sort - MB # Check deployment server client status index=_internal sourcetype=splunkd component=DC* | stats count by host, log_level, message # Find forwarders not sending data | rest /services/deployment/server/clients | table clientName, hostname, ip, lastPhoneHomeTime | eval hours_since = round((now() - strptime(lastPhoneHomeTime, "%Y-%m-%dT%H:%M:%S")) / 3600, 1) | where hours_since > 1

Interview Questions

Universal Forwarder (UF):

  • Lightweight (~20MB RAM), free license
  • Collects and forwards raw data only
  • No parsing, no searching capability
  • Best for: endpoints, servers, workstations

Heavy Forwarder (HF):

  • Full Splunk Enterprise instance (1-4GB RAM)
  • Can parse, filter, route, and transform data
  • Can search local data
  • Best for: syslog aggregation, data routing, edge processing

Decision: Use UF unless you need parsing at the edge (syslog from network devices) or complex routing.

  1. Initial deployment: SCCM, GPO, Ansible, or Puppet for mass installation with preconfigured deploymentclient.conf
  2. Central management: Deployment Server with server classes groups forwarders by OS, function, or location
  3. Configuration distribution: Apps placed in deployment-apps are pushed to matching server classes
  4. Monitoring: Use _internal index to track forwarder health, last check-in, data volume

Key files: deploymentclient.conf (on forwarder), serverclass.conf (on deployment server)

Additional Interview Questions

Systematic troubleshooting approach:

  1. Identify the symptom: What exactly is not working as expected?
  2. Check connectivity: Verify network paths and authentication
  3. Review logs: Check internal/audit logs for errors
  4. Validate configuration: Compare against documented standards
  5. Test incrementally: Isolate the problem component
  6. Document findings: Record what worked and what didn't

Key performance indicators:

  • Availability: Uptime percentage, service availability
  • Performance: Query response time, processing latency
  • Volume: Events per second, data ingestion rate
  • Quality: Parse success rate, enrichment coverage
  • Efficiency: Resource utilization, cost per GB

High availability strategies:

  • Redundancy: Multiple instances of critical components
  • Load balancing: Distribute traffic across instances
  • Failover: Automatic switchover on failure
  • Monitoring: Health checks and alerting
  • DR Planning: Documented recovery procedures

Troubleshooting Guide

Common Issues and Solutions

IssuePossible CauseSolution
Data not appearing Collection or parsing issue Check collector status, verify parser configuration
Slow performance Resource constraints or inefficient queries Review resource utilization, optimize queries
Authentication failures Credential or permission issues Verify credentials, check RBAC configuration
Missing fields Parser not extracting fields Review parsing rules, update field extraction
Duplicate events Multiple collection paths Audit data sources, implement deduplication

Diagnostic Steps

  1. Check Status: Verify component health in console
  2. Review Logs: Examine internal logs for errors
  3. Test Connectivity: Verify network paths and authentication
  4. Validate Config: Compare configuration against documentation
  5. Isolate Issue: Narrow down to specific component
  6. Escalate: Contact support if needed with diagnostics

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.
