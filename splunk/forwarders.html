Forwarders Deep Dive

Complete guide to Universal Forwarders, Heavy Forwarders, Deployment Server, and enterprise-scale forwarder management

Forwarder Types Comparison

Feature Universal Forwarder Heavy Forwarder
Size ~50MB Full Splunk (~500MB+)
Parsing Minimal (timestamp, linebreaking) Full parsing capability
Indexing No Optional (usually disabled)
Searching No Optional
Web UI No Yes

Universal Forwarder (UF)

Installation

Bash Linux Installation 
# Download and install
wget -O splunkforwarder.tgz "https://download.splunk.com/products/universalforwarder/releases/9.1.0/linux/splunkforwarder-9.1.0-linux-x86_64.tgz"
tar -xzf splunkforwarder.tgz -C /opt

# Start and accept license
/opt/splunkforwarder/bin/splunk start --accept-license --answer-yes --no-prompt --seed-passwd 'YourPassword123!'

# Enable boot start
/opt/splunkforwarder/bin/splunk enable boot-start -user splunk

# Configure deployment client
/opt/splunkforwarder/bin/splunk set deploy-poll deployment-server.company.com:8089 -auth admin:password
PowerShell Windows Silent Installation 
# Silent install with deployment server
msiexec.exe /i splunkforwarder-9.1.0-x64.msi `
    AGREETOLICENSE=yes `
    DEPLOYMENT_SERVER="deployment-server.company.com:8089" `
    RECEIVING_INDEXER="idx1.company.com:9997,idx2.company.com:9997" `
    WINEVENTLOG_SEC_ENABLE=1 `
    WINEVENTLOG_SYS_ENABLE=1 `
    WINEVENTLOG_APP_ENABLE=1 `
    /quiet

Heavy Forwarder (HF)

When to Use Heavy Forwarder

  • Syslog Reception: Network devices (firewalls, switches, routers) send syslog
  • Pre-Indexing Parsing: Complex parsing to reduce indexer load
  • Data Masking: Mask sensitive data (PII, credit cards) before indexing
  • Routing: Route data to different indexes based on content
  • Aggregation: Aggregate multiple UFs before sending to indexers
  • API Collection: Collect from REST APIs, databases

Configuration for Forwarding

conf outputs.conf - Heavy Forwarder 
# Disable local indexing
[indexAndForward]
index = false

[tcpout]
defaultGroup = indexer_cluster
forwardedindex.filter.disable = true
indexAndForward = false

[tcpout:indexer_cluster]
server = idx1.company.com:9997, idx2.company.com:9997, idx3.company.com:9997

Data Routing by Content

conf props.conf + transforms.conf - Routing 
# props.conf
[syslog]
TRANSFORMS-route = route_paloalto, route_cisco, route_default

# transforms.conf
[route_paloalto]
REGEX = TRAFFIC|THREAT|SYSTEM.*PAN-OS
DEST_KEY = _MetaData:Index
FORMAT = pan_logs

[route_cisco]
REGEX = %ASA-|%FWSM-
DEST_KEY = _MetaData:Index
FORMAT = cisco_logs

[route_default]
REGEX = .
DEST_KEY = _MetaData:Index
FORMAT = network

Deployment Server

Central management for forwarder configurations. Push apps and configurations to thousands of forwarders.

┌─────────────────────────────────────────────────────────────────┐
│                      DEPLOYMENT SERVER                           │
│                                                                  │
│  $SPLUNK_HOME/etc/deployment-apps/                               │
│  ├── Splunk_TA_windows/          (Windows inputs)                │
│  ├── Splunk_TA_nix/              (Linux/Unix inputs)             │
│  ├── org_all_forwarders/         (Base config - all)             │
│  ├── org_outputs/                (outputs.conf)                  │
│  └── org_windows_dc/             (Domain controllers)            │
│                                                                  │
│  serverclass.conf defines:                                       │
│  - Server classes (groups of forwarders)                         │
│  - Which apps go to which server classes                         │
│  - Whitelist/blacklist by hostname, IP, OS                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ Poll every 60s (phoneHome)
                              ▼
     ┌────────────┬────────────┬────────────┬────────────┐
     │    UF      │    UF      │    UF      │    UF      │
     │ (Windows)  │ (Linux)    │ (DC)       │ (WebSrv)   │
     └────────────┴────────────┴────────────┴────────────┘

serverclass.conf

conf serverclass.conf - Deployment Server 
# Global settings
[global]
repositoryLocation = $SPLUNK_HOME/etc/deployment-apps

# Server Class: All Windows Servers
[serverClass:Windows_Servers]
whitelist.0 = *.company.com
filterType = whitelist
# Match Windows hosts
machineTypesFilter = windows-x64

[serverClass:Windows_Servers:app:Splunk_TA_windows]
restartSplunkd = true

[serverClass:Windows_Servers:app:org_outputs]
restartSplunkd = true

# Server Class: Domain Controllers
[serverClass:Domain_Controllers]
whitelist.0 = dc*.company.com
filterType = whitelist

[serverClass:Domain_Controllers:app:org_windows_dc]
restartSplunkd = true

# Server Class: Linux Servers
[serverClass:Linux_Servers]
whitelist.0 = *.company.com
filterType = whitelist
machineTypesFilter = linux-x86_64

[serverClass:Linux_Servers:app:Splunk_TA_nix]
restartSplunkd = true

# Server Class: Heavy Forwarders
[serverClass:Heavy_Forwarders]
whitelist.0 = hf*.company.com
filterType = whitelist

[serverClass:Heavy_Forwarders:app:Splunk_TA_paloalto]
restartSplunkd = true

[serverClass:Heavy_Forwarders:app:org_syslog_inputs]
restartSplunkd = true

HTTP Event Collector (HEC)

REST API for sending data directly to Splunk without a forwarder.

Enable HEC

conf inputs.conf - HEC Configuration 
[http]
disabled = 0
port = 8088
enableSSL = 1
dedicatedIoThreads = 2

[http://my_hec_token]
disabled = 0
token = your-hec-token-here
index = main
indexes = main, security, application
sourcetype = json
# Optionally restrict by source IP
# allowedNetworks = 10.0.0.0/8, 192.168.0.0/16

Sending Data via HEC

Bash cURL Examples 
# Single event
curl -k https://splunk.company.com:8088/services/collector/event \
    -H "Authorization: Splunk your-hec-token" \
    -d '{"event": "User login successful", "sourcetype": "myapp", "index": "main"}'

# Event with timestamp and fields
curl -k https://splunk.company.com:8088/services/collector/event \
    -H "Authorization: Splunk your-hec-token" \
    -d '{
        "time": 1704812345,
        "host": "webserver01",
        "source": "myapp",
        "sourcetype": "json",
        "index": "application",
        "event": {
            "action": "login",
            "user": "jsmith",
            "status": "success",
            "src_ip": "192.168.1.100"
        }
    }'

# Raw data (unstructured)
curl -k https://splunk.company.com:8088/services/collector/raw \
    -H "Authorization: Splunk your-hec-token" \
    -d 'Jan  9 14:32:15 webserver sshd[12345]: Accepted publickey for jsmith'
Python Python HEC Client 
import requests
import json
import time

HEC_URL = "https://splunk.company.com:8088/services/collector/event"
HEC_TOKEN = "your-hec-token"

def send_to_splunk(event_data, sourcetype="json", index="main"):
    payload = {
        "time": time.time(),
        "host": "myapp-server",
        "sourcetype": sourcetype,
        "index": index,
        "event": event_data
    }
    
    headers = {
        "Authorization": f"Splunk {HEC_TOKEN}",
        "Content-Type": "application/json"
    }
    
    response = requests.post(
        HEC_URL,
        headers=headers,
        data=json.dumps(payload),
        verify=False  # Set True in production with valid cert
    )
    
    return response.status_code == 200

# Usage
send_to_splunk({
    "action": "payment_processed",
    "amount": 150.00,
    "user_id": "user123"
}, sourcetype="payment_api", index="transactions")

Metrics to Monitor

SPL Forwarder Health Queries 
// Forwarders not sending data (last 15 min)
| metadata type=hosts index=*
| eval last_seen=strftime(lastTime, "%Y-%m-%d %H:%M:%S")
| where lastTime < relative_time(now(), "-15m")
| table host, last_seen, totalCount
| sort -lastTime

// Data volume by forwarder
index=_internal source=*metrics.log group=tcpin_connections
| stats sum(kb) as KB by hostname
| eval MB = round(KB/1024, 2)
| sort -MB

// Forwarder queue health
index=_internal source=*metrics.log group=queue
| stats avg(current_size_kb) as avg_queue_kb by host, name
| where avg_queue_kb > 100

// Missing forwarders (expected vs actual)
| inputlookup expected_forwarders.csv
| join type=left host [
    | metadata type=hosts index=* 
    | eval status="active"
]
| fillnull value="missing" status
| where status="missing"

Interview Preparation

How would you deploy and manage 5,000 Universal Forwarders across your enterprise?

Deployment approach:

I'd use automated deployment through existing tools: SCCM/Intune for Windows, Ansible/Puppet/Chef for Linux. The installer would be pre-configured with deployment server address in deploymentclient.conf. For Windows, I'd create an MSI transform or use command-line parameters. For Linux, a package with post-install script.

Management with Deployment Server:

Set up a dedicated Deployment Server (or cluster for scale). Organize apps in deployment-apps directory by function: org_outputs (forwarding config), org_base (common settings), platform-specific TAs. Create server classes based on logical groupings: by OS, by function (DCs, web servers, databases), by location.

Monitoring at scale:

Use internal logs to monitor forwarder health. Create alerts for forwarders not checking in, queue backlogs, or connection failures. The Monitoring Console provides forwarder dashboards out of the box. I'd also maintain an expected forwarders list to catch missing installations.

Scaling considerations:

At 5,000 forwarders, the Deployment Server can handle it, but I'd stagger the phoneHome interval (add jitter) to avoid thundering herd. Consider multiple Deployment Servers for geographic distribution.
