Data Connectors

Data connectors are how you get data into Microsoft Sentinel. With 200+ native connectors plus custom options, understanding which to use and how to configure them is essential for effective security monitoring.

Key Concept

Data connectors handle authentication, data collection, and transformation to get security data into your Log Analytics workspace. Choose the right connector for your source - it affects cost, latency, and data fidelity.

Connector Categories

Microsoft Services

M365, Azure AD, Defender XDR, Azure Activity - native integration, often free ingestion

Cloud Providers

AWS CloudTrail, GCP, Azure services - API-based collection with polling

Security Products

Palo Alto, CrowdStrike, Fortinet - vendor-specific parsers and schemas

On-Premises

Windows Events, Syslog, CEF - requires AMA agent or log forwarder

Common Connector Setup

Windows Security Events via AMA

DCR for Windows Security Events
// Data Collection Rule - JSON snippet
{
  "dataSources": {
    "windowsEventLogs": [
      {
        "streams": ["Microsoft-SecurityEvent"],
        "xPathQueries": [
          "Security!*[System[(EventID=4624 or EventID=4625 or EventID=4688)]]"
        ],
        "name": "SecurityEventDataSource"
      }
    ]
  }
}

// Common Security Event IDs to collect:
// 4624 - Successful logon
// 4625 - Failed logon  
// 4688 - Process creation
// 4672 - Special privileges assigned
// 4720 - User account created
// 4726 - User account deleted

Syslog/CEF Collection

1

Deploy Log Forwarder VM

Create Linux VM with AMA agent. This receives syslog from network devices and forwards to Sentinel.

2

Configure DCR

Create Data Collection Rule for Syslog facilities (auth, authpriv, local0-7) and severity levels.

3

Point Sources to Forwarder

Configure firewalls, routers, and servers to send syslog to forwarder VM IP on port 514.

Interview Questions

Answer: Use the Azure Monitor Agent (AMA) with Data Collection Rules:

  1. Install AMA on Windows servers (via Azure Arc for non-Azure VMs)
  2. Create Data Collection Rule specifying which Event IDs to collect
  3. Associate DCR with the servers
  4. Data flows to SecurityEvent table in Log Analytics

Best practice: Filter Event IDs in DCR to reduce costs - don't collect everything.

Syslog: Raw log format, varies by vendor, goes to Syslog table

CEF (Common Event Format): Standardized format with predefined fields (src, dst, act, etc.), goes to CommonSecurityLog table

Which to use: If your device supports CEF, use it - structured data is easier to query and correlate.
