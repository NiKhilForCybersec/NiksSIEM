Retention & Storage Tiers

Complete guide to Log Analytics storage tiers, retention configuration, and cost optimization strategies for Microsoft Sentinel.

Storage Tier Overview

Analytics Logs (Default)

What: Full KQL query capability, analytics rules, hunting queries

Retention: 30-730 days (default 90 days for Sentinel tables)

Cost: Higher ingestion + retention cost, but full functionality

Use for: Security events, alerts, incidents, active investigation data

Basic Logs

What: Reduced cost, limited query capabilities

Retention: 8 days interactive + up to 4383 days archive

Cost: ~65% less than Analytics tier

Limitations: No scheduled alerts, limited KQL (single table, no joins)

Use for: Verbose logs, debug data, low-value telemetry

Archive Tier

What: Long-term retention, search jobs required to query

Retention: Up to 12 years (4383 days)

Cost: Lowest storage cost, pay per search job

Access: Must run "Search Job" to restore data for querying

Use for: Compliance retention, historical investigation, audit logs

Configure Retention

Per-Table Retention (Azure Portal)

Log Analytics workspace → Tables → [Select table] → Manage table

SettingDescriptionRange
Interactive retention Days data is queryable with full KQL 4-730 days
Total retention Total days stored (includes archive) 4-4383 days (12 years)
Table plan Analytics or Basic Analytics (default) / Basic

Configure via CLI/PowerShell

# Set retention for specific table (Azure CLI) az monitor log-analytics workspace table update \ --resource-group rg-sentinel \ --workspace-name sentinel-workspace \ --name SecurityEvent \ --retention-time 90 \ --total-retention-time 365 # Set table to Basic tier az monitor log-analytics workspace table update \ --resource-group rg-sentinel \ --workspace-name sentinel-workspace \ --name ContainerLog \ --plan Basic # PowerShell Update-AzOperationalInsightsTable ` -ResourceGroupName "rg-sentinel" ` -WorkspaceName "sentinel-workspace" ` -TableName "SecurityEvent" ` -RetentionInDays 90 ` -TotalRetentionInDays 365

Querying Archived Data

Create Search Job

Log Analytics workspace → Logs → Search job mode

// Search job query - searches archived data // Results stored in new table: SearchResults_[jobid] SecurityEvent | where TimeGenerated between (datetime(2023-01-01) .. datetime(2023-03-31)) | where EventID == 4625 | project TimeGenerated, Computer, Account, IpAddress

How search jobs work:

  1. Submit query against archived time range
  2. Job runs async, restoring data from archive
  3. Results stored in SearchResults table (retained 30 days)
  4. Query results table for analysis

CLI Method:

# Create search job via CLI az monitor log-analytics workspace search-job create \ --resource-group rg-sentinel \ --workspace-name sentinel-workspace \ --name "HistoricalInvestigation" \ --query "SecurityEvent | where EventID == 4625" \ --start-time "2023-01-01T00:00:00Z" \ --end-time "2023-03-31T23:59:59Z"

Data Export (Long-term Archive)

Export to Storage Account

Log Analytics workspace → Data Export → New export rule

Destination:Storage Account or Event Hub
Tables:Select tables to export
Format:JSON (one file per hour)
Retention:Configure in Storage Account lifecycle

Use cases:

  • Compliance requiring 7+ years retention
  • Lower cost than Log Analytics archive
  • Integration with external SIEM or data lake

Cost Optimization Strategies

Quick Wins

  1. Use Basic tier for verbose, low-security-value logs (Perf, ContainerLog, traces)
  2. Shorten retention on high-volume tables (CommonSecurityLog, Syslog)
  3. Filter at ingestion using DCR transformations to drop unwanted events
  4. Use archive tier instead of interactive for compliance-only data
  5. Data export to Storage Account for cheapest long-term storage

Monitor Costs

// Daily ingestion by table Usage | where TimeGenerated > ago(30d) | where DataType != "" | summarize IngestedGB = sum(Quantity) / 1000 by DataType | sort by IngestedGB desc // Billable data by solution Usage | where TimeGenerated > ago(30d) | where IsBillable == true | summarize BillableGB = sum(Quantity) / 1000 by Solution | sort by BillableGB desc

Interview Questions

Three tiers:

  1. Analytics Logs: Default tier, full KQL queries, analytics rules, hunting. 30-730 days retention. Use for security-relevant data.
  2. Basic Logs: 65% cheaper, limited KQL (no joins, single table). 8 days interactive + archive. Use for verbose, low-value logs.
  3. Archive: Lowest cost storage for retention beyond interactive period. Requires search jobs to query (async restore). Up to 12 years.

Best practice: Use Analytics for security tables (SecurityEvent, Alerts), Basic for verbose ops logs (ContainerLog, Perf), Archive for compliance retention.

Search Jobs:

  1. Create search job with time range that includes archived period
  2. Job runs asynchronously, restoring data from archive
  3. Results stored in SearchResults_[jobid] table
  4. Query results table for analysis (retained 30 days)

Note: Search jobs have costs (per GB scanned) and take time depending on data volume. Cannot use for real-time alerting.

Additional Interview Questions

Systematic troubleshooting approach:

  1. Identify the symptom: What exactly is not working as expected?
  2. Check connectivity: Verify network paths and authentication
  3. Review logs: Check internal/audit logs for errors
  4. Validate configuration: Compare against documented standards
  5. Test incrementally: Isolate the problem component
  6. Document findings: Record what worked and what didn't

Key performance indicators:

  • Availability: Uptime percentage, service availability
  • Performance: Query response time, processing latency
  • Volume: Events per second, data ingestion rate
  • Quality: Parse success rate, enrichment coverage
  • Efficiency: Resource utilization, cost per GB

High availability strategies:

  • Redundancy: Multiple instances of critical components
  • Load balancing: Distribute traffic across instances
  • Failover: Automatic switchover on failure
  • Monitoring: Health checks and alerting
  • DR Planning: Documented recovery procedures
