Data Collection Rules & Endpoints

DCR (Data Collection Rules) and DCE (Data Collection Endpoints) are the foundation of Azure Monitor's data pipeline. Master these components to control data ingestion, transformation, and routing.

Understanding DCR and DCE

DCE (Data Collection Endpoint)
  • Network endpoint for data ingestion
  • Regional Azure resource
  • Provides HTTPS ingestion URL
  • Required for custom logs
  • "Where data enters Azure"
DCR (Data Collection Rule)
  • Defines WHAT to collect
  • Specifies WHERE to send
  • Contains transformation logic
  • Can filter/modify data
  • "Rules for processing data"

DCR Components

Data Sources
What data to collect
  • Syslog: Facility, severity filters
  • Windows Events: XPath queries
  • Performance: Counters, sample rate
  • Custom Logs: File paths, formats
Streams
Data format definition
  • Built-in streams: Microsoft-Syslog, Microsoft-SecurityEvent
  • Custom streams: Custom-MyTable_CL
  • Defines column schema
  • Input format for transformation
Transformations
KQL to modify data
  • Filter: Drop unwanted events
  • Parse: Extract fields from text
  • Enrich: Add calculated fields
  • Project: Select/rename columns
Destinations
Where to send data
  • Log Analytics: Standard tables (Syslog, SecurityEvent)
  • Custom tables: YourTable_CL
  • Multiple destinations: Send to multiple workspaces

DCR Transformation Examples

Example 1: Filter Non-Security Syslog

KQL Transform - Drop Debug Events
// Drop debug/info events to save costs
source
| where SeverityLevel <= 4  // Keep only Warning (4) and above
| project-away Computer  // Remove unnecessary field

Example 2: Parse Custom Log Format

KQL Transform - Parse JSON Fields
// Parse JSON from raw message
source
| extend parsed = parse_json(RawData)
| extend 
    UserName = tostring(parsed.user),
    Action = tostring(parsed.action),
    SourceIP = tostring(parsed.source_ip),
    Timestamp = todatetime(parsed.timestamp)
| project TimeGenerated, UserName, Action, SourceIP

Example 3: Enrich with Geo-IP

KQL Transform - Add Geolocation
// Add geo-location data
source
| extend GeoInfo = geo_info_from_ip_address(SourceIP)
| extend 
    Country = tostring(GeoInfo.country),
    City = tostring(GeoInfo.city),
    Latitude = toreal(GeoInfo.latitude),
    Longitude = toreal(GeoInfo.longitude)
| project-away GeoInfo

DCR JSON Structure

ARM Template - DCR Definition
{
  "properties": {
    "dataSources": {
      "syslog": [
        {
          "streams": ["Microsoft-Syslog"],
          "facilityNames": ["auth", "authpriv", "local0"],
          "logLevels": ["Warning", "Error", "Critical"],
          "name": "syslogDataSource"
        }
      ]
    },
    "destinations": {
      "logAnalytics": [
        {
          "workspaceResourceId": "/subscriptions/.../workspaces/myworkspace",
          "name": "myWorkspace"
        }
      ]
    },
    "dataFlows": [
      {
        "streams": ["Microsoft-Syslog"],
        "destinations": ["myWorkspace"],
        "transformKql": "source | where SeverityLevel <= 4",
        "outputStream": "Microsoft-Syslog"
      }
    ]
  }
}

Enterprise DCR Strategy

Decision Tree: DCR Design Approach

Decision Tree: DCR Architecture Selection
═══════════════════════════════════════════

START: What data source type?
│
├─► STANDARD LOGS (Syslog, SecurityEvent, Perf)
│   │
│   └─► Do you need transformation?
│       ├─► YES → Create workspace transformation DCR
│       │         • Applies to ALL data in that table
│       │         • Good for org-wide filtering
│       │         • Cannot be per-source selective
│       │
│       └─► NO → Use built-in DCR (auto-created by connector)
│                • Minimal config required
│                • Standard schema maintained
│
├─► CUSTOM LOGS (Application logs, custom formats)
│   │
│   ├─► Is source AMA-based (VM with agent)?
│   │   └─► Create DCR with:
│   │       • File path pattern (/var/log/app/*.log)
│   │       • Custom stream definition
│   │       • Transformation KQL
│   │       • DCE NOT required (AMA knows workspace)
│   │
│   └─► Is source API-based (application sending via HTTP)?
│       └─► Create DCR + DCE:
│           • DCE = ingestion URL endpoint
│           • DCR = schema + transform + destination
│           • App calls: POST https://<DCE>/dataCollectionRules/<DCR>/streams/<stream>
│
├─► MULTI-DESTINATION (same data to multiple workspaces)
│   │
│   └─► Single DCR with multiple destinations:
│       • Production workspace (full data)
│       • Dev/Test workspace (sampled data)
│       • Regional workspace (filtered by location)
│
└─► HIGH-VOLUME REDUCTION (firewall, proxy, NetFlow)
    │
    └─► Transformation priorities:
        1. Filter first (where clause) → biggest impact
        2. Remove fields (project-away) → moderate impact
        3. Aggregate if applicable (summarize) → complex but effective
        4. Sample as last resort (rand() < 0.1) → loses data

DCR Troubleshooting Decision Tree

Decision Tree: DCR Not Working
═══════════════════════════════

SYMPTOM: Data not arriving in workspace
│
├─► Check: Is AMA installed and running?
│   │
│   ├─► LINUX: systemctl status azuremonitoragent
│   │   └─► Not running → Check /var/opt/microsoft/azuremonitoragent/log/
│   │
│   └─► WINDOWS: Get-Service "Azure Monitor Agent"
│       └─► Not running → Check Event Viewer → Applications
│
├─► Check: Is DCR associated with the VM?
│   │
│   └─► Portal: VM → Extensions → AMA → View DCR associations
│       OR: az monitor data-collection rule association list --resource <vm-id>
│       └─► No association → Associate DCR to VM
│
├─► Check: Is DCR configuration valid?
│   │
│   ├─► Data sources defined? → Check dataSources section
│   ├─► Streams mapped? → Check dataFlows section  
│   ├─► Destinations correct? → Check destinations → workspaceId
│   └─► Transformation valid? → Test KQL in Log Analytics first
│
├─► Check: Network connectivity (for DCE)
│   │
│   └─► Can VM reach DCE endpoint?
│       • Test: curl https://<dce-name>.<region>.ingest.monitor.azure.com
│       • Check NSG rules for outbound 443
│       • Check firewall/proxy allows Azure Monitor endpoints
│
└─► Check: Transformation errors
    │
    └─► Logs Ingestion API errors surface in:
        • AzureMetrics table (DCR metrics)
        • _LogOperation table (ingestion failures)
        
        Common KQL errors:
        • "column not found" → Schema mismatch
        • "invalid cast" → Type conversion failed
        • "query timeout" → Transform too complex

What Enterprises Usually Miss

Critical DCR/DCE Gaps

  • No DCR versioning: ARM templates not in Git, no rollback capability when transform breaks production
  • Testing transforms in prod: KQL errors in DCR silently drop data - always test in dev workspace first
  • Missing DCE for custom logs: Forgetting DCE when using Logs Ingestion API causes 403 errors
  • Over-filtering: Aggressive transforms that drop security-relevant events to save costs
  • No ingestion monitoring: Not alerting on _LogOperation failures or DCR metrics anomalies
  • Schema drift: Source app updates log format, DCR transform breaks, no one notices for weeks
  • Single DCE bottleneck: All custom logs through one DCE - no regional distribution or failover
  • Workspace transformation conflicts: Workspace-level DCR transform + table-level transform = unexpected behavior
  • DCR sprawl: 50+ DCRs with no naming convention, no owner documentation, impossible to audit
  • No capacity planning: DCE has limits (1 GB/min default) - high-volume sources can hit throttling

DCR Health Monitoring Queries

KQL - Monitor DCR Ingestion Health
// Query 1: Detect DCR ingestion failures
_LogOperation
| where Category == "Ingestion"
| where Level == "Error" or Level == "Warning"
| summarize FailureCount = count() by Operation, Detail, bin(TimeGenerated, 1h)
| order by TimeGenerated desc

// Query 2: Volume per DCR (requires custom tracking)
// Add a field in your DCR transform to track source
CommonSecurityLog
| where TimeGenerated > ago(24h)
| summarize EventCount = count(), 
            DataVolumeMB = sum(_BilledSize) / 1024 / 1024
    by DeviceVendor, DeviceProduct, bin(TimeGenerated, 1h)
| order by DataVolumeMB desc

// Query 3: Detect sudden volume drops (ingestion gap)
let baseline = Syslog
    | where TimeGenerated between (ago(7d) .. ago(1d))
    | summarize AvgHourly = count() / (24 * 6);
Syslog
| where TimeGenerated > ago(1h)
| summarize CurrentCount = count()
| extend Baseline = toscalar(baseline)
| extend PercentOfBaseline = CurrentCount * 100.0 / Baseline
| where PercentOfBaseline < 50  // Alert if <50% of normal

Interview Questions

The data flow:

  1. AMA (Azure Monitor Agent): Collects data from source (VM)
  2. DCR (Data Collection Rule): Tells AMA what to collect, how to transform, where to send
  3. DCE (Data Collection Endpoint): Network endpoint where data is sent

Relationships:

  • AMA is associated with one or more DCRs
  • DCR can have a DCE for custom logs (optional for standard logs)
  • Multiple DCRs can share one DCE

Cost reduction strategies:

  • Filter at source: Drop events before they're ingested (saves ingestion cost)
  • Remove fields: project-away unnecessary columns
  • Aggregate: Summarize high-volume data

Example impact:

  • Firewall: Drop "allow" traffic → 80-90% reduction
  • Syslog: Filter info/debug → 60-70% reduction
  • Field removal: 10-20% additional savings
