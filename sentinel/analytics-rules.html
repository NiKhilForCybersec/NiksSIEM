Analytics Rules

Analytics rules are the detection engine of Microsoft Sentinel. They run KQL queries on schedule or in near-real-time, generating alerts that become incidents for investigation.

Core Concept

Analytics rules are automated detection logic that continuously monitor your data for threats. When a rule triggers, it creates an alert. Related alerts are grouped into incidents for analyst investigation.

Rule Types

Scheduled Rules

Custom KQL queries running every 5 min to 14 days. Most flexible - you control the logic completely.

NRT (Near-Real-Time)

Run every minute with ~1 min latency. Limited to single-table queries. For time-critical detections.

Fusion

ML-based correlation detecting multi-stage attacks. Cannot customize - enable/disable only.

ML Behavior Analytics

Anomaly detection based on learned baselines. UEBA rules for user/entity behavior changes.

Creating Scheduled Rules

Example: Brute Force Detection Rule
// Rule Query
let threshold = 10;
let timeframe = 1h;
SigninLogs
| where TimeGenerated > ago(timeframe)
| where ResultType != 0  // Failed logins
| summarize 
    FailedAttempts = count(),
    FirstAttempt = min(TimeGenerated),
    LastAttempt = max(TimeGenerated),
    IPAddresses = make_set(IPAddress),
    TargetApps = make_set(AppDisplayName)
    by UserPrincipalName
| where FailedAttempts > threshold
| extend AccountName = tostring(split(UserPrincipalName, "@")[0])
| extend UPNSuffix = tostring(split(UserPrincipalName, "@")[1])

// Entity Mapping (in rule config):
// - Account: AccountName, UPNSuffix
// - IP: IPAddresses

Interview Questions

Scheduled rules:

  • Run every 5 min to 14 days
  • Can query multiple tables with joins
  • Full KQL capabilities
  • Use for: Most detection scenarios

NRT rules:

  • Run every minute, ~1 min latency
  • Single table only, no joins
  • Limited KQL (no let statements)
  • Use for: Critical, time-sensitive detections

Strategies:

  1. Tune thresholds: Analyze baseline activity, set realistic limits
  2. Add exclusions: Whitelist known good IPs, users, processes
  3. Time-based filtering: Exclude maintenance windows
  4. Correlation: Require multiple indicators, not single events
  5. Anomaly-based: Use statistical baselines instead of static thresholds
