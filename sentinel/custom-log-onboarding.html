Custom Log Onboarding - Complete Guide

Complete guide to ingesting custom logs into Microsoft Sentinel using DCR (Data Collection Rules), custom tables, and the Log Ingestion API.

Modern vs Legacy Methods

Modern (Recommended): DCR-based ingestion with Azure Monitor Agent (AMA) or Log Ingestion API

Legacy: Log Analytics agent (MMA) with Custom Logs - being deprecated

This guide focuses on the modern DCR-based approach.

PHASE 1 Planning & Prerequisites

1.1 Prerequisites

  • Log Analytics Workspace with Sentinel enabled
  • Azure subscription with Contributor role
  • Sample log files (10-20 events minimum)
  • Understanding of log format (JSON, CSV, syslog, custom)

1.2 Choose Ingestion Method

MethodUse CaseComponents
AMA + DCR File-based logs on VMs Azure Monitor Agent, DCR, DCE
Log Ingestion API API/script-based ingestion DCR, DCE, Entra ID App
Logstash Existing Logstash pipelines Logstash + Azure plugin
Syslog via AMA RFC 3164/5424 syslog AMA on Linux collector
PHASE 2 Create Custom Table

Azure Portal Method

Log Analytics workspace → Tables → Create → New custom log (DCR-based)

1
Table name: MyApp_CL

Custom tables must end with _CL suffix

2
Upload sample data or define schema

Upload JSON/CSV sample or manually define columns

ARM Template / API Method

// Create custom table via API PUT https://management.azure.com/subscriptions/{subscriptionId}/resourceGroups/{resourceGroup}/providers/Microsoft.OperationalInsights/workspaces/{workspaceName}/tables/MyApp_CL?api-version=2022-10-01 { "properties": { "schema": { "name": "MyApp_CL", "columns": [ {"name": "TimeGenerated", "type": "datetime"}, {"name": "User", "type": "string"}, {"name": "Action", "type": "string"}, {"name": "SourceIP", "type": "string"}, {"name": "Result", "type": "string"}, {"name": "RawData", "type": "string"} ] }, "retentionInDays": 90, "totalRetentionInDays": 365 } }
PHASE 3 Create Data Collection Endpoint (DCE)

Monitor → Data Collection Endpoints → Create

Name:dce-myapp-eastus
Region:East US (same as workspace)
Resource group:rg-sentinel

After creation, note these values:

  • Logs Ingestion URI: https://dce-myapp-eastus-xxxx.eastus-1.ingest.monitor.azure.com
  • Resource ID: /subscriptions/.../dataCollectionEndpoints/dce-myapp-eastus
PHASE 4 Create Data Collection Rule (DCR)

For Log Ingestion API

Monitor → Data Collection Rules → Create

// DCR for Log Ingestion API { "location": "eastus", "properties": { "dataCollectionEndpointId": "/subscriptions/.../dataCollectionEndpoints/dce-myapp-eastus", "streamDeclarations": { "Custom-MyApp_CL": { "columns": [ {"name": "TimeGenerated", "type": "datetime"}, {"name": "User", "type": "string"}, {"name": "Action", "type": "string"}, {"name": "SourceIP", "type": "string"}, {"name": "Result", "type": "string"}, {"name": "RawData", "type": "string"} ] } }, "dataSources": {}, "destinations": { "logAnalytics": [ { "workspaceResourceId": "/subscriptions/.../workspaces/sentinel-workspace", "name": "sentinel-workspace" } ] }, "dataFlows": [ { "streams": ["Custom-MyApp_CL"], "destinations": ["sentinel-workspace"], "transformKql": "source | extend TimeGenerated = now()", "outputStream": "Custom-MyApp_CL" } ] } }

For File-based Logs (AMA)

// DCR for file monitoring with AMA { "properties": { "dataSources": { "logFiles": [ { "streams": ["Custom-MyApp_CL"], "filePatterns": ["/var/log/myapp/*.log"], "format": "text", "name": "myapp-logs" } ] }, "destinations": { "logAnalytics": [ { "workspaceResourceId": "/subscriptions/.../workspaces/sentinel-workspace", "name": "sentinel-workspace" } ] }, "dataFlows": [ { "streams": ["Custom-MyApp_CL"], "destinations": ["sentinel-workspace"], "transformKql": "source | parse RawData with * 'USER=' User ' ACTION=' Action ' SRC=' SourceIP ' RESULT=' Result | extend TimeGenerated = now()", "outputStream": "Custom-MyApp_CL" } ] } }
PHASE 5 Configure Data Ingestion

5A. Log Ingestion API (for custom scripts)

Step 1: Create Entra ID App Registration

Entra ID → App registrations → New registration

1
Name: log-ingestion-app, create client secret
2
Grant "Monitoring Metrics Publisher" role on DCR

Step 2: Send Logs via Python

import requests import json from datetime import datetime from azure.identity import ClientSecretCredential from azure.monitor.ingestion import LogsIngestionClient # Configuration tenant_id = "your-tenant-id" client_id = "your-app-client-id" client_secret = "your-client-secret" dce_endpoint = "https://dce-myapp-eastus-xxxx.eastus-1.ingest.monitor.azure.com" dcr_immutable_id = "dcr-xxxxxxxx" stream_name = "Custom-MyApp_CL" # Authenticate credential = ClientSecretCredential(tenant_id, client_id, client_secret) client = LogsIngestionClient(endpoint=dce_endpoint, credential=credential) # Prepare logs logs = [ { "TimeGenerated": datetime.utcnow().isoformat() + "Z", "User": "jsmith", "Action": "login", "SourceIP": "10.0.1.50", "Result": "success" }, { "TimeGenerated": datetime.utcnow().isoformat() + "Z", "User": "admin", "Action": "config_change", "SourceIP": "10.0.1.51", "Result": "success" } ] # Send logs client.upload(rule_id=dcr_immutable_id, stream_name=stream_name, logs=logs) print("Logs sent successfully")

3
Verify file path in DCR matches actual log location
PHASE 6 Data Transformation (KQL in DCR)

Transformation Examples

Parse Key-Value Logs:

source | parse RawData with * "USER=" User " ACTION=" Action " SRC=" SourceIP " RESULT=" Result | extend TimeGenerated = now() | project TimeGenerated, User, Action, SourceIP, Result

Parse JSON Logs:

source | extend parsed = parse_json(RawData) | extend TimeGenerated = todatetime(parsed.timestamp), User = tostring(parsed.user), Action = tostring(parsed.action), SourceIP = tostring(parsed.src_ip), Result = tostring(parsed.result) | project TimeGenerated, User, Action, SourceIP, Result

Parse CSV Logs:

source | parse RawData with Timestamp:datetime "," User:string "," Action:string "," SourceIP:string "," Result:string | extend TimeGenerated = Timestamp | project TimeGenerated, User, Action, SourceIP, Result

Filter and Enrich:

source | where RawData !contains "DEBUG" // Filter out debug logs | extend TimeGenerated = now(), Environment = "Production", DataSource = "MyApp" | project TimeGenerated, Environment, DataSource, RawData
PHASE 7 Verify Data Ingestion
Sentinel → Logs
// Check if data exists MyApp_CL | take 10 // Check data volume MyApp_CL | where TimeGenerated > ago(1h) | summarize count() by bin(TimeGenerated, 5m) | render timechart // Verify field extraction MyApp_CL | where TimeGenerated > ago(1h) | project TimeGenerated, User, Action, SourceIP, Result | take 20
PHASE 8 Create Analytics Rules

Example Detection Rule

// Multiple failed logins from same IP MyApp_CL | where TimeGenerated > ago(5m) | where Action == "login" and Result == "failure" | summarize FailedAttempts = count() by SourceIP, User | where FailedAttempts > 10

Interview Questions

  1. Create custom table: Tables → New custom log (DCR-based) → Define schema ending with _CL
  2. Create DCE: Data Collection Endpoint in same region as workspace
  3. Create DCR: Data Collection Rule defining source, transformation KQL, destination
  4. Configure ingestion: Either associate AMA VMs with DCR, or use Log Ingestion API with Entra ID app
  5. Transform: Use KQL in DCR dataFlows.transformKql to parse and normalize
  6. Verify: Query custom table in Sentinel Logs

Key components: DCE (endpoint), DCR (rules), Custom table (destination), AMA or API (source)

Troubleshooting Guide

Common Issues and Solutions

IssuePossible CauseSolution
Data not appearing Collection or parsing issue Check collector status, verify parser configuration
Slow performance Resource constraints or inefficient queries Review resource utilization, optimize queries
Authentication failures Credential or permission issues Verify credentials, check RBAC configuration
Missing fields Parser not extracting fields Review parsing rules, update field extraction
Duplicate events Multiple collection paths Audit data sources, implement deduplication

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.
