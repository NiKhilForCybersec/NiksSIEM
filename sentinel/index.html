Microsoft Sentinel

Microsoft Sentinel is a scalable, cloud-native SIEM (Security Information and Event Management) and SOAR (Security Orchestration, Automation, and Response) solution built on Azure. It delivers intelligent security analytics across your enterprise, providing a single solution for alert detection, threat visibility, proactive hunting, and threat response.

Prerequisites

Azure subscription with Contributor role
Log Analytics workspace understanding
Basic networking (TCP/IP, DNS, firewalls)
Security operations fundamentals

Foundational ~45 min read

Why Microsoft Sentinel?

Key Value Proposition

Sentinel eliminates infrastructure management entirely. No servers to deploy, no storage to provision, automatic scaling during security incidents. You pay only for what you ingest. For Microsoft-centric organizations, free ingestion of M365, Azure AD, and Defender data makes it extremely cost-effective.

Core Capabilities

200+ Data Connectors

Native connectors for Microsoft services, AWS, GCP, firewalls (Palo Alto, Fortinet, Cisco), and custom sources via CEF/Syslog/REST API

KQL Query Language

Kusto Query Language for threat hunting, investigation, detection rules, and workbook visualizations. Extremely powerful and fast.

Analytics Rules

Scheduled queries, Near-Real-Time (NRT) rules, ML-based anomaly detection, Microsoft Threat Intelligence, and Fusion correlation

Automation (SOAR)

Logic Apps playbooks with 400+ connectors for automated enrichment, containment, ticketing, and notification workflows

UEBA & ML

User and Entity Behavior Analytics, anomaly detection, and machine learning models for advanced threat detection

Workbooks

Interactive dashboards with pre-built templates. Create custom visualizations combining KQL queries with rich formatting

Pricing Model

Cost Understanding is Critical

Sentinel costs consist of two components: (1) Log Analytics data ingestion, and (2) Sentinel analytics charges. Both are based on GB ingested. Understanding this is essential for cost optimization.

Pricing Tier Log Analytics Sentinel Total/GB Best For
Pay-As-You-Go ~$2.76 ~$2.46 ~$5.22 Variable, <100GB/day
Commitment 100GB ~$1.96 ~$1.96 ~$3.92 Predictable 100GB+/day
Commitment 500GB ~$1.57 ~$1.57 ~$3.14 Enterprise 500GB+/day
Commitment 1TB+ ~$1.25 ~$1.25 ~$2.50 Large enterprise

Free Data Sources (No Sentinel Charges)

These sources are free to ingest into Sentinel:

  • Azure Activity Logs - All Azure control plane operations
  • Office 365 Audit Logs - With E5 or relevant M365 license
  • Microsoft Defender XDR - Alerts and incidents (raw logs still charged)
  • Azure AD Sign-in/Audit - With P1/P2 or relevant license
  • Microsoft Defender for Cloud - Alerts (free tier)

Cost Optimization Strategies

Commitment Tiers

Switch to commitment tiers once you have predictable volume. 100GB tier saves ~25%, 500GB saves ~40%

Basic Logs

Use Basic Logs tier for high-volume, low-value data. 8x cheaper but limited query capabilities

Archive Tier

Move data older than 90 days to archive. Much cheaper long-term storage with restore capability

DCR Filtering

Use Data Collection Rules to filter/transform at ingestion time. Drop unnecessary fields before storage

Interview Questions

Answer: Microsoft Sentinel is a cloud-native SIEM and SOAR solution built on Azure Log Analytics. Key differences from traditional SIEM:

  • No infrastructure: Fully managed - no servers, storage, or capacity planning
  • Elastic scaling: Automatically scales during incidents without pre-provisioning
  • Pay-per-use: Only pay for GB ingested, no licensing tiers
  • Native integrations: Deep integration with M365, Azure AD, Defender products
  • Built-in SOAR: Logic Apps automation included, not separate product
  • ML-powered: Fusion detection, UEBA, and anomaly detection out-of-box

Trade-offs: Cloud-only (no on-prem option), requires Azure subscription, KQL learning curve

Answer: Sentinel cost optimization strategies:

1. Commitment Tiers:

  • Switch from Pay-As-You-Go once volume is predictable
  • 100GB/day tier saves ~25%, 500GB saves ~40%
  • Can increase tier anytime, decrease after 31 days

2. Data Tiering:

  • Analytics Logs: Full query capability, default tier
  • Basic Logs: 8x cheaper, limited queries (good for verbose logs)
  • Archive: Very cheap long-term storage, restore when needed

3. Ingestion Optimization:

  • Use DCRs to filter/drop unnecessary fields at ingestion
  • Leverage free data sources (Azure Activity, M365 with license)
  • Avoid ingesting duplicate data from multiple connectors

4. Query Optimization:

  • Use summarize to reduce data scanned
  • Time-bound queries appropriately
  • Avoid expensive regex on large datasets

Answer: Sentinel has four types of analytics rules:

1. Scheduled Rules:

  • KQL query runs on schedule (5 min to 14 days)
  • Most common type, fully customizable
  • Configure alert grouping, entity mapping, MITRE mapping

2. NRT (Near-Real-Time) Rules:

  • Run every minute with ~1 minute latency
  • Limited to single table queries
  • Use for critical, time-sensitive detections

3. Fusion Rules:

  • ML-based correlation across multiple signals
  • Detects multi-stage attacks automatically
  • Cannot be customized, only enabled/disabled

4. ML Behavior Analytics:

  • Anomaly detection based on learned baselines
  • UEBA rules for user/entity behavior
  • Requires time to establish baseline

Answer: KQL (Kusto Query Language) is the query language used throughout Sentinel and Azure Monitor. It's essential because:

Where KQL is used:

  • All analytics rules and detection logic
  • Threat hunting queries
  • Workbook visualizations
  • Investigation and pivoting
  • Custom functions and parsers

Key KQL characteristics:

  • Pipe-based syntax (similar to PowerShell/Unix)
  • Optimized for time-series log data
  • Rich operators: join, union, summarize, extend
  • Built-in ML functions for anomaly detection

Example:

SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType != 0  // Failed logins
| summarize FailureCount = count() by UserPrincipalName, IPAddress
| where FailureCount > 10
| sort by FailureCount desc

Answer: Sentinel and Defender XDR have deep bidirectional integration:

Data Flow:

  • Defender XDR alerts and incidents sync to Sentinel automatically
  • Raw telemetry (DeviceEvents, EmailEvents, etc.) can be streamed
  • Alert ingestion is free; raw logs are charged

Unified Investigation:

  • Incidents from both platforms appear in Sentinel
  • Deep link to Defender portal for endpoint investigation
  • Entity pages show Defender insights

Unified XDR Experience:

  • Microsoft now offers unified portal (security.microsoft.com)
  • Single pane for Defender + Sentinel incidents
  • Cross-product hunting with Advanced Hunting

Response Actions:

  • Sentinel playbooks can trigger Defender actions
  • Isolate device, disable user, block IP from Sentinel

KQL Mastery

5

Automation

Create Logic Apps playbooks for enrichment, containment, and response. Integrate with external systems.
