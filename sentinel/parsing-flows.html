Parsing & Normalization

Raw logs need parsing and normalization before they're useful for detection. Learn ASIM parsers, custom parsing, and data transformation techniques.

Why Normalization?

The Problem

Every vendor names fields differently. Windows calls it "TargetUserName", Azure AD uses "UserPrincipalName", Okta uses "actor.displayName". ASIM normalizes these to a common schema so ONE query works across ALL sources.

ASIM (Advanced Security Information Model)

ASIM Parser Example
// Without ASIM - must query each source separately
SigninLogs | project User = UserPrincipalName, ...
| union (SecurityEvent | project User = TargetUserName, ...)
| union (Okta_CL | project User = actor_displayName_s, ...)

// With ASIM - single normalized query
imAuthentication
| where TimeGenerated > ago(1h)
| where EventResult == "Failure"
| summarize count() by TargetUsername, SrcIpAddr

Interview Questions

ASIM (Advanced Security Information Model) is Microsoft's normalization framework:

  • Defines standard schemas for common security events
  • Provides parser functions that normalize vendor data
  • Enables single query across multiple sources
  • Out-of-box detections work across all normalized sources

Example: Instead of writing separate queries for Windows, Azure AD, and Okta authentication failures, use imAuthentication | where EventResult == "Failure"
