Parsing & Data Transformation

Guide to parsing and transforming data in Microsoft Sentinel using KQL in DCR transformations and query-time parsing.

Transformation Points

WhenMethodUse Case
Ingestion-time DCR transformKql Parse once, store structured. Reduces query cost.
Query-time KQL parse operators Ad-hoc parsing, schema changes without re-ingestion
Function Saved function Reusable parsing logic across queries

DCR Ingestion-Time Transformations

Use transformKql in Data Collection Rules to parse data before storage.

Example: Parse Key-Value Syslog

# Input log: # Feb 15 14:30:00 firewall: USER=jsmith ACTION=login SRC=10.0.1.50 RESULT=success # DCR transformKql: source | parse RawData with * "USER=" User:string " ACTION=" Action:string " SRC=" SourceIP:string " RESULT=" Result:string | extend TimeGenerated = now() | project TimeGenerated, User, Action, SourceIP, Result

Example: Parse JSON

# Input: {"timestamp":"2024-02-15T14:30:00Z","user":"admin","event":"login"} # DCR transformKql: source | extend parsed = parse_json(RawData) | extend TimeGenerated = todatetime(parsed.timestamp), User = tostring(parsed.user), EventType = tostring(parsed.event) | project TimeGenerated, User, EventType

Example: Parse CSV

# Input: 2024-02-15,admin,login,10.0.1.50,success # DCR transformKql: source | parse RawData with Timestamp:datetime "," User:string "," Action:string "," SourceIP:string "," Result:string | extend TimeGenerated = Timestamp | project TimeGenerated, User, Action, SourceIP, Result

Example: Filter and Transform

# Drop debug logs and enrich source | where RawData !contains "DEBUG" | extend TimeGenerated = now(), Severity = case( RawData contains "ERROR", "Error", RawData contains "WARN", "Warning", "Info" ), Environment = "Production" | project TimeGenerated, Severity, Environment, RawData

Query-Time KQL Parsing

parse operator

Extract fields using pattern matching

# Basic parse Syslog | parse SyslogMessage with * "user=" User:string " " * # Multiple fields Syslog | parse SyslogMessage with * "src=" SrcIP:string " dst=" DstIP:string " port=" Port:int # With regex flag Syslog | parse kind=regex SyslogMessage with @"user=(?P\w+)"

extract function

Regex-based extraction

# Extract IP address Syslog | extend SrcIP = extract(@"src=(\d+\.\d+\.\d+\.\d+)", 1, SyslogMessage) # Extract email domain SigninLogs | extend Domain = extract(@"@(.+)$", 1, UserPrincipalName)

split and array functions

Split strings into arrays

# Split by delimiter | extend Parts = split(Message, "|") | extend Field1 = Parts[0], Field2 = Parts[1] # Parse CSV-like data | extend Fields = split(RawData, ",") | extend User = Fields[0], Action = Fields[1], IP = Fields[2]

parse_json and parse_xml

Parse structured formats

# JSON parsing | extend Data = parse_json(RawData) | extend User = Data.user, Action = Data.action # Nested JSON | extend Data = parse_json(RawData) | extend User = Data.identity.user, IP = Data.network.sourceIp # XML parsing | extend Data = parse_xml(RawData) | extend EventID = Data.Event.System.EventID

ASIM Parsers (Schema Normalization)

ASIM (Advanced Security Information Model) normalizes data from different sources into common schemas.

Using Built-in ASIM Parsers

# Use ASIM parser instead of raw table _Im_Authentication | where TimeGenerated > ago(1h) | where EventResult == "Failure" | summarize FailureCount = count() by TargetUsername, SrcIpAddr # Available ASIM parsers: # _Im_Authentication - Login events # _Im_NetworkSession - Network connections # _Im_Dns - DNS queries # _Im_WebSession - Web/proxy logs # _Im_ProcessEvent - Process execution # _Im_FileEvent - File activity # _Im_RegistryEvent - Registry changes

ASIM Parser Benefits

  • Write one query, work across all data sources
  • Consistent field names (SrcIpAddr, DstIpAddr, TargetUsername)
  • Built-in parsers for common products
  • Required for some Content Hub analytics rules

Saved Functions

Create reusable parsing logic as saved functions.

# Create function in Log Analytics # Saved Functions → Create new # Function name: ParseFirewallLogs # Function alias: ParseFirewallLogs Syslog | where ProcessName == "firewall" | parse SyslogMessage with * "src=" SrcIP:string " dst=" DstIP:string " port=" DstPort:int " action=" Action:string | project TimeGenerated, SrcIP, DstIP, DstPort, Action # Usage: ParseFirewallLogs | where Action == "deny" | summarize count() by SrcIP

Best Practices

  • Parse at ingestion when possible: Reduces query-time cost, enables indexing
  • Use ASIM for cross-source queries: Normalize before correlating different data types
  • Test transformations: Use DCR debug mode before deploying
  • Handle nulls: Use coalesce() and null checks
  • Type casting: Always cast extracted values (tostring(), toint())

Interview Questions

Two approaches:

  1. Ingestion-time (DCR): Use transformKql in Data Collection Rule to parse before storage. More efficient for queries.
  2. Query-time: Use KQL parse operators (parse, extract, split, parse_json) in queries. More flexible but slower.

Common operators: parse for pattern matching, extract for regex, parse_json for JSON, split for delimiters.

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.

Comprehensive Troubleshooting

Systematic Troubleshooting Methodology

  1. Define the Problem: Clearly articulate what is not working as expected. Gather specific error messages, timestamps, and affected components.
  2. Gather Information: Collect relevant logs, metrics, and configuration details. Document recent changes that might be related.
  3. Analyze Data: Review collected information for patterns and anomalies. Compare against baseline behavior and known-good configurations.
  4. Develop Hypothesis: Form theories about potential root causes. Prioritize based on likelihood and impact.
  5. Test Solutions: Implement fixes in a controlled manner. Test one change at a time to isolate the effective solution.
  6. Document Resolution: Record the root cause and solution. Update runbooks and knowledge base for future reference.

Multi-pronged approach:

  • Tune Detection Rules: Adjust thresholds, add exclusions for known-good behavior
  • Implement Alert Aggregation: Group related alerts into incidents
