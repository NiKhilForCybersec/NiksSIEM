SOC Operations Guide

Daily workflows, triage procedures, escalation paths, and operational metrics for running an effective SOC with Microsoft Sentinel

SOC Tier Model

Tier 1 - Alert Triage

Focus: Initial alert review and classification

  • Monitor incident queue
  • Initial true/false positive determination
  • Gather basic context
  • Assign severity and priority
  • Escalate to Tier 2 if confirmed

Target: 15 min per alert initial triage

Tools: Incident queue, Entity pages, Hunting queries

Tier 2 - Investigation

Focus: Deep investigation and response

  • Full incident investigation
  • Root cause analysis
  • Containment actions
  • Evidence collection
  • Coordinate with IT teams

Target: 4 hours for high severity

Tools: Investigation graph, Notebooks, MDE portal

Tier 3 - Hunt & Engineer

Focus: Proactive hunting and improvement

  • Threat hunting campaigns
  • Detection engineering
  • Playbook development
  • Threat intelligence analysis
  • SOC metrics and optimization

Target: Continuous improvement

Tools: Notebooks, MITRE coverage, TI platforms

Incident Triage Workflow

1
Initial Assessment (5 min)
Read alert title, description, affected entities. Is this actionable?
2
Context Gathering (10 min)
Check entity pages for user/device. Review recent activity. Check if known good. 
// Get user context
let targetUser = "jsmith@company.com";
SigninLogs
| where TimeGenerated > ago(7d)
| where UserPrincipalName == targetUser
| summarize 
    TotalSignins = count(),
    UniqueIPs = dcount(IPAddress),
    UniqueLocations = dcount(Location),
    FailedSignins = countif(ResultType != 0)
| extend RiskIndicators = case(
    UniqueLocations > 5, "Multiple Locations",
    FailedSignins > 10, "High Failures",
    "Normal"
)
3
Determine True/False Positive
Is this legitimate activity? Check with user/manager if needed. Document reasoning.
4
Take Action
  • False Positive: Close incident, update rule tuning if recurring
  • True Positive: Escalate, initiate containment, document findings
  • Needs More Info: Add tasks, reach out to stakeholders

Common Investigation Playbooks

Compromised Account Investigation

  1. Confirm compromise: Review sign-in logs for impossible travel, new device, unusual location
  2. Scope impact: What did the attacker access? Check audit logs, email rules, file access
  3. Contain: Reset password, revoke sessions, block sign-ins if needed
  4. Investigate persistence: Check for MFA changes, app consents, inbox rules
  5. Remediate: Remove malicious rules, re-enable account with MFA
  6. Document: Timeline, impact assessment, lessons learned
KQL Account Compromise Investigation Query 
let suspectUser = "jsmith@company.com";
let investigationWindow = 7d;
// Sign-in analysis
SigninLogs
| where TimeGenerated > ago(investigationWindow)
| where UserPrincipalName == suspectUser
| summarize 
    SigninCount = count(),
    UniqueIPs = make_set(IPAddress),
    UniqueLocations = make_set(Location),
    UniqueDevices = make_set(DeviceDetail.displayName),
    RiskySignins = countif(RiskLevelDuringSignIn != "none")
by bin(TimeGenerated, 1d)
| order by TimeGenerated desc

Malware Detection Investigation

  1. Confirm detection: Is the file truly malicious? Check hash in VirusTotal, sandbox
  2. Isolate if needed: Use MDE to isolate host if active threat
  3. Determine entry: How did malware arrive? Email, web, USB?
  4. Check lateral movement: Any connections to other hosts?
  5. Remediate: Remove malware, patch vulnerability if exploited
  6. Hunt: Search for same IOCs across environment
KQL Malware IOC Hunt 
let maliciousHash = "abc123...";
let maliciousDomain = "evil.com";
let maliciousIP = "1.2.3.4";
// Search across data sources
union 
(DeviceFileEvents | where SHA256 == maliciousHash | extend Source = "FileEvent"),
(DeviceNetworkEvents | where RemoteUrl contains maliciousDomain | extend Source = "Network"),
(DeviceNetworkEvents | where RemoteIP == maliciousIP | extend Source = "Network"),
(CommonSecurityLog | where DestinationIP == maliciousIP | extend Source = "Firewall")
| project TimeGenerated, Source, DeviceName, AccountName, FileName, RemoteUrl, RemoteIP

Phishing Investigation

  1. Analyze email: Check sender, links, attachments in Defender for Office
  2. Identify recipients: Who else received this email?
  3. Check for clicks: Did anyone click the link or open attachment?
  4. If clicked: Initiate compromised account playbook
  5. Block IOCs: Add URLs/domains to block lists
  6. Notify users: Security awareness if widespread campaign

// Mean time to respond (MTTR)
SecurityIncident
| where TimeGenerated > ago(7d)
| where Status == "Closed"
| extend ResponseTime = datetime_diff('minute', ClosedTime, CreatedTime)
| summarize MTTR_Minutes = avg(ResponseTime) by Severity

// Top triggered rules
SecurityIncident
| where TimeGenerated > ago(7d)
| summarize Count = count() by Title
| top 10 by Count
KQL Health Monitoring Queries 
// Data ingestion lag by table
union withsource=TableName *
| where TimeGenerated > ago(1h)
| summarize 
    LatestEvent = max(TimeGenerated),
    AvgLag = avg(datetime_diff('second', ingestion_time(), TimeGenerated))
by TableName
| where AvgLag > 300 // Flag tables with >5 min lag
| order by AvgLag desc

// Connector health
SentinelHealth
| where TimeGenerated > ago(24h)
| where SentinelResourceType == "Data connector"
| summarize 
    LastStatus = arg_max(TimeGenerated, Status),
    FailureCount = countif(Status != "Success")
by SentinelResourceName
| where FailureCount > 0 or LastStatus != "Success"

Interview Preparation

Walk me through how you would triage a high-severity alert in Sentinel.

My triage process follows a structured approach:

First 5 minutes - Initial Assessment:

I read the alert title and description to understand what triggered it. I check the entities involved - user, device, IP address. I look at the alert severity and confidence. The key question: is this actionable or informational?

Next 10 minutes - Context Gathering:

I use entity pages to get historical context. For a user, I check their sign-in patterns, group memberships, and recent activity. For a device, I look at security posture and recent events. I check if this user or device has had similar alerts before.

True/False Positive Decision:

Based on the context, I determine if this is legitimate behavior. For example, an impossible travel alert might be false positive if the user has a VPN or cloud proxy. If unsure, I reach out to the user's manager or IT contact.

Action:

If false positive: I close the incident with detailed notes on why, and consider tuning the rule if it's a recurring pattern.

If true positive: I escalate to Tier 2, initiate containment if needed (disable account, isolate device), and document findings in the incident.

If needs more info: I add investigation tasks, set a follow-up time, and move to next priority incident.

The goal is to spend no more than 15 minutes on initial triage - enough to make a decision but not so much that the queue backs up.

How would you measure SOC effectiveness and what metrics would you track?

I focus on metrics that drive improvement across efficiency, quality, and coverage:

Efficiency Metrics:

MTTA (Mean Time to Acknowledge) - how quickly analysts pick up incidents. Target under 15 minutes for critical. MTTR (Mean Time to Resolve) - end-to-end time from alert to closure. Target varies by severity - 4 hours for critical, 24 hours for high.

Quality Metrics:

False Positive Rate - percentage of incidents closed as benign. If over 10-15%, we need to tune detections. Reopened Incident Rate - if incidents keep being reopened, investigation quality is an issue. Escalation Rate - too high means Tier 1 isn't empowered enough, too low might mean things are being missed.

Coverage Metrics:

MITRE ATT&CK coverage - what techniques do we have detections for? I use Sentinel's MITRE page to track this. Data source coverage - are all critical systems sending logs? Detection-to-incident ratio - are alerts actually generating incidents or being filtered out?

Business Metrics:

Security incidents with business impact - true measure of effectiveness. Dwell time for confirmed breaches - how long was an attacker in the environment before detection?

I report these weekly to the team and monthly to leadership, with trends over time to show improvement. The key is not just measuring but acting on the data - if MTTR is high, we look at automation opportunities.

Describe how you would handle a suspected ransomware incident.

Ransomware requires immediate, coordinated response:

Immediate Actions (first 15 minutes):

First, I'd verify the alert is legitimate - check if files are actually being encrypted, look for ransom notes, check for known ransomware process names. Simultaneously, I'd escalate to incident response and management - ransomware is always critical.

If confirmed, containment is priority one: isolate the affected device using MDE's network isolation. This stops lateral movement while keeping the device on for forensics. If multiple devices are affected, I'd work with network team to segment the affected subnet.

Scope Assessment (next hour):

Run hunting queries to find all affected devices - look for the same file hashes, process names, or network connections. Check authentication logs for the initial compromised account. Determine the initial access vector - usually phishing or exposed RDP.

Containment Expansion:

Disable any compromised accounts. Block identified malicious IPs and domains on the firewall. If there's a C2 channel, work with network team to sinkhole it.

Recovery Planning:

Work with backup team to assess recovery options. Don't pay the ransom - coordinate with legal and management on that decision. Preserve evidence for potential law enforcement involvement.

Post-Incident:

Full root cause analysis. Implement controls to prevent recurrence - often this means better email filtering, MFA enforcement, and endpoint hardening. Update detection rules based on lessons learned.

The key is speed and coordination - I'd be on a bridge call with IT, management, and other stakeholders throughout the incident.
