Event Hub Integration

Stream high-volume data into Microsoft Sentinel using Azure Event Hubs. Ideal for Azure Diagnostics, custom applications, and third-party integrations.

When to Use Event Hub

Use CaseWhy Event Hub
Azure Diagnostics at scaleStream from many resources to single ingestion point
High-volume streamingHandles millions of events per second
Third-party with Event Hub outputSome security tools support Event Hub natively
Custom application logsSend via Event Hub SDK from any app
Multi-tenant scenariosCentral hub for logs from multiple subscriptions

Architecture

┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │ Data Sources │ │ Event Hub │ │ Sentinel │ │ │ │ Namespace │ │ │ │ Azure Resources │────▶│ │────▶│ Log Analytics │ │ Applications │ │ Event Hubs │ │ Workspace │ │ Third-party │ │ (partitions) │ │ │ └─────────────────┘ └─────────────────┘ └─────────────────┘

Components

  • Event Hub Namespace: Container for Event Hubs, defines throughput
  • Event Hub: Individual stream (like a topic in Kafka)
  • Consumer Group: View of event stream for readers
  • Partitions: Parallel streams for scalability

Setup Event Hub for Sentinel

Step 1: Create Event Hub Namespace

Portal → Event Hubs → Create

1
Configure namespace:
Name:eh-sentinel-logs
Pricing tier:Standard (for partitioning)
Throughput Units:Start with 1, auto-inflate if needed
Region:Same as Log Analytics workspace

Step 2: Create Event Hub

Event Hub Namespace → + Event Hub

2
Configure event hub:
Name:security-logs
Partition count:4 (adjust based on volume)
Message retention:1-7 days

Step 3: Create Consumer Group

Event Hub → Consumer groups → + Consumer group

3
Name: sentinel-consumer
Each reader needs its own consumer group. Sentinel needs a dedicated one.

Step 4: Create Shared Access Policy

Event Hub → Shared access policies → + Add

4
Policy name: sentinel-listen
Claims: ☑️ Listen
Copy the connection string after creation.

Step 5: Configure Sentinel Connector

Sentinel → Data connectors → Azure Event Hub

5
Configuration:
  • Event Hub namespace
  • Event Hub name
  • Consumer group: sentinel-consumer
  • Target table in Log Analytics

Streaming Azure Diagnostics

Configure Resource Diagnostic Settings

[Any Azure Resource] → Diagnostic settings → + Add diagnostic setting

# Azure CLI example az monitor diagnostic-settings create \ --name "send-to-eventhub" \ --resource "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.KeyVault/vaults/{vault}" \ --event-hub "security-logs" \ --event-hub-rule "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.EventHub/namespaces/{ns}/authorizationRules/RootManageSharedAccessKey" \ --logs '[{"category": "AuditEvent", "enabled": true}]'

Sending from Custom Applications

Python SDK Example

from azure.eventhub import EventHubProducerClient, EventData import json connection_str = "Endpoint=sb://eh-sentinel-logs.servicebus.windows.net/;SharedAccessKeyName=send;SharedAccessKey=xxx" eventhub_name = "security-logs" producer = EventHubProducerClient.from_connection_string( conn_str=connection_str, eventhub_name=eventhub_name ) with producer: event_data_batch = producer.create_batch() log_entry = { "TimeGenerated": "2024-02-15T10:30:00Z", "Application": "MySecurityApp", "EventType": "Authentication", "User": "admin@company.com", "Result": "Success", "SourceIP": "10.0.1.50" } event_data_batch.add(EventData(json.dumps(log_entry))) producer.send_batch(event_data_batch)

# Check if data is flowing // Replace CustomLogs_CL with your target table CustomLogs_CL | where TimeGenerated > ago(1h) | take 10 # Monitor Event Hub metrics AzureMetrics | where ResourceProvider == "MICROSOFT.EVENTHUB" | where MetricName == "IncomingMessages" | summarize sum(Total) by bin(TimeGenerated, 5m) | render timechart

Best Practices

  • Same region: Keep Event Hub in same region as Log Analytics workspace
  • Partition count: Start with 4, increase for higher throughput
  • Consumer groups: Create dedicated consumer group for Sentinel
  • Auto-inflate: Enable for handling traffic spikes
  • Retention: 1-2 days typically sufficient (Sentinel reads near real-time)
  • Monitoring: Set up alerts for Event Hub quota usage

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.

Comprehensive Troubleshooting

Systematic Troubleshooting Methodology

  1. Define the Problem: Clearly articulate what is not working as expected. Gather specific error messages, timestamps, and affected components.
  2. Gather Information: Collect relevant logs, metrics, and configuration details. Document recent changes that might be related.
  3. Analyze Data: Review collected information for patterns and anomalies. Compare against baseline behavior and known-good configurations.
  4. Develop Hypothesis: Form theories about potential root causes. Prioritize based on likelihood and impact.
  5. Test Solutions: Implement fixes in a controlled manner. Test one change at a time to isolate the effective solution.
  6. Document Resolution: Record the root cause and solution. Update runbooks and knowledge base for future reference.
