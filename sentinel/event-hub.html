Event Hub Integration

Stream high-volume data into Microsoft Sentinel using Azure Event Hubs. Ideal for Azure Diagnostics, custom applications, and third-party integrations.

When to Use Event Hub

Use CaseWhy Event Hub
Azure Diagnostics at scaleStream from many resources to single ingestion point
High-volume streamingHandles millions of events per second
Third-party with Event Hub outputSome security tools support Event Hub natively
Custom application logsSend via Event Hub SDK from any app
Multi-tenant scenariosCentral hub for logs from multiple subscriptions

Architecture

┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │ Data Sources │ │ Event Hub │ │ Sentinel │ │ │ │ Namespace │ │ │ │ Azure Resources │────▶│ │────▶│ Log Analytics │ │ Applications │ │ Event Hubs │ │ Workspace │ │ Third-party │ │ (partitions) │ │ │ └─────────────────┘ └─────────────────┘ └─────────────────┘

Components

  • Event Hub Namespace: Container for Event Hubs, defines throughput
  • Event Hub: Individual stream (like a topic in Kafka)
  • Consumer Group: View of event stream for readers
  • Partitions: Parallel streams for scalability

Setup Event Hub for Sentinel

Step 1: Create Event Hub Namespace

Portal → Event Hubs → Create

1
Configure namespace:
Name:eh-sentinel-logs
Pricing tier:Standard (for partitioning)
Throughput Units:Start with 1, auto-inflate if needed
Region:Same as Log Analytics workspace

Step 2: Create Event Hub

Event Hub Namespace → + Event Hub

2
Configure event hub:
Name:security-logs
Partition count:4 (adjust based on volume)
Message retention:1-7 days

Step 3: Create Consumer Group

Event Hub → Consumer groups → + Consumer group

3
Name: sentinel-consumer
Each reader needs its own consumer group. Sentinel needs a dedicated one.

Step 4: Create Shared Access Policy

Event Hub → Shared access policies → + Add

4
Policy name: sentinel-listen
Claims: ☑️ Listen
Copy the connection string after creation.

Step 5: Configure Sentinel Connector

Sentinel → Data connectors → Azure Event Hub

5
Configuration:
  • Event Hub namespace
  • Event Hub name
  • Consumer group: sentinel-consumer
  • Target table in Log Analytics

Streaming Azure Diagnostics

Configure Resource Diagnostic Settings

[Any Azure Resource] → Diagnostic settings → + Add diagnostic setting

# Azure CLI example az monitor diagnostic-settings create \ --name "send-to-eventhub" \ --resource "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.KeyVault/vaults/{vault}" \ --event-hub "security-logs" \ --event-hub-rule "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.EventHub/namespaces/{ns}/authorizationRules/RootManageSharedAccessKey" \ --logs '[{"category": "AuditEvent", "enabled": true}]'

Sending from Custom Applications

Python SDK Example

from azure.eventhub import EventHubProducerClient, EventData import json connection_str = "Endpoint=sb://eh-sentinel-logs.servicebus.windows.net/;SharedAccessKeyName=send;SharedAccessKey=xxx" eventhub_name = "security-logs" producer = EventHubProducerClient.from_connection_string( conn_str=connection_str, eventhub_name=eventhub_name ) with producer: event_data_batch = producer.create_batch() log_entry = { "TimeGenerated": "2024-02-15T10:30:00Z", "Application": "MySecurityApp", "EventType": "Authentication", "User": "admin@company.com", "Result": "Success", "SourceIP": "10.0.1.50" } event_data_batch.add(EventData(json.dumps(log_entry))) producer.send_batch(event_data_batch)

Best Practices

  • Same region: Keep Event Hub in same region as Log Analytics workspace
  • Partition count: Start with 4, increase for higher throughput
  • Consumer groups: Create dedicated consumer group for Sentinel
  • Auto-inflate: Enable for handling traffic spikes
  • Retention: 1-2 days typically sufficient (Sentinel reads near real-time)
  • Monitoring: Set up alerts for Event Hub quota usage

Troubleshooting

IssueSolution
No data in SentinelCheck Event Hub metrics for incoming messages
High latencyIncrease partition count, check consumer lag
Connection failuresVerify SAS policy has Listen permission
ThrottlingUpgrade throughput units or enable auto-inflate

Interview Questions

  • High-volume streaming: When you need to ingest millions of events
  • Multi-resource aggregation: Collect Azure Diagnostics from many resources
  • Third-party integration: When source supports Event Hub output
  • Custom applications: Stream from apps using Event Hub SDK
  • Multi-tenant: Central ingestion from multiple subscriptions

Alternative: For simpler scenarios, use direct data connectors or Log Ingestion API.

Additional Interview Questions

Systematic troubleshooting approach:

  1. Identify the symptom: What exactly is not working as expected?
  2. Check connectivity: Verify network paths and authentication
  3. Review logs: Check internal/audit logs for errors
  4. Validate configuration: Compare against documented standards
  5. Test incrementally: Isolate the problem component
  6. Document findings: Record what worked and what didn't

Key performance indicators:

  • Availability: Uptime percentage, service availability
  • Performance: Query response time, processing latency
  • Volume: Events per second, data ingestion rate
  • Quality: Parse success rate, enrichment coverage
  • Efficiency: Resource utilization, cost per GB

High availability strategies:

  • Redundancy: Multiple instances of critical components
  • Load balancing: Distribute traffic across instances
  • Failover: Automatic switchover on failure
  • Monitoring: Health checks and alerting
  • DR Planning: Documented recovery procedures
