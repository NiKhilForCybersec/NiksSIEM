BIOC Rule Structure

Rule Components

Every BIOC rule consists of the following elements:

# BIOC Rule JSON Structure
{
    "name": "Rule Name",
    "description": "Detailed description of what the rule detects",
    "severity": "high",  // critical, high, medium, low, informational
    "category": "malware",  // malware, exploitation, persistence, etc.
    "mitre_tactic": "TA0003",
    "mitre_technique": "T1059.001",
    "xql_query": "dataset = xdr_data | filter ...",
    "enabled": true,
    "alert_config": {
        "group_by": ["agent_hostname"],
        "time_window": "5m",
        "threshold": 1
    }
}

Severity Levels

Severity Description Response Time Example
Critical Confirmed active threat Immediate Ransomware execution, C2 communication
High Likely malicious activity 15 minutes Credential dumping, lateral movement
Medium Suspicious behavior 1 hour Unusual PowerShell, persistence mechanism
Low Anomalous activity 4 hours Policy violation, minor deviation
Informational For tracking/audit 24 hours Administrative action, compliance log

BIOC Rule Examples

Credential Access Detection

# BIOC: LSASS Memory Access Detection
# Detects processes accessing LSASS memory (credential dumping indicator)
dataset = xdr_data 
| filter event_type = ENUM.EVENT_TYPE_OPEN_REMOTE_PROCESS_HANDLE
| filter target_process_name contains "lsass"
| filter action_process_image_name not in (
    "MsMpEng.exe",
    "csrss.exe",
    "svchost.exe",
    "lsass.exe",
    "wininit.exe"
)
| filter action_process_requested_access contains "PROCESS_VM_READ"
| fields agent_hostname, 
         action_process_image_name,
         action_process_image_path,
         action_process_command_line,
         target_process_name,
         action_process_username,
         action_process_sha256
| limit 100
# Rule Configuration
{
    "name": "LSASS Memory Access - Possible Credential Dumping",
    "severity": "high",
    "mitre_tactic": "TA0006",
    "mitre_technique": "T1003.001"
}

Execution - Encoded PowerShell

# BIOC: Encoded PowerShell Command Detection
# Detects base64 encoded PowerShell commands often used by attackers

| filter event_type = ENUM.EVENT_TYPE_PROCESS

Exfiltration - Large Data Transfer

# BIOC: Large Outbound Data Transfer
# Detects unusually large data transfers to external IPs
dataset = xdr_data 
| filter event_type = ENUM.EVENT_TYPE_NETWORK
| filter action_external_direction = "OUTBOUND"
| filter action_remote_ip != "10.*" and 
         action_remote_ip != "172.16.*" and 
         action_remote_ip != "192.168.*"

         action_remote_ip,

         action_total_bytes_sent,

Common Exclusion Patterns

# Exclude known legitimate tools
| filter action_process_image_name not in (
    "MsMpEng.exe",      # Windows Defender
    "CylanceSvc.exe",   # Cylance
    "CrowdStrike*",     # CrowdStrike
    "SentinelOne*"      # SentinelOne
)
# Exclude trusted paths
| filter action_process_image_path not contains "\\Windows\\System32\\"
| filter action_process_image_path not contains "\\Program Files\\Microsoft\\"
# Exclude service accounts
| filter action_process_username not in (
    "NT AUTHORITY\\SYSTEM",
    "NT AUTHORITY\\LOCAL SERVICE",
    "NT AUTHORITY\\NETWORK SERVICE"
)
# Exclude by hash (verified safe)
| filter action_process_sha256 not in (
    "abc123...",  # Known good hash
    "def456..."   # Known good hash
)
# Exclude signed binaries (use with caution)
| filter action_process_signature_status != "Signed"

Q: How do you create and tune custom detection rules in Cortex XDR?

A: I follow a structured approach. First, I research the threat behavior I want to detect and identify which MITRE ATT&CK technique it maps to. Then I use the Investigation module to write and test XQL queries against historical data to ensure the logic captures the behavior. I analyze the results for false positives and add appropriate exclusions for legitimate tools and processes. Once validated, I create a BIOC rule with the appropriate severity level and configure alerting thresholds. After deployment, I monitor the alert volume and false positive rate, continuously tuning the rule based on analyst feedback. I also maintain documentation linking each rule to its MITRE technique for coverage analysis.

Q: What's the difference between IOCs and BIOCs in Cortex XDR?

A: IOCs (Indicators of Compromise) are static indicators like file hashes, IP addresses, or domain names that represent known threats. They're reactive, meaning you need to already know about a threat to detect it. BIOCs (Behavioral Indicators of Compromise) detect suspicious behaviors regardless of the specific malware or tool used. For example, instead of blocking a known Mimikatz hash (IOC), a BIOC rule detects any process accessing LSASS memory for reading, which would catch both Mimikatz and any new credential dumping tool. BIOCs are more resilient against attackers who modify their tools to evade hash-based detection.

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.

Comprehensive Troubleshooting

Systematic Troubleshooting Methodology

  1. Define the Problem: Clearly articulate what is not working as expected. Gather specific error messages, timestamps, and affected components.
  2. Gather Information: Collect relevant logs, metrics, and configuration details. Document recent changes that might be related.
  3. Analyze Data: Review collected information for patterns and anomalies. Compare against baseline behavior and known-good configurations.
  4. Develop Hypothesis: Form theories about potential root causes. Prioritize based on likelihood and impact.
  5. Test Solutions: Implement fixes in a controlled manner. Test one change at a time to isolate the effective solution.
  6. Document Resolution: Record the root cause and solution. Update runbooks and knowledge base for future reference.

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.
