Audit Log Queries

Common Audit Queries for Compliance

# User Login Activity Report # ═══════════════════════════════════════════════════════════════ dataset = audit_log | filter event_type = "USER_LOGIN" | filter _time > now() - 30d | comp count() as login_count, min(_time) as first_login, max(_time) as last_login by user_email, source_ip | sort last_login desc # ═══════════════════════════════════════════════════════════════ # Failed Login Attempts (Security Event) # ═══════════════════════════════════════════════════════════════ dataset = audit_log | filter event_type = "USER_LOGIN_FAILED" | filter _time > now() - 7d | comp count() as failed_attempts by user_email, source_ip | filter failed_attempts > 3 | sort failed_attempts desc # ═══════════════════════════════════════════════════════════════ # Configuration Changes (Change Management Evidence) # ═══════════════════════════════════════════════════════════════ dataset = audit_log | filter event_type in ("RULE_CREATED", "RULE_MODIFIED", "RULE_DELETED", "INTEGRATION_MODIFIED", "EXCLUSION_CREATED") | filter _time > now() - 30d | fields _time, user_email, event_type, target_name, change_details | sort _time desc # ═══════════════════════════════════════════════════════════════ # Alert Handling Activity (SLA Compliance) # ═══════════════════════════════════════════════════════════════ dataset = audit_log | filter event_type in ("ALERT_ASSIGNED", "ALERT_RESOLVED", "ALERT_ESCALATED") | filter _time > now() - 7d | comp count() as actions by user_email, event_type | sort actions desc # ═══════════════════════════════════════════════════════════════ # Data Export Events (Data Governance) # ═══════════════════════════════════════════════════════════════ dataset = audit_log | filter event_type in ("QUERY_EXPORT", "REPORT_DOWNLOAD", "INCIDENT_EXPORT") | filter _time > now() - 30d | fields _time, user_email, event_type, export_size, export_format | sort _time desc

Interview Questions

Preparing for an external audit requires understanding what the auditors need to see and having evidence readily available. I approach this systematically based on the audit framework.

First, I review the audit scope and identify which XSIAM capabilities map to which controls. For PCI-DSS, that's primarily Requirement 10 (logging) and 11 (monitoring). For SOC 2, it's the Security trust criteria. I create a control-to-evidence mapping document that lists each control and the specific XSIAM artifacts that demonstrate compliance.

For each control, I prepare the evidence in advance. This includes screenshots of configurations, exports of detection rules, sample query results showing the system is working, and documented procedures. Auditors want to see not just that controls exist, but that they're operating effectively - so I include metrics like alert volumes and response times.

I also prepare for walkthroughs. Auditors will want to see the system in action - log in, show data ingestion, demonstrate alert triage, run a query. I practice these demonstrations and have test scenarios ready that don't expose sensitive production data but show realistic workflows.

Finally, I maintain an audit-ready posture year-round rather than scrambling before audits. Scheduled reports run monthly to capture evidence continuously. Configuration changes are documented in change management. This way, when audit time comes, I'm compiling existing evidence rather than generating it from scratch.

Executive and technical reports serve fundamentally different purposes, so the content and presentation differ significantly.

For executives, I focus on business impact and risk posture. The report fits on one page with clear red/yellow/green status indicators. Key metrics include: number of security incidents (trending up or down?), mean time to detect and respond (are we getting faster?), comparison to industry benchmarks, and any critical incidents that required executive attention. I avoid technical jargon and focus on outcomes. Executives want to know "are we secure?" not "how does the correlation engine work?"

For technical audiences like SOC managers or security architects, reports are detailed and actionable. They include: alert volumes by source and severity, false positive rates by rule (which need tuning?), analyst productivity metrics, detection coverage against MITRE ATT&CK, and specific recommendations for improvement. These reports might be 10-20 pages with charts, tables, and drill-down data.

I also create role-specific reports. Compliance officers get audit-focused reports mapped to control frameworks. IT operations gets reports on integration health and data pipeline status. Development teams might get application-specific security reports if we're monitoring their systems.

The key principle is matching the report to what that audience needs to make decisions. Executives make strategic decisions, so they need strategic information. Analysts make tactical decisions, so they need operational data.
