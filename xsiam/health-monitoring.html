Health Monitoring

Comprehensive guide to monitoring XSIAM platform health, agent status, data ingestion rates, and proactive alerting to ensure your security operations run smoothly.

Health Monitoring Architecture

Why Platform Health Matters

A security platform that isn't healthy can't protect you. Silent failures in data ingestion mean detection gaps. Unhealthy agents mean blind spots on endpoints. Proactive health monitoring ensures you know about problems before attackers exploit the gaps.

Health Monitoring Queries

Data Ingestion Health

# Query 1: EPS by Data Source (Last 24 Hours) # ───────────────────────────────────────────── # Identifies which data sources are active and their volume dataset = xdr_data | filter _time > now() - 24h | comp count() as event_count by _vendor, _product | alter eps = divide(event_count, 86400) | sort eps desc | limit 50 # Expected: Consistent EPS across your known data sources # Alert if: Any source drops to 0 or >50% below baseline
# Query 2: Ingestion Latency Check # ───────────────────────────────────────────── # Measures time between event generation and availability dataset = xdr_data | filter _time > now() - 1h | alter ingestion_delay = timestamp_diff(_insert_time, _time, "MINUTE") | comp avg(ingestion_delay) as avg_delay, max(ingestion_delay) as max_delay, percentile(ingestion_delay, 95) as p95_delay by _vendor # Healthy: avg_delay < 5 minutes # Warning: avg_delay 5-15 minutes # Critical: avg_delay > 15 minutes
# Query 3: Parsing Failure Detection # ───────────────────────────────────────────── # Finds events that failed to parse correctly dataset = xdr_data | filter _time > now() - 24h | filter _raw_log != null and xdm.event.type = null | comp count() as parse_failures by _vendor, _product | sort parse_failures desc # Any results indicate parsing issues that need investigation # Compare against total volume to calculate failure rate

Agent Health Queries

# Query 4: Disconnected Agents # ───────────────────────────────────────────── # Finds agents that haven't checked in recently dataset = endpoints | filter last_seen < now() - 30m | alter hours_since_contact = timestamp_diff(now(), last_seen, "HOUR") | fields endpoint_name, endpoint_type, os_type, last_seen, hours_since_contact, endpoint_status | sort hours_since_contact desc # Priority by duration: # 30min - 4hr: Investigate (might be powered off) # 4hr - 24hr: Urgent (likely connectivity issue) # >24hr: Critical (possible uninstall or compromise)
# Query 5: Agent Version Distribution # ───────────────────────────────────────────── # Shows agent version breakdown for upgrade planning dataset = endpoints | comp count() as agent_count by agent_version, os_type | sort agent_version desc, agent_count desc # Goal: 90%+ on current or N-1 version # Flag any agents more than 2 versions behind
# Query 6: Agent Resource Usage # ───────────────────────────────────────────── # Identifies agents consuming excessive resources dataset = xdr_data | filter _time > now() - 1h | filter event_type = "agent_resource_usage" | filter cpu_percent > 50 or memory_mb > 500 | comp avg(cpu_percent) as avg_cpu, avg(memory_mb) as avg_memory, count() as sample_count by endpoint_name | filter avg_cpu > 30 or avg_memory > 300 | sort avg_cpu desc # Investigate agents with consistently high resource usage

Broker VM Health Queries

# Query 7: Broker VM Performance # ───────────────────────────────────────────── # Monitors Broker VM resource utilization dataset = broker_vm_logs | filter _time > now() - 4h | comp avg(cpu_percent) as avg_cpu, avg(memory_percent) as avg_mem, avg(disk_percent) as avg_disk, max(queue_depth) as max_queue by broker_name | alter health_status = if( avg_cpu > 85 or avg_mem > 90 or max_queue > 5000, "CRITICAL", avg_cpu > 70 or avg_mem > 75 or max_queue > 1000, "WARNING", "HEALTHY" ) # Action thresholds: # WARNING: Review capacity, plan scaling # CRITICAL: Immediate action required
# Query 8: Data Source Flow by Broker # ───────────────────────────────────────────── # Verifies data is flowing through each Broker dataset = xdr_data | filter _time > now() - 1h | filter collection_method = "broker_vm" | comp count() as events by broker_name, _vendor | sort broker_name, events desc # Compare against expected data sources per Broker # Zero events from expected source = collection failure

Automated Health Alerting

Critical Health Alert Rules

# Alert Rule 1: Data Source Gone Silent # ───────────────────────────────────────────── # Triggers when a data source stops sending data Name: DATA_SOURCE_SILENT Severity: HIGH Frequency: Every 15 minutes XQL Query: dataset = xdr_data | filter _time > now() - 1h | comp count() as current_hour by _vendor, _product | join type=left ( dataset = xdr_data | filter _time between (now() - 25h) and (now() - 24h) | comp count() as baseline by _vendor, _product ) as baseline on _vendor, _product | filter current_hour < multiply(baseline.baseline, 0.1) or current_hour = 0 | filter baseline.baseline > 100 // Only alert on significant sources Action: - PagerDuty alert to on-call - Create incident ticket - Notify SOC Slack channel
# Alert Rule 2: Mass Agent Disconnection # ───────────────────────────────────────────── # Triggers when significant number of agents disconnect Name: MASS_AGENT_DISCONNECT Severity: CRITICAL Frequency: Every 5 minutes XQL Query: dataset = endpoints | comp count() as total_agents, countif(endpoint_status = "disconnected") as disconnected | alter disconnect_pct = multiply(divide(disconnected, total_agents), 100) | filter disconnect_pct > 10 // More than 10% disconnected Action: - Immediate PagerDuty (wake people up) - Auto-escalate to platform team - Begin incident response procedure
# Alert Rule 3: Broker VM Resource Critical # ───────────────────────────────────────────── # Triggers when Broker approaches resource exhaustion Name: BROKER_RESOURCE_CRITICAL Severity: HIGH Frequency: Every 5 minutes XQL Query: dataset = broker_vm_logs | filter _time > now() - 15m | comp avg(cpu_percent) as cpu, avg(memory_percent) as memory, avg(disk_percent) as disk, max(queue_depth) as queue by broker_name | filter cpu > 85 or memory > 90 or disk > 85 or queue > 5000 Action: - Alert platform team - Trigger capacity review - If disk > 95%: Emergency cleanup procedure
# Alert Rule 4: Ingestion Latency Spike # ───────────────────────────────────────────── # Triggers when data takes too long to become queryable Name: INGESTION_LATENCY_HIGH Severity: MEDIUM Frequency: Every 10 minutes XQL Query: dataset = xdr_data | filter _time > now() - 30m | alter latency_min = timestamp_diff(_insert_time, _time, "MINUTE") | comp avg(latency_min) as avg_latency, percentile(latency_min, 95) as p95_latency | filter avg_latency > 15 or p95_latency > 30 Action: - Notify platform team - Check for processing backlog - Review recent configuration changes

Interview Questions

Platform health monitoring needs to be proactive, not reactive. You want to know about issues before they impact detection capability. My approach layers multiple monitoring methods to catch problems at different levels.

At the infrastructure level, I monitor Broker VMs using both XSIAM internal logs and traditional infrastructure monitoring. The Broker sends health telemetry to XSIAM, but I also have external monitoring - Datadog or Prometheus - watching CPU, memory, disk, and network from outside. This catches scenarios where the Broker is so overwhelmed it can't even report its own health. Key thresholds: CPU sustained above 70% triggers capacity planning, above 85% triggers immediate action.

For data ingestion health, I build baseline profiles for each data source. Know your normal EPS by source, by hour of day, by day of week. Retail environments see different patterns than manufacturing. I create alerts when any source deviates more than 30% from baseline or goes to zero. The zero-event alert is critical - a silent failure where a firewall stops logging is worse than a noisy failure because nobody notices until there's an incident in that blind spot.

Agent health monitoring tracks both coverage and currency. I report on total agent count versus expected deployment (do we have agents on all endpoints in the CMDB?), disconnected agent percentage, and version distribution. The disconnected query runs every 5 minutes with escalating alerts: >5% disconnected pages the on-call, >20% triggers incident response because that level of dropout suggests network or platform issue.

Finally, detection health. Alerts should be generating at predictable rates. If your brute force detection typically fires 50 times a day and suddenly drops to zero, either attackers stopped or your detection broke. I trend alert volumes by rule and investigate significant deviations.

Fifteen percent is significant - that's well above normal churn from laptops being offline. My troubleshooting starts with understanding the scope and pattern before diving into individual agents.

First, I'd characterize the disconnected population. Are they random across the environment or concentrated? Query the disconnected agents grouped by: operating system, location/subnet, department, last seen time, and agent version. Patterns tell you where to look. If all disconnected agents are Windows 10 version 21H2, maybe a recent Windows update conflicted with the agent. If they're all in the Atlanta office, it's probably network. If they're spread evenly, it might be a platform or agent update issue.

Next, I check the timeline. When did disconnections start? Does it correlate with a change window? Check XSIAM's own audit logs for recent platform changes, and check your change management system for network or endpoint changes. A firewall rule change or proxy configuration update often causes mass disconnection.

For diagnosis, I'd pick a sample of disconnected endpoints - maybe 5 from different characteristics. If I can reach them (RDP, SCCM, endpoint management tool), check the Cortex agent service status. Is it running? Check the agent logs - usually in C:\ProgramData\Cortex XDR\ on Windows. Common issues: certificate problems (clock drift causes cert validation failure), network connectivity (agent can't reach cloud endpoints), resource exhaustion (agent crashed due to memory), or policy conflict with other security tools.

If agents are running but not connecting, test network connectivity from the endpoint. Can it resolve the Cortex cloud endpoints? Can it establish TLS connections on 443? Are there proxy issues? Packet capture might be needed if basic connectivity looks okay but connections are failing.

Throughout this process, I'm documenting findings and communicating status. At 15% disconnection, stakeholders need updates. I'd have a bridge call running if this extends beyond 30 minutes, with representation from network team, endpoint team, and potentially Palo Alto support.

Silent failures are the most dangerous kind because nobody knows there's a problem. A firewall that stops logging doesn't generate an error - it just... stops. Your SIEM has a blind spot, and you won't know until an incident happens in that coverage gap.

The solution is positive confirmation monitoring - instead of waiting for errors, actively verify that expected data is arriving. I implement this at multiple levels. First, inventory your data sources. Every integration should be documented: source name, expected log types, approximate EPS range, who owns it, and criticality level. You can't monitor what you don't know about.

Second, build baseline profiles. For each source, establish what "normal" looks like. Your primary firewall might send 50,000 EPS during business hours, 10,000 overnight. Your Okta integration might be 500 events during business hours, near zero overnight. Store these baselines and compare current rates against them.

The monitoring query runs on a schedule - every 15 minutes for critical sources, hourly for others. It counts events received per source in the last period and compares against baseline. Alert conditions: source at zero events (critical), source below 50% of baseline (warning), source above 200% of baseline (investigate - might indicate attack or misconfiguration).

For critical sources, I add heartbeat monitoring. Configure the source to send a synthetic "heartbeat" event every 5 minutes. If the heartbeat stops, the source has a problem. This catches scenarios where the logging system is healthy but network path is broken - the heartbeat uses the same path as real logs.

I also monitor for implicit signals. If a firewall integration shows zero denied connections for an hour, that's suspicious - there's always some noise. If an authentication source shows no failed logins for 4 hours during business hours, something's probably wrong. Absence of expected events is itself an event worth alerting on.

Executive and operations dashboards serve fundamentally different purposes, so the metrics need to match the audience's needs and decision-making context.

For executives, the question is "Is our security platform working and are we getting value?" They don't need EPS by data source - they need confidence indicators. I'd include: Overall platform availability (99.9% uptime last 30 days), detection coverage percentage (endpoints with agents / total endpoints), mean time to detect and respond trends, and cost metrics like cost per protected endpoint or cost per investigated incident. Visual indicators should be simple - green/yellow/red status, trend arrows showing improvement or degradation. If an executive glances at this dashboard, they should know in 5 seconds whether things are okay.

The operations dashboard is for people who need to act on problems. Here I want detailed, actionable metrics. Data ingestion section: EPS by source with sparklines showing 24-hour trend, sources below baseline highlighted in yellow/red, ingestion latency gauges. Agent health section: total/connected/disconnected counts with drill-down capability, version distribution pie chart, agents not seen in 24+ hours list. Infrastructure section: Broker VM resource utilization, queue depths, processing lag. Integration section: status of each API integration, last successful sync, error counts.

The key difference is time horizon and granularity. Executives care about trends over weeks or months - is our security posture improving? Operations cares about the last hours and minutes - is something broken right now? The operations dashboard should refresh every minute; the executive dashboard might be daily or weekly snapshots.

I'd also separate alerting. Operations dashboards connect to PagerDuty and SOC alerting - a critical issue generates immediate notification. Executive dashboards generate weekly summary emails - "Platform health report: 3 incidents this week, all resolved within SLA."

Global SOC health monitoring needs to account for follow-the-sun operations while maintaining consistent standards. The challenge is ensuring nothing falls through the cracks during shift handoffs and that regional teams have visibility into their relevant infrastructure.

First, establish a single source of truth for platform health. One centralized health dashboard that all regions use, with filtering capabilities for regional views. Don't let each region build their own monitoring - you'll get inconsistent coverage and duplicated effort. The platform team owns the health monitoring infrastructure; regional SOCs consume it.

Shift handoff is critical for health issues. I implement structured handoff procedures: outgoing shift documents any active health issues, investigation status, and expected resolution. The handoff meeting includes health status as a standing agenda item. For overnight or weekend coverage, ensure on-call escalation paths work across time zones - if the Americas on-call can't reach platform support, they need to be able to escalate to EMEA or APAC teams.

Alerting needs to be time-zone aware. A Broker VM in Singapore shouldn't wake up the Americas on-call at 3 AM if APAC is staffed. Route alerts based on the impacted infrastructure's location and the current staffed regions. Critical alerts (mass agent disconnection, platform outage) go to everyone; regional issues go to the relevant team.

I also build regional health views into the dashboard. EMEA SOC leads see EMEA Broker status, EMEA agent health, EMEA-specific data sources prominently. They can see global status too, but their region's health is front and center. This ownership model means someone is always responsible for each component.

Documentation and runbooks must be accessible globally. Store in a wiki or knowledge base that's available regardless of location. When APAC encounters an issue at 2 AM their time, they need to be able to follow the runbook without waiting for Americas to wake up. Include escalation contacts for each region and for the vendor.
