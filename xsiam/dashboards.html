XSIAM Dashboards - Complete Configuration Guide

Build effective SOC dashboards with XQL-powered widgets. This guide covers dashboard creation, widget types, XQL queries for each visualization, and production-ready SOC dashboard examples.

Dashboard Concepts

How XSIAM Dashboards Work

XSIAM dashboards are built from widgets. Each widget runs an XQL query and displays results in a visualization (chart, table, number, etc.). Dashboards refresh on schedule or manually.

Creating a Dashboard

XSIAM Console → Dashboards → + New Dashboard
1
Create New Dashboard

Dashboards (left nav) → + New Dashboard → Enter name (e.g., "SOC Operations")

2
Set Dashboard Properties
Name:SOC Operations Dashboard
Refresh Interval:5 minutes (or manual)
Default Time Range:Last 24 hours
Shared:Yes (visible to team)
3
Add Widgets

Click "+ Add Widget" → Select visualization type → Enter XQL query

Widget Types & XQL Queries

1. Number Widget (Single Stat)

Open Critical Incidents

Shows a single number prominently - perfect for KPIs

// XQL Query for Number Widget dataset = xdr_incident_raw | filter status != "RESOLVED" and status != "CLOSED" | filter severity = "HIGH" or severity = "CRITICAL" | comp count = count()
23
Open Critical Incidents

2. Pie/Donut Chart

Incidents by Severity

Shows distribution across categories

// XQL Query for Pie Chart dataset = xdr_incident_raw | filter _time > ago(24h) | comp count = count() by severity

Widget Config: Group By = severity, Value = count

3. Bar Chart

Top 10 Alert Sources

Shows ranked comparison

// XQL Query for Bar Chart dataset = xdr_data | filter _time > ago(24h) | filter event_type = ALERT | comp alert_count = count() by agent_hostname | sort desc alert_count | limit 10

4. Line Chart (Time Series)

Alert Volume Over Time

Shows trends over time - essential for spotting anomalies

// XQL Query for Time Series dataset = xdr_data | filter _time > ago(7d) | filter event_type = ALERT | comp alert_count = count() by bin(_time, 1h) | sort asc _time

Key: Use bin(_time, interval) to bucket time. Options: 1h, 4h, 1d, etc.

5. Table Widget

Recent High-Severity Alerts

Shows detailed records

// XQL Query for Table dataset = xdr_incident_raw | filter _time > ago(24h) | filter severity in ("HIGH", "CRITICAL") | fields _time, incident_id, name, severity, status, assigned_to | sort desc _time | limit 20

Production SOC Dashboard Example

Complete SOC Operations Dashboard

Copy these XQL queries to build a comprehensive SOC dashboard:

Widget 1: Open Incidents (Number)

dataset = xdr_incident_raw | filter status in ("NEW", "UNDER_INVESTIGATION") | comp count = count()

Widget 2: MTTR - Mean Time to Respond (Number)

// Calculate MTTR for incidents closed in last 7 days dataset = xdr_incident_raw | filter status = "RESOLVED" and resolved_time > ago(7d) | alter response_time_hours = timestamp_diff(resolved_time, creation_time, "HOUR") | comp avg_mttr = avg(response_time_hours) | alter avg_mttr = round(avg_mttr, 1)

Widget 3: Incidents by Severity (Pie Chart)

dataset = xdr_incident_raw | filter _time > ago(24h) | comp count = count() by severity

Widget 4: Top Targeted Hosts (Bar Chart)

dataset = xdr_data | filter _time > ago(24h) | filter event_type = ALERT | comp alerts = count() by agent_hostname | sort desc alerts | limit 10

Widget 5: Alert Trend (Line Chart - 7 Days)

dataset = xdr_data | filter _time > ago(7d) | filter event_type = ALERT | comp count = count() by bin(_time, 4h), severity | sort asc _time

Widget 6: Top MITRE Techniques Detected (Bar Chart)

dataset = xdr_incident_raw | filter _time > ago(30d) | filter mitre_technique_id != null | comp count = count() by mitre_technique_id, mitre_technique_name | sort desc count | limit 10

Widget 7: Firewall Blocked Connections (Line Chart)

dataset = panw_ngfw_traffic_raw | filter _time > ago(24h) | filter action = "deny" | comp blocked = count() by bin(_time, 1h) | sort asc _time

Widget 8: Failed Logins by User (Table)

dataset = xdr_data | filter _time > ago(24h) | filter event_type = AUTH and action_outcome = FAILURE | comp failed_logins = count() by xdm.target.user.username, xdm.source.ipv4 | filter failed_logins >= 5 | sort desc failed_logins | limit 20

Executive Dashboard Widgets

Security Posture Score (Number)

// Calculate score based on open incidents and coverage dataset = xdr_incident_raw | filter status in ("NEW", "UNDER_INVESTIGATION") | comp critical = countif(severity = "CRITICAL"), high = countif(severity = "HIGH"), medium = countif(severity = "MEDIUM") | alter score = 100 - (critical * 10) - (high * 5) - (medium * 2) | alter score = if(score < 0, 0, score) | fields score

Month-over-Month Incident Trend

dataset = xdr_incident_raw | filter _time > ago(60d) | alter month = format_timestamp("%Y-%m", _time) | comp incidents = count() by month | sort asc month

Endpoint Coverage (Number)

dataset = xdr_agents | filter agent_status = "CONNECTED" | comp protected_endpoints = count()

Interview Questions

When designing a SOC operations dashboard, I start by identifying the key questions analysts need answered at a glance. What requires immediate attention? What's my current workload? Are there any anomalies I should know about? The dashboard should answer these in seconds without requiring users to dig through menus or run queries.

I structure the dashboard in priority zones. Top left is the "attention" zone - open critical incidents as a big red number, any active incidents being worked. This is where eyes land first. Top right shows queue health - total open alerts, backlog trend, and unassigned items. Middle section displays trends - alert volume over time helps spot spikes that indicate either an attack or a noisy rule needing attention.

For the queries, I keep them simple and fast. Dashboard widgets need to return quickly, so I avoid complex joins or large time ranges. For the open incidents count: dataset = xdr_incident_raw | filter status in ("NEW", "UNDER_INVESTIGATION") | comp count(). For the trend line, I use hourly buckets for the last 24 hours, which balances granularity with performance.

I also include operational widgets like top triggered rules (helps identify noisy detections) and incidents by analyst (shows workload distribution). The dashboard refreshes every 5 minutes during business hours - frequent enough to catch changes, not so frequent that it impacts performance. I share it with the team and put it on a wall display so everyone can see the current state without logging in.

Executive dashboards require a completely different approach than operational ones. Executives don't care about individual alerts - they care about risk posture, trends, and whether their investment in security is paying off. I focus on answering: "Are we more or less secure than last month? What are the biggest risks? Are we meeting our SLAs?"

I start with a security posture score - a single number that aggregates multiple factors. It might be calculated from open critical vulnerabilities, unresolved high-severity incidents, endpoint coverage percentage, and mean time to respond. This gives executives a quick pulse check: green is good, yellow means attention needed, red requires a conversation.

Trend widgets are more important than point-in-time metrics at this level. I show month-over-month incident trends, MTTR improvement over time, and coverage metrics (what percentage of endpoints have agents, what percentage of cloud accounts are monitored). These tell the story of security program maturity, which is what executives actually want to understand.

I avoid technical jargon. Instead of "BIOC alerts," I show "Malware detections." Instead of "correlation rules," I show "Custom threat detections." I use the color coding consistently - red for areas needing investment or attention, green for areas performing well. And I always include a single table widget at the bottom showing the top 5 incidents from the month with plain-English descriptions, because executives often want to understand specific examples they can reference in board meetings.

Time-series visualizations are the backbone of security monitoring because they reveal patterns that point-in-time numbers miss. A spike of 500 alerts might look alarming, but if the baseline is 600, it's actually quiet. Context is everything, and time-series provides that context.

The key XQL pattern is using bin(_time, interval) to bucket events. For operational dashboards showing the last 24 hours, I use 1-hour buckets: dataset = xdr_data | filter _time > ago(24h) | comp count() by bin(_time, 1h) | sort asc _time. For weekly or monthly views, I use daily buckets. The bucket size should give you enough data points to see trends without being so granular that the line becomes noisy.

Multiple series on one chart tells a richer story. Instead of just total alerts, I'll add by severity to show critical vs. high vs. medium over time. This reveals whether severity distribution is changing. I also use multiple y-axes when comparing different scales - like alert volume (hundreds) versus incidents created (tens) on the same timeline.

For anomaly detection, I sometimes add a baseline calculation. In XSIAM, I can compute a rolling average and show it alongside actual values, making deviations obvious. The formula involves comparing today's count against the average of the same day/hour over the past 4 weeks. When the actual line diverges significantly from the baseline, that's worth investigating even if the absolute number doesn't seem alarming.

There's a core set of metrics every SOC should track that align with NIST incident response phases and operational efficiency. For detection, I track mean time to detect (MTTD) - the gap between when something malicious happens and when we alert on it. This requires correlation between incident timeline and alert creation timestamps.

For response, MTTR (mean time to respond/resolve) is the gold standard. I calculate this as the difference between incident creation and resolution: alter response_hours = timestamp_diff(resolved_time, creation_time, "HOUR") | comp avg(response_hours). I break this down by severity since critical incidents should resolve faster than informational ones. I also track time-to-triage separately from time-to-resolution because different teams own different phases.

For efficiency, I track false positive rate per detection rule. This drives tuning prioritization - rules with 80% FP rates need attention. I visualize this as a bar chart sorted by FP rate, showing count and percentage. Alert volume per analyst helps identify when the team is overloaded and needs additional headcount or automation.

Coverage metrics are often overlooked but critical. What percentage of endpoints have agents? What percentage of cloud workloads are monitored? What percentage of network traffic flows through monitored paths? I show these as gauge widgets - green above 95%, yellow 85-95%, red below 85%. Gaps in coverage are security risks that technical metrics won't reveal because you can't alert on what you don't see.

Each platform approaches dashboarding differently based on their architecture and target users. Splunk has the most mature and flexible dashboarding - you can build practically anything with SPL and the Simple XML or Dashboard Studio frameworks. The trade-off is complexity; building a polished Splunk dashboard requires learning their dashboard markup language or using the studio which has a learning curve.

XSIAM dashboards are more opinionated but easier to get started. The widget builder is point-and-click, and XQL queries are straightforward. Built-in visualizations cover the standard use cases well - numbers, pie charts, bar charts, line charts, tables. The limitation is customization; you can't do the same level of drilldown interactivity or custom visualizations that Splunk enables. For 90% of SOC use cases, XSIAM dashboards are sufficient and faster to build.

Sentinel uses Azure Workbooks, which integrate with the broader Azure ecosystem. If you're already using Azure Monitor and want unified dashboards across infrastructure and security, Workbooks are powerful. KQL is expressive and handles time-series well. The visualization options are solid and include Azure-specific widgets like maps and heatmaps. The main challenge is that Workbooks can feel clunky for pure security operations - they're designed for broader observability use cases.

For quick operational dashboards, I prefer XSIAM for its simplicity. For complex executive reporting with drill-downs and custom visualizations, Splunk wins. For organizations heavily invested in Azure wanting unified monitoring, Sentinel Workbooks make sense. In practice, I often build the operational dashboard in the native platform and use a separate tool like Power BI or Grafana for executive reporting that pulls from multiple sources.
