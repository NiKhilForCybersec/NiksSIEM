Threat Intelligence - Complete Configuration Guide

How to configure, import, and operationalize threat intelligence in XSIAM. Covers IOC management, feed integration, and using TI in detection rules.

What is Threat Intelligence in XSIAM?

Definition

Threat Intelligence (TI) in XSIAM consists of Indicators of Compromise (IOCs) that are matched against your logs to detect known threats. When network traffic, file hashes, or URLs match a known-bad indicator, an alert is generated.

IOC Types Supported

SHA256 Hash SHA1 Hash MD5 Hash IPv4 Address IPv6 Address Domain URL Email

Using Threat Intel in XQL Queries

Query 1: Find All IOC Matches in Last 24 Hours

dataset = threat_intel_matches | filter _time > ago(24h) | fields _time, indicator_value, indicator_type, severity, matched_data_source, matched_field, matched_value | sort desc _time

Query 2: Correlate IOC Match with Endpoint Context

// Join TI matches with endpoint data for full context dataset = threat_intel_matches | filter indicator_type = "IP" | join type=inner ( dataset = xdr_data | filter event_type = NETWORK ) as endpoint_data on matched_value = endpoint_data.xdm.target.ipv4 | fields _time, indicator_value, endpoint_data.agent_hostname, endpoint_data.action_process_image_name

Query 3: Check Specific IOC Across All Data

// Search for a specific suspicious IP dataset = xdr_data, panw_ngfw_traffic_raw | filter xdm.target.ipv4 = "185.220.101.1" or xdm.source.ipv4 = "185.220.101.1" | fields _time, xdm.source.ipv4, xdm.target.ipv4, xdm.source.host.hostname, xdm.event.type | sort desc _time

Configuring IOC Matching Rules

Threat Intel → IOC Rules → + New Rule

What Are IOC Rules?

IOC Rules define what happens when an IOC matches. You can configure different actions based on severity, IOC type, or feed source.

Creating an IOC Rule

1
Define Match Criteria
IOC Types: IP, Domain (select which types this rule applies to)
Severity: High, Critical (only match high-confidence IOCs)
Feed: Unit 42, AlienVault (specific feeds only)
2
Define Actions
  • Generate Alert: Create incident in XSIAM (always recommended)
  • Trigger Playbook: Auto-run XSOAR playbook for enrichment/response
  • Block at Firewall: Push block rule to NGFW (if integrated)
  • Isolate Endpoint: Auto-isolate affected host (use carefully!)

Interview Questions

Setting up threat intel in XSIAM requires a phased approach that balances coverage with alert fatigue. I start with the built-in Unit 42 feed since it's already integrated and curated by Palo Alto's research team. This gives immediate coverage with minimal configuration - just verify it's enabled and create IOC rules for the actions you want on matches.

Next, I add free community feeds selectively. Abuse.ch feeds like URLhaus and Feodo Tracker are excellent for commodity malware with low false positive rates. I configure these as TAXII or CSV URL feeds with hourly polling. For AlienVault OTX, I'm more careful - it's community-sourced so I set it to only alert on High severity indicators and require correlation with other signals before creating incidents.

The key operational decision is what action to take on matches. I create tiered IOC rules: Critical severity IOCs from premium feeds trigger immediate alerts and can auto-run enrichment playbooks. Medium severity gets alerts but lower priority. Low severity I often just log without alerting - it's useful for hunting but would create too much noise in the alert queue. I also set up a weekly review dashboard showing matches per feed, false positive rates, and any missed detections from incident reviews.

After 30 days, I audit the results. Which feeds generated the most false positives? Which IOC types were most useful? This data drives optimization - I've disabled feeds that had 80%+ FP rates and added sector-specific ISAC feeds when they proved valuable. Threat intel is only useful if analysts trust it, so maintaining quality is essential.

False positive management in threat intel is a delicate balance. The temptation is to whitelist aggressively when you're drowning in alerts, but overly broad exclusions can blind you to real threats. My approach is to be surgical with exclusions and track everything.

When I see a false positive, I first determine why it's triggering. If a legitimate CDN IP is flagged, I don't just whitelist that IP - I investigate why it's in the feed. Sometimes IPs get flagged because they hosted malware temporarily. If the threat is no longer present and it's shared infrastructure like Cloudflare or AWS, I add it to exclusions with a note explaining the reasoning and the feed that flagged it.

I prefer tuning at the rule level rather than blanket exclusions. Instead of excluding all matches from a noisy feed, I create IOC rules that only alert when there's corroborating evidence - like an IOC match plus a suspicious process spawn, or IOC match plus data exfiltration behavior. This keeps the detection active but raises the threshold for alerting.

For tracking, I maintain metrics on exclusions: how many, when created, by whom, for what reason. I review these quarterly - sometimes I find exclusions for IPs that have changed hands and are now hosting threats again. The exclusion list should be a living document, not a dumping ground. I also track FP rates per feed using XQL: if a feed consistently shows 50%+ false positives, it might be time to deprioritize or remove it entirely.

IOC-based detection is reactive - you're looking for known-bad indicators that someone else has already identified from a previous attack. Behavioral detection is proactive - you're looking for suspicious patterns that could indicate an attack, even if you've never seen that specific malware or infrastructure before. Both are essential, but they serve different purposes.

IOCs are great for catching commodity threats and known attacker infrastructure. If an attacker uses the same C2 server across multiple campaigns, IOC feeds will catch it. The limitation is that sophisticated attackers rotate infrastructure constantly - a C2 domain might only be active for hours before they move to a new one. By the time it's in a feed, the attack might be over. IOCs also can't catch zero-days or custom malware with unique hashes.

Behavioral detection catches the attack techniques regardless of the specific indicators. PowerShell downloading and executing content from the internet is suspicious whether the URL is in a feed or not. Credential dumping from LSASS is malicious whether the tool is Mimikatz or a custom variant. This is why MITRE ATT&CK-aligned behavioral rules are so valuable - they detect the technique, not the tool.

In practice, I use both in layers. IOC matching catches the known threats cheaply and quickly - it's basically free detection. Behavioral rules catch the sophisticated threats that evade IOC feeds. When building a detection strategy, I make sure I have behavioral coverage for critical attack techniques like credential access, lateral movement, and data exfiltration, then layer IOC feeds on top for additional coverage. The ideal alert correlates both: "This host connected to a known C2 IP AND exhibited credential dumping behavior" is much higher confidence than either signal alone.

Most organizations use threat intel passively - they load IOCs and wait for matches. Proactive threat hunting with TI means using intelligence to guide where you look, not just what you look for. This requires thinking about TI as context rather than just indicators.

I start with threat reports, not just IOC lists. When Unit 42 or Mandiant publishes a report on a new campaign, I read the full TTPs, not just the IOC appendix. If they describe the attacker using scheduled tasks for persistence with a specific naming pattern, I'll hunt for that pattern in my environment even without a specific hash to look for. This behavioral approach catches variants the report might not have covered.

Another technique is infrastructure hunting. Attackers often register domains with similar patterns or use the same registrars and hosting providers. If I get an IOC for one C2 domain, I'll use passive DNS and certificate transparency logs to find related domains that might not be in feeds yet. XSIAM's threat intel module can be enriched with this discovered infrastructure before it shows up in commercial feeds.

I also use threat intel to prioritize hunting hypotheses. If reports indicate my industry sector is being targeted by a specific APT group, I'll prioritize hunting for their known techniques. If ransomware groups are exploiting a new vulnerability, I'll hunt for exploitation attempts even before patches are available. The key mindset shift is treating TI as a guide for investigation rather than just a blocklist - it tells you what adversaries are capable of, not just what specific indicators they've used in the past.

Each platform handles threat intel differently, and understanding these differences is crucial for effective integration. XSIAM has native TI management built into the platform - feeds, IOC matching, and response actions are all integrated. The Unit 42 feed is included and correlates across all data sources automatically. The limitation is flexibility; you're somewhat constrained to Palo Alto's approach and supported feed formats.

Sentinel's approach centers on Microsoft's threat intel and the TI Data Connector framework. The Microsoft Defender Threat Intelligence feed is excellent if you're in the Microsoft ecosystem. For external feeds, you typically ingest via the TI Indicators API or connect threat intelligence platforms like MISP. Sentinel's strength is the tight integration with Microsoft 365 Defender - IOC matches can trigger automatic response across email, endpoint, and identity. The challenge is that external feed integration requires more configuration and often Logic Apps for automation.

Splunk is the most flexible but also the most manual. Enterprise Security has threat intel framework with support for various feed formats, and you can use apps like MISP42Splunk for community platforms. Splunk's lookup tables make IOC matching performant even at scale. The downside is that you're building more yourself - response actions require separate integrations with your EDR, firewall, etc. There's no built-in "block this hash" button like XSIAM has.

For organizations choosing between them: XSIAM is best if you want integrated TI with minimal configuration. Sentinel excels if you're Microsoft-centric and want native Defender integration. Splunk wins if you need maximum flexibility and have the engineering resources to build custom integrations. I've worked with all three, and the key is matching the platform to your team's capabilities and existing infrastructure.
