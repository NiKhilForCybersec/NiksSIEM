Triage Playbook

Standard Triage Process (Tier 1)

  1. Review Incident Summary: Severity, alert count, affected entities
  2. Check XSIAM Stitching: Are related alerts properly grouped?
  3. Identify Key Indicators: IPs, hashes, users, hosts involved
  4. Context Enrichment: Check TI matches, asset criticality, user role
  5. Initial Assessment: True positive probability, scope estimation
  6. Decision: Self-handle, escalate, or close as false positive
  7. Document: Add notes before any status change

SOC Metrics & KPIs

Operational Metrics

XQL: SOC Performance Metrics
// Daily incident volume and status
dataset = xdr_incidents
| filter _time > ago(30d)
| alter day = format_timestamp(_time, "%Y-%m-%d")
| comp count() as total,
    count(status = "RESOLVED_TRUE_POSITIVE") as true_positives,
    count(status = "RESOLVED_FALSE_POSITIVE") as false_positives
    by day
| alter fp_rate = round(multiply(divide(false_positives, total), 100), 1)
// MTTR by severity
dataset = xdr_incidents
| filter status contains "RESOLVED"
| filter _time > ago(30d)
| alter resolution_hours = timestamp_diff(resolved_timestamp, creation_timestamp, "HOUR")
| comp avg(resolution_hours) as avg_mttr,
    percentile(resolution_hours, 90) as p90_mttr
    by severity
| sort asc severity
// Analyst workload distribution
dataset = xdr_incidents
| filter _time > ago(7d)
| filter assigned_to != null
| comp count() as incidents_handled,
    count(status contains "RESOLVED") as resolved
    by assigned_to
| alter resolution_rate = round(multiply(divide(resolved, incidents_handled), 100), 1)
| sort desc incidents_handled

Key Performance Indicators

KPI Target Measurement
MTTD (Mean Time to Detect) < 1 hour Event time to alert creation
MTTA (Mean Time to Acknowledge) < 15 minutes Alert creation to assignment
MTTR (Mean Time to Respond) < 4 hours Alert creation to resolution
False Positive Rate < 25% FP closures / total closures
Alert-to-Incident Ratio 100:1 or better Alerts / incidents (stitching effectiveness)
Automation Rate > 50% Auto-handled / total alerts
Detection Coverage > 70% MITRE Techniques with active detection

Automation Strategy

Automation Tiers

Progressive Automation Approach

  • Tier 0 - Auto-Close: Known false positives, authorized scanners, test traffic
  • Tier 1 - Enrichment: Auto-enrich all alerts with TI, asset, identity context
  • Tier 2 - Triage Assist: Auto-collect evidence, suggest severity, recommend actions
  • Tier 3 - Containment: Auto-isolate confirmed threats (with approval workflow)
  • Tier 4 - Full Response: End-to-end automation for high-confidence scenarios

Automation Candidates

Scenario Automation Level Playbook Actions
Phishing email reported Full automation Extract IOCs → Check TI → Block sender → Remove from mailboxes → Close
Known malware hash Full automation Quarantine file → Scan host → Block hash → Alert closed
Brute force detected Semi-automation Enrich IP → Check geo → Block if external → Analyst review
Suspicious process Enrichment only Collect process tree → Get file hash → TI lookup → Analyst review
Lateral movement Containment ready Map scope → Present to analyst → One-click isolation option

Post-Incident Review

PIR Template

  • Detection: How was the incident detected? Could we have detected sooner?
  • Response: What went well? What were the bottlenecks?
  • Containment: Was containment effective? Any gaps in capabilities?
  • Communication: Were stakeholders informed appropriately?
  • Improvements: What changes will prevent recurrence or improve response?
  • Action Items: Specific tasks with owners and due dates

Interview Answer: "How do you measure SOC effectiveness?"

"I measure SOC effectiveness across three dimensions: speed, accuracy, and coverage. For speed, I track MTTD, MTTA, and MTTR - how quickly we detect threats, acknowledge them, and resolve them. My targets are typically under 1 hour for detection, 15 minutes for acknowledgment, and 4 hours for resolution on high-priority incidents. For accuracy, I monitor false positive rate by detection rule and overall - anything above 25% triggers a tuning initiative. I also track true positive conversion rate to ensure we're catching real threats. For coverage, I maintain a MITRE ATT&CK mapping showing which techniques have active detections. In XSIAM, I also track the alert-to-incident ratio - XSIAM's stitching should give us at least 100:1 reduction, meaning analysts focus on incidents, not individual alerts. I present these metrics weekly to the team and monthly to leadership."

Interview Answer: "How do you handle a major incident in XSIAM?"

"For major incidents, I follow a structured process. First, I ensure the incident is escalated properly - Tier 3 analyst and SOC manager notified immediately. I establish a communication bridge or war room in XSOAR for coordination. I use XSIAM's incident view to understand the full scope - which hosts, users, and systems are affected. I leverage the causality chain view to understand the attack path. For containment, I use Live Terminal for immediate response actions - isolating compromised hosts, killing malicious processes, collecting evidence. I coordinate with IT for broader network-level containment if needed. Throughout, I document everything in the incident timeline. After containment, I work with the team on eradication and recovery. Post-incident, I lead a PIR to identify improvements - new detections, playbook updates, or process changes. The key is having pre-defined playbooks so we're not making decisions under pressure."
