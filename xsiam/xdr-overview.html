Understanding XDR

XDR (Extended Detection and Response) extends traditional EDR by correlating data across endpoints, networks, cloud, and identity sources to provide unified threat detection and response.

In XSIAM, Cortex XDR is the detection engine that processes normalized data (XDM) and generates alerts using behavioral analytics, machine learning, and rule-based detection.

BIOC Rules - Deep Dive

BIOC (Behavioral Indicator of Compromise) rules are the core endpoint detection mechanism in Cortex XDR. They detect malicious behaviors based on process execution, file operations, network connections, and registry modifications.

BIOC Rule Categories

üîÑ

Process BIOC

Detects suspicious process behaviors including execution chains, command-line arguments, and parent-child relationships.

  • Process name patterns (e.g., mimikatz, psexec)
  • Suspicious parent-child (Word ‚Üí PowerShell)
  • Command-line encoding detection
  • Process injection indicators
  • Living-off-the-land binaries (LOLBins)
üìÅ

File BIOC

Monitors file system operations for indicators of malicious activity, ransomware, or data exfiltration.

  • File creation in suspicious paths
  • Mass file modifications (ransomware)
  • Extension changes (document ‚Üí executable)
  • Sensitive file access patterns
  • Script drops in temp directories
üåê

Network BIOC

Analyzes network connections initiated by endpoint processes for C2 communication and data exfiltration.

  • Connections to rare destinations
  • Non-standard port usage
  • DNS tunneling indicators
  • Beaconing patterns
  • Large outbound data transfers
üîß

Registry BIOC

Tracks Windows registry modifications that indicate persistence, privilege escalation, or defense evasion.

  • Run/RunOnce key additions
  • Service creation (persistence)
  • Security policy modifications
  • Scheduled task registry changes
  • COM object hijacking

üíº Interview Answer: BIOC Rule Development Process
"Walk me through how you would create a custom BIOC rule for a new threat."

Step 1: Threat Research - Analyze the threat's TTPs from threat intel, sandbox analysis, or incident reports. Identify unique behavioral patterns.

Step 2: Field Identification - Determine which XDM fields can detect this behavior: process name, command line, parent process, file paths, network connections.

Step 3: Logic Design - Build AND/OR conditions that are specific enough to catch the threat but not so broad they generate false positives.

Step 4: Testing in Query - Use XQL in Investigation to test the logic against historical data. Validate true positives and check for false positives.

Step 5: Create BIOC Rule - Configure the rule with appropriate severity, MITRE mapping, and response actions.

Step 6: Monitor & Tune - Deploy in alert-only mode first, monitor for a week, add exclusions for legitimate tools, then enable blocking if appropriate.

üí° Severity Assignment Best Practices

Consider: (1) Confidence level of the detection, (2) Potential impact if true positive, (3) Attack stage (later stages = higher severity), (4) Asset criticality involved. A low-confidence detection on a critical asset may warrant medium severity, while high-confidence on a regular workstation might be medium.

üíº Interview Answer: Detection Engineering Philosophy
"How do you approach building detection content for a new environment?"

1. Baseline First: Understand what's normal in the environment. Run discovery queries to see common processes, network patterns, authentication flows.

2. Crown Jewels: Identify critical assets and build detections specifically protecting them - database servers, domain controllers, executive endpoints.

3. Kill Chain Coverage: Ensure detection at multiple attack stages. If initial access is missed, catch them at execution. If execution is missed, catch lateral movement.

4. Layer Detections: Use BIOC for endpoint behaviors, Analytics for anomalies, Correlation for cross-source attacks. Defense in depth applies to detection too.

5. Iterate: Detection engineering is never "done." Review alerts weekly, tune false positives, add new detections based on threat intel, and validate coverage with purple team exercises.

Interview Questions

BIOC and Analytics BIOC are complementary detection mechanisms that serve different purposes. Understanding when to use each is crucial for building effective detection coverage.

BIOC (Behavioral Indicators of Compromise) rules are deterministic‚Äîthey trigger when specific conditions are met. You define the exact behavior you're looking for: a specific process name with certain command-line arguments, file creation in a particular path, or registry modification at a known persistence location. They're excellent for known attack patterns, documented TTPs, and specific IOCs. If I know exactly what malicious behavior looks like, I write a BIOC rule.

Analytics BIOC uses machine learning to establish baselines and detect deviations. Instead of defining "alert when X happens," you're saying "alert when something unusual happens compared to normal." It's powerful for detecting novel threats, insider threats, and attacks that don't match known signatures. The downside is it requires time to learn the environment and can generate false positives if the baseline period includes anomalous activity.

In practice, I use both together. BIOC rules catch known-bad behaviors with high confidence‚Äîthings like encoded PowerShell commands or Mimikatz execution. Analytics BIOC catches the unknown-unknowns‚Äîa user suddenly accessing resources they've never touched, or a process making network connections to a destination never seen before. The combination provides defense in depth for detection.

Ransomware detection requires a layered approach because ransomware attacks progress through multiple stages, and we want to catch them as early as possible rather than waiting until encryption begins.

For initial access and execution, I create BIOC rules for common delivery mechanisms: suspicious macro execution from Office applications, script interpreters launched by email clients, or executables running from user Downloads or Temp folders. These might not be ransomware-specific, but they're common initial foothold techniques.

For defense evasion and privilege escalation, I look for shadow copy deletion (vssadmin.exe, wmic, or PowerShell commands targeting shadow copies), disabling of security tools, and process injection behaviors. The shadow copy detection is particularly important because it's a strong ransomware indicator‚Äîlegitimate software rarely deletes volume shadow copies.

For the encryption stage itself, I use File BIOC rules to detect mass file modifications: high rates of file renames, unusual extension changes, or creation of ransom note files (readme.txt, decrypt_instructions.html patterns). Analytics BIOC can supplement this by detecting file access patterns that deviate from a user's normal behavior.

I also create correlation rules that combine signals across the kill chain. A single shadow copy deletion might be a false positive from IT operations, but shadow copy deletion plus mass file modifications plus network connection to a rare external host is high-confidence ransomware. The correlation rule triggers only when multiple indicators appear within a time window.

Finally, I ensure we have automated response actions tied to high-severity ransomware detections‚Äînetwork isolation of the affected endpoint to prevent lateral spread, even before a human analyst reviews the alert.

All three are leading XDR platforms, but they have different architectural approaches and strengths. Understanding these differences helps when choosing a platform or integrating multiple solutions.

XSIAM/Cortex XDR has the advantage of tight integration with Palo Alto's ecosystem‚Äîfirewall logs, Prisma Cloud, and GlobalProtect VPN all feed into the same detection engine. The BIOC system provides granular endpoint detection, and the XQL query language is powerful for custom correlation. The unified SIEM+XDR+SOAR approach means detections can immediately trigger automated playbooks without crossing tool boundaries.

Microsoft Defender for Endpoint integrates deeply with the Microsoft ecosystem‚ÄîAzure AD, Intune, Office 365. If you're a Microsoft shop, the native integration provides excellent coverage for identity-based attacks and cloud security. MDE's detection engine is strong, and the recently expanded Sentinel integration allows correlation across Microsoft security products. The weakness is that non-Microsoft environments require more integration work.

CrowdStrike Falcon is known for its lightweight agent and strong threat intelligence. The Overwatch managed threat hunting service provides 24/7 expert coverage that many organizations can't build in-house. CrowdStrike excels at adversary tracking and provides detailed attribution when available. The platform is agent-centric, so network and cloud coverage typically requires additional products.

For rule creation, XSIAM uses XQL, Sentinel/MDE uses KQL, and CrowdStrike uses a proprietary rule format. The detection logic is conceptually similar, but translating rules between platforms requires understanding each platform's data schema and query capabilities. In practice, organizations often run multiple XDR solutions during migrations or in complex environments, so understanding how detections translate between platforms is valuable.

Tuning is one of the most important and ongoing tasks in detection engineering. The goal is finding the sweet spot where you catch real attacks without overwhelming analysts with noise. My approach is systematic and data-driven.

First, I establish what's triggering false positives. I query the alerts dataset to understand patterns: which rules fire most frequently, what hosts or users appear repeatedly, and whether there are temporal patterns (like scheduled tasks triggering every night). This data tells me where to focus tuning efforts.

For tuning, I prefer allowlisting over disabling rules. If a rule detecting PowerShell downloads fires for IT admin scripts, I add an exception for the specific admin accounts or source paths rather than disabling the rule entirely. This preserves detection for unexpected contexts while eliminating known-good noise.

I'm careful about over-tuning. Every exclusion is a potential blind spot. If I exclude "admin" users from a detection, and an attacker compromises an admin account, I've created a detection gap. I document every exclusion with justification and periodically review whether exclusions are still needed.

For threshold-based rules, I use historical data to set intelligent thresholds. Instead of guessing "5 failed logins is suspicious," I analyze the environment: what's the 95th percentile of failed logins per user? Setting the threshold slightly above that baseline catches genuine anomalies while avoiding false positives from legitimate lockouts.

Finally, I track tuning effectiveness over time. A rule that generated 100 alerts per day before tuning should be closer to 5-10 after tuning, and those remaining alerts should have a higher true positive rate. If tuning made no difference, I need to rethink the detection logic entirely.

The alert-to-incident pipeline is where XSIAM's XDR capabilities really shine. The platform automatically correlates related alerts into unified incidents, which dramatically reduces analyst workload and provides better attack context.

When a detection rule triggers, it creates an alert with associated metadata: timestamp, affected entities (host, user, IP), severity, MITRE ATT&CK mapping, and the evidence that triggered the detection. This alert goes into the alerts queue.

XSIAM then applies incident grouping logic to determine if this alert should join an existing incident or create a new one. The default grouping keys are typically based on shared entities‚Äîalerts affecting the same host, user, or related to the same process tree often group together. Time windows also matter; alerts within a configured window (often 24-72 hours) are more likely to group.

The incident stitching is intelligent. If I have an alert for suspicious PowerShell execution on Host A, and then an alert for lateral movement from Host A to Host B, XSIAM recognizes the relationship and groups them. The resulting incident shows the attack progression: initial execution ‚Üí lateral movement, with all evidence from both alerts accessible in one view.

Incident severity is calculated from the highest-severity alert in the group, though analysts can override this. The incident also inherits MITRE ATT&CK mappings from all grouped alerts, giving a visual representation of attack coverage across the kill chain.

For investigation, the incident provides a causality chain view showing how events relate‚Äîwhich process launched which child, which network connections were made, which files were accessed. This context makes triage much faster than investigating individual alerts in isolation.
