Attack Surface Management (Cortex Xpanse)

Complete guide to configuring and using Cortex Xpanse for external attack surface discovery. Covers initial setup, asset discovery, finding remediation, and integration with XSIAM.

What is Attack Surface Management?

Definition

Attack Surface Management (ASM) continuously discovers and monitors your organization's internet-facing assets. It answers: "What can attackers see from the outside?"

Cortex Xpanse is Palo Alto's ASM product, integrated into XSIAM.

What Xpanse Discovers

Infrastructure

IP ranges, domains, subdomains, cloud IPs

Web Properties

Websites, APIs, login portals, dev environments

Cloud Resources

AWS, Azure, GCP public resources, S3 buckets

Certificates

SSL/TLS certs, expiring certs, weak crypto

Where to Find ASM in XSIAM Console

XSIAM Console → Attack Surface (left navigation)

ASM Sub-sections:

Attack Surface → Overview Dashboard: Asset counts, findings by severity, trends
Attack Surface → Assets All discovered assets (IPs, domains, certs)
Attack Surface → Findings Security issues discovered (exposed services, vulns)
Attack Surface → Policies Configure what generates alerts
Attack Surface → Settings Seed data (domains, IPs to monitor)

How to Set Up Xpanse

Step 1: Configure Seed Data

Attack Surface → Settings → Seed Data

What is Seed Data?

Seed data tells Xpanse what belongs to your organization. It uses this as a starting point to discover related assets.

1
Add Root Domains

Enter your organization's primary domains:

yourcompany.com
yourcompany.co.uk
acquired-company.com
2
Add IP Ranges

Enter known public IP ranges (CIDR notation):

203.0.113.0/24 (Data center 1)
198.51.100.0/24 (Data center 2)
3
Add ASN (Optional)

If you have your own ASN, add it for automatic IP discovery

4
Wait for Discovery

Initial discovery takes 24-48 hours

Xpanse will discover subdomains, related IPs, cloud resources, and more

Understanding and Managing Findings

Viewing Findings

Attack Surface → Findings

Findings are security issues discovered on your attack surface. They're categorized by severity and type.

Example Findings with Remediation

Exposed RDP Service

CRITICAL

What: Remote Desktop Protocol (port 3389) exposed to internet on 203.0.113.45

Why it's bad: RDP is frequently brute-forced. BlueKeep and other RDP vulns are common attack vectors.

Where: 203.0.113.45:3389 (server-dc01.yourcompany.com)

Remediation Steps:
  1. Immediately block port 3389 inbound at firewall
  2. Implement VPN or jump host for remote access
  3. Enable Network Level Authentication (NLA)
  4. Review logs for brute force attempts
Console Action: Findings → Select finding → Assign to → [Team/Person] → Set Due Date

Unprotected S3 Bucket

HIGH

What: AWS S3 bucket "yourcompany-marketing-assets" allows public read access

Why it's bad: Public buckets can leak sensitive data. Often contain PII, credentials, or proprietary info.

Where: s3://yourcompany-marketing-assets (AWS us-east-1)

Remediation Steps:
  1. Review bucket contents for sensitive data
  2. Enable "Block Public Access" in AWS console
  3. Review bucket policy and ACLs
  4. Enable S3 access logging

Expiring SSL Certificate

MEDIUM

What: SSL certificate for api.yourcompany.com expires in 14 days

Why it's bad: Expired certs cause outages and security warnings, impacting customer trust

Remediation Steps:
  1. Identify cert owner/team
  2. Renew certificate
  3. Consider implementing auto-renewal (Let's Encrypt, ACM)

Configuring ASM Policies

Attack Surface → Policies → Edit Policy

Policies control which findings generate alerts and their severity.

Example Policy Configuration

Finding Type Default Severity Generate Alert? Auto-Remediate?
Exposed RDP Critical Yes - Immediate Optional (XSOAR playbook)
Exposed SSH High Yes No
Expiring Cert (30 days) Low Yes No
Expiring Cert (7 days) High Yes - Escalate No
New Subdomain Informational No (just log) No

ASM + XSIAM Integration

Automatic Incident Creation

Critical/High ASM findings automatically create incidents in XSIAM:

Incidents → Filter by Source = "Xpanse"

XSOAR Playbook Automation

Create playbooks to auto-remediate findings:

  • Exposed RDP: Auto-create firewall ticket, notify IT ops
  • Public S3: Auto-check for sensitive files, alert data owner
  • Expiring cert: Auto-create renewal ticket in ServiceNow

Interview Questions

Definition: ASM is the continuous discovery, inventory, and monitoring of an organization's external-facing digital assets.

Why it matters:

  • Shadow IT: Teams spin up resources IT doesn't know about
  • M&A blind spots: Acquired companies bring unknown infrastructure
  • Cloud sprawl: Easy to create public resources accidentally
  • Attackers' view: See what attackers see before they exploit it

In XSIAM: Cortex Xpanse provides ASM. You configure seed data (domains, IPs), it discovers related assets, identifies security issues (exposed services, weak crypto), and creates incidents for remediation.

Immediate response (within hours):

  1. Verify: Confirm the finding is accurate (Attack Surface → Findings → View details)
  2. Assess risk: Is this a production server? What data is on it?
  3. Block immediately: Create emergency firewall rule to block port 3389 from internet
  4. Check for compromise: Review auth logs for brute force attempts or successful unknown logins

Follow-up remediation:

  1. Implement proper remote access (VPN, jump host, Azure Bastion)
  2. Enable Network Level Authentication
  3. Patch for known RDP vulnerabilities (BlueKeep, etc.)
  4. Add to monitoring - create correlation rule for RDP brute force
