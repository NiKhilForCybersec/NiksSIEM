Data Normalization

XDM (XSIAM Data Model)

XDM Normalization: Purpose: - Standardize field names across sources - Enable cross-source correlation - Consistent query experience Common XDM Fields: ├── xdm.source.ipv4 ├── xdm.target.ipv4 ├── xdm.source.user.username ├── xdm.target.host.hostname ├── xdm.event.type ├── xdm.event.outcome └── xdm.alert.severity Mapping Process: 1. Raw data ingested 2. Parsing rules applied 3. Fields mapped to XDM 4. Normalized data stored

Common Issues: No Data Arriving: 1. Check source connectivity 2. Verify collector status 3. Check parsing rules 4. Review firewall rules Parsing Failures: 1. Review raw logs 2. Check parser configuration 3. Update regex patterns 4. Test with sample data Missing Fields: 1. Verify source configuration 2. Check field extraction 3. Update XDM mapping

Data Quality Best Practices

Ensuring Data Quality

Data Quality Checklist: COMPLETENESS: ☐ All expected sources sending data ☐ No unexpected gaps in timeline ☐ All fields populated correctly ACCURACY: ☐ Timestamps correct (timezone) ☐ Field values make sense ☐ No parsing errors TIMELINESS: ☐ Data arriving in near real-time ☐ No significant latency ☐ Alerts triggering promptly CONSISTENCY: ☐ Same events parse consistently ☐ Field names normalized ☐ XDM mapping complete

Data Validation Queries

// Check for data gaps dataset = xdr_data | comp count() by bin(_time, 1h) | where count < 1000 | sort asc _time // Check parsing success dataset = _raw_log | filter parsing_status != "success" | comp count() by parsing_status, vendor, product // Verify field population dataset = panw_ngfw | comp count() as total, count(src_ip != null) as has_src, count(dst_ip != null) as has_dst | alter src_pct = round(has_src * 100.0 / total, 2) // Check latency dataset = xdr_data | alter latency_mins = timestamp_diff(_insert_time, _time, "MINUTE") | where latency_mins > 5 | comp avg(latency_mins) by agent_hostname

Data Licensing

XSIAM Licensing Model

Licensing Considerations: Pro per Endpoint: - Priced per endpoint - XDR agent data included - Limited third-party data Pro per TB: - Priced per data volume - All data types supported - Third-party integrations - More flexible Data Counting: - Compressed data size - After normalization - Excludes XDR agent data (in per-endpoint) Optimization: - Filter unnecessary logs at source - Use appropriate retention - Monitor usage dashboards
  • Cloud-native architecture: Auto-scales to handle load
  • Broker VMs: Buffer and forward logs
  • Distributed processing: Parallel parsing and normalization
  • Compression: Efficient storage of normalized data
  • Hot/cold tiering: Optimize based on data age
  • Hot storage: Recent data, fast queries, full functionality, higher cost
  • Cold storage: Archived data, slower access, requires rehydration, lower cost

Strategy: Keep frequently queried data in hot storage, archive older data to cold for compliance.
