Data Ingestion Methods - Complete Configuration Guide

Comprehensive guide to all XSIAM data ingestion methods. For each method: what it is, when to use it, how to configure it step-by-step, and how to verify data is flowing.

Which Ingestion Method Should I Use?

Decision Tree

Q1: What type of source are you onboarding?
→ Endpoint (Windows, Mac, Linux workstation/server)

Use: XDR Agent - Install Cortex agent for rich telemetry and response

→ Network device (Firewall, Router, Switch, IDS/IPS)

Use: Broker VM Syslog - Deploy Broker VM, configure syslog forwarding

→ Cloud platform (AWS, Azure, GCP)

Use: Cloud Collector - Native API integration, no infrastructure needed

→ SaaS application (Okta, M365, Salesforce)

Use: Cloud Collector - Check if native integration exists first

→ Custom app with API/webhook capability

Use: HTTP Collector - Send logs via REST API

→ Windows servers without agent (Domain Controllers, etc.)

Use: Broker VM + Windows Event Forwarding

Method 1: XDR Agent

For endpoints - richest telemetry, response capabilities

When to Use

  • Windows, macOS, or Linux workstations and servers
  • You need process execution, file operations, network connections
  • You want remote response capabilities (isolate, kill, shell)

What Data You Get

Process executionCommand line, parent process, user, hash
File operationsCreate, modify, delete, rename
Network connectionsSrc/dst IP, port, bytes, process
Registry changesKey create/modify/delete (Windows)
Module loadsDLLs loaded by processes

How to Configure

Settings → Endpoints → Agent Installation
1
Download Agent Package

Settings → Endpoints → Agent Installation → Select OS → Download

2
Install on Endpoint
# Windows msiexec /i CortexXDR_x64.msi /qn DISTRIBUTION_ID="your-distribution-id" # macOS sudo installer -pkg CortexXDR.pkg -target / # Linux sudo rpm -ivh cortex-xdr-agent.rpm
3
Configure Data Collection Policy

Settings → Endpoints → Policy Management → Profiles → Edit → Data Collection

Enable: Process Execution, Command Line Logging, Network Connections, File Operations

4
Verify Data
dataset = xdr_data | filter agent_hostname = "YOUR-HOSTNAME" | filter _time > ago(1h) | comp count() by event_type

Method 2: Broker VM (Syslog)

For on-premises devices - firewalls, servers, network equipment

When to Use

  • Firewalls, routers, switches, load balancers
  • Linux servers sending syslog
  • Applications with syslog output
  • Any device that can send syslog (TCP/UDP)

Prerequisites

  • VMware or Hyper-V environment for Broker VM
  • Network path from log sources to Broker VM
  • Outbound HTTPS (443) from Broker VM to internet

How to Configure

Part A: Deploy Broker VM

Settings → Data Sources → Broker VM → + Add Broker VM
1
Download OVA and Registration Token

Click "Download OVA" and copy the registration token

2
Deploy in VMware

vSphere → Deploy OVF Template → Configure: 8 vCPU, 32 GB RAM, 1 TB disk

3
Configure Network & Register
# Access VM console, login with admin/admin broker-vm> configure network IP Address: 10.0.1.50 Subnet: 255.255.255.0 Gateway: 10.0.1.1 DNS: 10.0.1.10 broker-vm> register Token: [paste your token] # Wait for "Registration successful"

Part B: Create Syslog Data Source

Settings → Data Sources → Add Data Source → Syslog Collector
4
Configure Syslog Listener
Name:paloalto-firewall-logs
Broker VM:[Select your Broker VM]
Protocol:TCP (recommended)
Port:514
Vendor:Palo Alto Networks
Product:Firewall

Part C: Configure Source Device

5
Point Firewall to Broker VM

On Palo Alto NGFW:

# Device → Server Profiles → Syslog → Add Name: XSIAM-Broker Server: 10.0.1.50 Transport: TCP Port: 514 Format: BSD # Objects → Log Forwarding → Add Apply to security rules → Log at Session End
6
Verify Data
// Check Broker VM broker-vm> show listeners // Query in XSIAM dataset = panw_ngfw_traffic_raw | filter _time > ago(1h) | limit 10

Method 3: Cloud Collectors

For cloud platforms and SaaS - native API integration

When to Use

  • AWS (CloudTrail, GuardDuty, VPC Flow Logs)
  • Azure (Activity Log, Entra ID, Defender)
  • GCP (Cloud Audit Logs)
  • SaaS apps (Okta, Microsoft 365, Salesforce)

How to Configure: AWS CloudTrail Example

Settings → Data Sources → Cloud Collectors → + Add
1
Create Cloud Collector

Select "Amazon Web Services" → "CloudTrail"

2
Configure AWS Credentials
AWS Account ID:123456789012
Region:us-east-1
SQS Queue URL:https://sqs.us-east-1.amazonaws.com/123456789012/xsiam-cloudtrail
IAM Role ARN:arn:aws:iam::123456789012:role/XSIAMCloudTrailRole
3
Configure AWS Side
# Create IAM Role with trust policy for XSIAM # Attach policy: CloudTrailReadOnlyAccess, SQS permissions # Create SQS Queue for CloudTrail notifications # Configure CloudTrail to send to S3 with SQS notification
4
Verify Data
dataset = aws_cloudtrail_raw | filter _time > ago(1h) | comp count() by eventName | sort desc count | limit 20

How to Configure: Okta Example

1
Create Okta API Token

Okta Admin → Security → API → Tokens → Create Token

2
Configure in XSIAM
Okta Domain:yourcompany.okta.com
API Token:[paste token]
3
Verify Data
dataset = okta_okta_raw | filter _time > ago(1h) | limit 10

Method 4: HTTP Collector

For API-based log submission

When to Use

  • Custom applications that can send HTTP POST
  • Sources without syslog capability
  • Cloud services with webhook output
  • Custom scripts pulling from APIs

How to Configure

Settings → Data Sources → HTTP Collector → + Add
1
Create HTTP Collector
Name:custom-app-http
Vendor:Custom
Product:MyApplication
2
Get Endpoint URL and Token

After saving, XSIAM provides:

URL: https://api-{tenant}.xsiam.paloaltonetworks.com/logs/v1/event Token: eyJhbGciOiJIUzI1NiIs...
3
Send Logs via HTTP POST
curl -X POST "https://api-{tenant}.xsiam.paloaltonetworks.com/logs/v1/event" \ -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..." \ -H "Content-Type: application/json" \ -d '[{"timestamp":"2024-02-15T14:30:00Z","user":"jsmith","action":"login"}]'
4
Verify Data
dataset = custom_myapplication_raw | filter _time > ago(1h) | limit 10

Interview Questions

Four main methods:

  1. XDR Agent: For endpoints. Install on Windows/Mac/Linux for process, file, network telemetry plus response capabilities.
  2. Broker VM: For on-prem devices. Deploy VM to collect syslog from firewalls, servers, network equipment. Buffers and securely uploads to cloud.
  3. Cloud Collectors: For cloud/SaaS. Native integrations with AWS, Azure, GCP, Okta, M365. Direct cloud-to-cloud, no infrastructure needed.
  4. HTTP Collector: For custom sources. REST API endpoint for applications that can send HTTP POST. Good for webhooks and custom scripts.

Decision framework: Endpoint → Agent. Network device → Broker VM. Cloud/SaaS → Check for native collector first. Custom app → HTTP if no syslog.
