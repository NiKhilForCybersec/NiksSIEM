ML Tuning & False Positive Management

Learning Period Required

ML-based detections require a learning period (typically 14-30 days) to build accurate baselines. During this period, expect higher false positive rates. Don't disable ML detections prematurely - allow time for models to stabilize.

Tuning Strategies

Interview Questions

Evaluating ML-based detections requires different metrics than rule-based detection because you're dealing with probabilistic rather than deterministic alerts. I focus on several key areas.

True Positive Rate (Precision) - Of the alerts generated, how many were actual threats or policy violations? Low precision means analyst time is wasted on false positives. I track this by requiring analysts to classify ML alerts as TP/FP during triage.

Detection Coverage - Are the ML models finding things that rule-based detection missed? I look for cases where ML alerts led to incident discovery that wouldn't have been caught otherwise. This validates the value of behavioral analytics.

Time to Detection - For confirmed incidents, did ML detect earlier than traditional methods? Behavioral anomalies often appear before signatures, so ML should theoretically catch attacks earlier in the kill chain.

False Positive Trending - Is the FP rate decreasing over time as models learn? After the initial learning period, FP rates should stabilize and gradually decrease with analyst feedback. If they're not, something is wrong with the data or model configuration.

Baseline Accuracy - Spot-check behavioral profiles for accuracy. Does the system correctly understand user patterns? Incorrect baselines lead to both false positives (alerting on normal behavior) and false negatives (missing anomalies because the baseline is wrong).

ML and UEBA are powerful but have real limitations that security teams need to understand to use them effectively.

Learning period vulnerability - During the initial baseline period, ML can't detect anomalies because it doesn't know what's normal yet. If an attacker is already present when you deploy UEBA, their behavior becomes part of the baseline.

Low-and-slow attacks - Sophisticated attackers who stay within normal behavioral bounds won't trigger anomaly detection. If they access only resources the compromised user normally accesses, at normal times, from expected locations, ML sees nothing unusual.

Concept drift - User behavior changes legitimately over time (new projects, role changes, remote work). ML systems must adapt to these changes, but rapid adaptation can allow attackers to gradually shift the baseline maliciously.

Data quality dependency - ML is only as good as its input data. Missing log sources, inconsistent field mappings, or data quality issues lead to inaccurate baselines and unreliable detection.

Explainability challenges - When ML alerts fire, it can be hard to explain exactly why to stakeholders. "The model detected an anomaly" is less satisfying than "the user logged in from Russia at 3 AM."

These limitations don't mean ML is useless - it catches things rules miss. But it should complement, not replace, rule-based detection. The best detection strategies layer multiple approaches.
