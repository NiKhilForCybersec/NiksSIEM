Correlation Rules

Create custom detection rules in XSIAM using XQL. Correlation rules run on scheduled intervals to detect threats across all ingested data sources.

What Are Correlation Rules?

Correlation rules are scheduled XQL queries that generate alerts when specific conditions are met. Unlike BIOC rules (which run on endpoint agents in real-time), correlation rules:

  • Run on all data in Cortex Data Lake (not just endpoint data)
  • Execute on a schedule (every 5 min to 24 hours)
  • Can correlate across multiple data sources (firewall + endpoint + identity)
  • Support aggregation and thresholds (e.g., 5 failed logins in 10 minutes)

Correlation Rules vs BIOC Rules

AspectCorrelation RulesBIOC Rules
Execution LocationCortex Data Lake (cloud)XDR Agent (endpoint)
Data SourcesAll ingested dataEndpoint telemetry only
LatencyMinutes (scheduled)Real-time
Use CaseCross-source correlation, thresholdsProcess chains, behavioral
LanguageXQLYAML-based

Creating Correlation Rules

Detection & Response → Detection Rules → Correlation

Step-by-Step Process

  1. Define the detection logic: Write XQL query that returns events to alert on
  2. Set scheduling: How often the rule runs and time window to search
  3. Configure alert settings: Severity, MITRE mapping, alert name
  4. Map entities: Define which fields represent host, user, IP for incident correlation
  5. Test and tune: Run query manually, review results, adjust thresholds

Rule Configuration

FieldDescriptionBest Practice
Rule Name Display name for the rule Use format: "[Source] - [Technique] - [Action]"
Example: "Firewall - Port Scan - Blocked"
Severity High, Medium, Low, Informational High = confirmed malicious, Medium = suspicious, Low = anomalous
MITRE ATT&CK Tactic and technique mapping Map to most specific technique (T1xxx.xxx)
Schedule How often rule executes Match to detection urgency (critical = 5min, low = 1hr)
Time Window How far back to search Should overlap schedule (15min window for 10min schedule)
Alert Grouping How to group multiple matches Group by entity (user, host) to reduce alert volume

XQL Query Structure for Correlation Rules

Correlation rule queries must return fields that XSIAM can use for alerting:

// Basic correlation rule structure dataset = <dataset_name> | filter <conditions> | <aggregation or processing> | alter xdm.alert.name = "Alert Name", xdm.alert.description = "Description of what was detected", xdm.alert.severity = "high" // high, medium, low, informational

Production-Ready Correlation Rules

T1110.001 Brute Force Authentication Attack

Detection: Multiple failed logins followed by success from same source

Severity: High

dataset = xdr_identity_data | filter event_type = "authentication" | filter auth_result in ("failure", "success") | comp failures = countif(auth_result = "failure"), successes = countif(auth_result = "success"), first_failure = minif(_time, auth_result = "failure"), success_time = maxif(_time, auth_result = "success") by src_ip, target_user over 15m | filter failures >= 5 and successes >= 1 | filter success_time > first_failure // Success came after failures | alter xdm.alert.name = concat("Brute Force Success: ", target_user), xdm.alert.description = concat( "User ", target_user, " had ", to_string(failures), " failed logins followed by successful login from ", src_ip ), xdm.alert.severity = "high", xdm.source.ipv4 = src_ip, xdm.source.user.username = target_user

Schedule: Every 5 minutes | Window: 15 minutes

T1048 Potential Data Exfiltration

Detection: Unusually large outbound data transfer to external IP

Severity: High

dataset = panw_ngfw_traffic_raw | filter direction = "outbound" | filter dst_ip_public = true | comp total_bytes = sum(bytes_sent), connection_count = count(), destinations = count_distinct(dst_ip) by src_ip, src_host over 1h | filter total_bytes > 1073741824 // 1 GB | alter xdm.alert.name = concat("Large Outbound Transfer from ", src_host), xdm.alert.description = concat( "Host ", src_host, " transferred ", to_string(round(total_bytes / 1048576, 2)), " MB to ", to_string(destinations), " external destinations" ), xdm.alert.severity = "high", xdm.source.host.hostname = src_host, xdm.source.ipv4 = src_ip

Schedule: Every 30 minutes | Window: 1 hour

T1021.002 SMB Lateral Movement

Detection: Single host connecting to multiple internal hosts via SMB

Severity: Medium

dataset = panw_ngfw_traffic_raw | filter app = "ms-ds-smb" or dst_port in (445, 139) | filter src_ip_private = true and dst_ip_private = true | comp unique_targets = count_distinct(dst_ip), total_connections = count() by src_ip, src_host over 30m | filter unique_targets >= 5 // Connected to 5+ internal hosts | alter xdm.alert.name = concat("SMB Scanning from ", src_host), xdm.alert.description = concat( "Host ", src_host, " connected to ", to_string(unique_targets), " internal hosts via SMB in 30 minutes" ), xdm.alert.severity = "medium", xdm.source.host.hostname = src_host, xdm.source.ipv4 = src_ip

Schedule: Every 10 minutes | Window: 30 minutes

T1078 Impossible Travel Detection

Detection: User authentication from geographically distant locations in short time

Severity: Medium

dataset = xdr_identity_data | filter event_type = "authentication" and auth_result = "success" | sort _time asc | comp locations = arrayconcat(geo_country), ips = arrayconcat(src_ip), times = arrayconcat(_time), login_count = count() by target_user over 2h | filter login_count >= 2 | filter array_length(arraydistinct(locations)) >= 2 // Multiple countries | alter xdm.alert.name = concat("Impossible Travel: ", target_user), xdm.alert.description = concat( "User ", target_user, " authenticated from multiple countries: ", arraystring(arraydistinct(locations), ", ") ), xdm.alert.severity = "medium", xdm.source.user.username = target_user

Schedule: Every 15 minutes | Window: 2 hours

T1071 Known Malicious Domain Communication

Detection: DNS query or connection to threat intel flagged domain

Severity: High

dataset = panw_ngfw_dns_raw | filter query_type in ("A", "AAAA", "CNAME") | join type=inner ( dataset = threat_intel_indicators | filter indicator_type = "domain" | filter confidence >= 80 | fields indicator_value, threat_type, source ) as ti on query_name = ti.indicator_value | alter xdm.alert.name = concat("Malicious Domain: ", query_name), xdm.alert.description = concat( "Host ", src_host, " queried known malicious domain ", query_name, " (", ti.threat_type, " - Source: ", ti.source, ")" ), xdm.alert.severity = "high", xdm.source.host.hostname = src_host, xdm.source.ipv4 = src_ip

Schedule: Every 5 minutes | Window: 10 minutes

T1078.002 New Admin Account Created

Detection: User added to privileged group (Domain Admins, Enterprise Admins)

Severity: High

dataset = msft_security_events_raw | filter event_id in (4728, 4732, 4756) // Member added to security group | filter target_group_name in ( "Domain Admins", "Enterprise Admins", "Schema Admins", "Administrators", "Account Operators", "Backup Operators" ) | alter xdm.alert.name = concat("Privileged Group Modified: ", target_group_name), xdm.alert.description = concat( "User ", added_member, " was added to ", target_group_name, " by ", actor_user, " from ", src_host ), xdm.alert.severity = "high", xdm.source.user.username = actor_user, xdm.source.host.hostname = src_host

Schedule: Every 5 minutes | Window: 10 minutes

Aggregation Patterns

Threshold-Based Detection

// Count events exceeding threshold | comp count() as event_count by src_ip over 15m | filter event_count >= 100 // Sum values exceeding threshold | comp sum(bytes) as total_bytes by src_host over 1h | filter total_bytes > 1073741824 // 1 GB // Distinct count threshold | comp count_distinct(dst_ip) as unique_dests by src_ip over 30m | filter unique_dests >= 50

Time-Window Correlation

// Events in sequence (A then B) | comp event_a_time = minif(_time, event_type = "A"), event_b_time = maxif(_time, event_type = "B") by entity | filter event_b_time > event_a_time // B happened after A // Events within time window | comp first_event = min(_time), last_event = max(_time) by entity over 1h | filter timestamp_diff(last_event, first_event, "MINUTE") <= 10

Statistical Anomaly Detection

// Baseline comparison (requires historical data) | comp avg(event_count) as baseline by src_host over 7d | join type=inner ( // Current activity dataset = current_data | comp count() as current_count by src_host over 1h ) as current on src_host = current.src_host | filter current.current_count > baseline * 3 // 3x normal

Testing and Tuning

Pre-Deployment Testing

  1. Run query manually: Execute in Investigation → Query Center
  2. Review result volume: Check how many alerts it would generate
  3. Validate detections: Confirm results are true positives
  4. Check performance: Note query execution time
  5. Test edge cases: Verify threshold boundaries work correctly

Tuning Strategies

ProblemSolution
Too many false positives Add exclusions, increase thresholds, narrow scope
Missing detections Lower thresholds, broaden filters, extend time window
Alert fatigue Group by entity, increase threshold, add context filters
Slow query performance Add time filters early, reduce fields, limit datasets

Version Control Best Practices

  • Export rules to YAML/JSON and store in Git repository
  • Document changes in rule description or comments
  • Use naming conventions that include version (e.g., "v2")
  • Test changes in test mode before enabling in production

Regular Review Cadence

  • Weekly: Review high-volume rules, tune false positives
  • Monthly: Review rule effectiveness metrics
  • Quarterly: Full rule inventory audit, retire unused rules
  • On-Demand: After incidents, after threat intel updates

Performance Optimization

Query Optimization Tips

// BAD: Filter after aggregation dataset = panw_ngfw_traffic_raw | comp sum(bytes) as total by src_ip over 1h | filter total > 1000000000 // GOOD: Filter early to reduce data dataset = panw_ngfw_traffic_raw | filter bytes > 10000 // Only large transfers | filter direction = "outbound" // Narrow scope | comp sum(bytes) as total by src_ip over 1h | filter total > 1000000000 // BAD: Select all fields dataset = xdr_data | filter ... // GOOD: Select only needed fields dataset = xdr_data | filter ... | fields _time, src_ip, dst_ip, event_type, bytes

Interview Questions

My correlation rule development follows a structured process that ensures the rule is effective, tuned, and documented before it goes into production.

I start with the use case definition. Before writing any XQL, I document what attack or behavior I'm trying to detect and map it to MITRE ATT&CK. This forces me to think clearly about the detection goal and helps with later documentation. For example, "Detect brute force authentication attempts - T1110.001 Password Guessing."

Next, I identify the data sources. I determine which datasets contain the relevant events and verify that data is actually flowing and being parsed correctly. For brute force, I'd look at identity_logs, azure_ad_signin_raw, or whatever authentication source is relevant. I run exploratory queries to understand what the data looks like.

Then I build the XQL query iteratively. I start with basic filters to find the event type I care about, then add aggregation logic (like counting failed logins per user), then threshold filtering. I test each step manually in Query Center to verify results. For the brute force example: dataset = identity_logs | filter outcome = "FAILURE" | filter _time > ago(15m) | comp failed_count = count() by username, source_ip | filter failed_count >= 10

After the query logic is solid, I configure the rule settings: schedule interval (how often it runs), time window (how far back it looks), severity, and MITRE mapping. I ensure the time window is at least 1.5x the schedule interval to prevent gaps. I also configure entity mapping so the alert includes host and user information for incident correlation.

I deploy in test mode first—the rule runs but alerts go to a test queue instead of production. I review alerts over a few days to identify false positives and refine exclusions. Common exclusions include service accounts, automated systems, or known scanner IPs.

Finally, I document the rule: what it detects, expected false positive sources, tuning guidance for future maintainers, and the MITRE ATT&CK coverage it provides. Good documentation makes rule maintenance sustainable as the team changes.

Correlation rules and BIOC rules are complementary detection mechanisms that run in different places and serve different purposes. Choosing the right one depends on what you're detecting and how fast you need the alert.

BIOC rules run on the endpoint agents themselves in real-time. They have access to rich endpoint context—process trees, parent-child relationships, command-line arguments, file operations, registry changes. Because they run locally, they can detect and respond almost instantly. The limitation is they can only see endpoint data; they don't have visibility into network traffic, cloud logs, or identity events from other sources.

Correlation rules run in the cloud on the Cortex Data Lake on a scheduled basis. They can query any data source—endpoint telemetry, firewall logs, cloud audit logs, identity events, custom data sources. This makes them essential for cross-source correlation like "user authenticated from unusual location AND accessed sensitive files." The trade-off is latency; rules run every few minutes to hours, not in real-time.

My selection criteria: If I'm detecting an endpoint-specific behavior with a known pattern—like suspicious process chains, encoded PowerShell, or file creation in temp directories—I use a BIOC rule for speed. If I need to correlate across sources, apply complex aggregation logic, or use baseline comparisons, I use a correlation rule. Many sophisticated detections use both: a BIOC rule for immediate detection of suspicious execution, and a correlation rule that adds context from network and identity logs for higher-fidelity alerting.

False positive reduction is a balancing act—every exclusion or threshold increase is a potential blind spot. My approach prioritizes visibility into what I'm excluding and why.

First, I analyze the false positives systematically. I query the alerts to understand patterns: Are they coming from specific hosts? Specific users? Specific time windows? If 90% of false positives come from three service accounts, I can exclude those accounts specifically rather than raising the threshold globally.

For threshold-based rules, I prefer data-driven tuning over guessing. Instead of saying "10 failed logins is suspicious," I query historical data to find the 95th or 99th percentile of normal activity. Setting the threshold just above that baseline catches genuine anomalies while filtering normal variance.

I'm careful about over-excluding. Every exclusion should be documented with justification. If I exclude an admin group from a detection, and an attacker compromises an admin account, that's a gap. Where possible, I create separate rules for excluded populations with adjusted thresholds rather than blind exclusions.

For high-noise rules, I sometimes add a confidence scoring approach. Instead of alerting on a single indicator, I require multiple indicators to be present. For example, don't alert on just "connection to rare domain"—alert on "connection to rare domain AND from a process running from temp directory AND during non-business hours." Each additional condition increases confidence and reduces false positives.

I also review exclusions periodically. Business processes change—that scanner IP that was legitimate six months ago might have been decommissioned and reassigned. Annual exclusion reviews catch stale entries that are now creating blind spots.

Timing configuration is one of the most common sources of detection gaps in correlation rules, and there are several aspects to get right.

The most important rule is: time window must be greater than schedule interval. If a rule runs every 15 minutes but only looks back 15 minutes, there's no overlap, and events at the boundary can be missed or double-counted depending on exact timing. My standard is setting the window to at least 1.5x the schedule interval—a rule running every 15 minutes should look back at least 22-23 minutes.

Data ingestion latency matters too. Logs don't appear instantly in XSIAM—there's collection, parsing, and indexing delay. If your environment has 5-minute average ingestion latency, your time window needs to account for that. A rule looking at "last 10 minutes" might miss events from minutes 6-10 if they're still being processed.

For deduplication, I use query logic that accounts for overlap. If the same event appears in consecutive rule executions, the rule should handle that gracefully—either by using unique event identifiers or by designing the aggregation to be idempotent.

I also monitor rule execution health. XSIAM provides visibility into rule runs—did it complete successfully? How long did it take? Did it timeout? A rule that consistently times out isn't providing reliable coverage. If execution time is approaching the schedule interval, I need to optimize the query or increase the interval.

Finally, I test with known-bad data. During development, I inject test events that should trigger the rule and verify they actually do. This catches timing issues that aren't obvious from just reading the configuration.

Data exfiltration detection requires correlating multiple signals because individual indicators are often too noisy on their own. My approach combines volume-based detection, behavioral indicators, and context enrichment.

For volume-based detection, I look at outbound data transfers that exceed normal thresholds. The query aggregates bytes transferred per user or host per time window and compares to a baseline. Something like: dataset = network_logs | filter direction = "outbound" | comp total_bytes = sum(bytes_out) by source_user | join (baseline subquery) | filter total_bytes > baseline * 3. The baseline comparison is crucial—some users legitimately transfer large amounts of data.

For behavioral indicators, I correlate with suspicious precursor activities. Large data transfer alone is noise, but large data transfer after someone accessed files they've never accessed before, or after staging data to a temp directory, is much higher confidence. The correlation rule joins network volume data with file access patterns from endpoint telemetry.

Destination matters too. Transfer to cloud storage providers (Dropbox, Google Drive, personal OneDrive) is riskier than internal transfers. I maintain a list of known cloud storage domains and weight those connections higher in the detection logic.

Time of day context helps reduce false positives. Backups and legitimate large transfers often happen during scheduled maintenance windows. Exfiltration often happens during off-hours when the attacker thinks they're less likely to be noticed. Including time-based context improves signal quality.

The final rule combines these signals with confidence scoring. An alert might say "User X transferred 500MB to Google Drive (unusual), after accessing HR salary files for the first time (unusual), at 2 AM (unusual)." Any single indicator might be explained; the combination is highly suspicious.
