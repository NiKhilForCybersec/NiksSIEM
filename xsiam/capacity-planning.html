Capacity Planning Framework

Why Capacity Planning Matters

Proper capacity planning prevents two expensive problems: over-provisioning (wasting money on unused capacity) and under-provisioning (performance problems, data loss, detection gaps). For XSIAM, capacity planning focuses on data volume, Broker VM sizing, and license allocation.

Broker VM Sizing Guide

Broker VM Purpose

Broker VMs collect logs from on-premises sources that can't send directly to cloud. They aggregate, compress, and forward data securely. Proper sizing ensures no data loss during peak periods and leaves headroom for growth.

Deployment Size EPS Capacity vCPU Memory Disk Network
Small Up to 5,000 EPS 4 vCPU 16 GB 200 GB SSD 1 Gbps
Medium 5,000 - 15,000 EPS 8 vCPU 32 GB 500 GB SSD 1 Gbps
Large 15,000 - 40,000 EPS 16 vCPU 64 GB 1 TB SSD 10 Gbps
Enterprise 40,000+ EPS 32 vCPU 128 GB 2 TB SSD 10 Gbps

Broker VM Sizing Formula

# Broker VM Sizing Calculations # ═══════════════════════════════════════════ ## CPU Calculation Base: 2 vCPU Per 5,000 EPS: +2 vCPU Formula: vCPU = 2 + (EPS / 2,500) Example (20,000 EPS): vCPU = 2 + (20,000 / 2,500) = 10 vCPU Recommendation: Round up to 12 vCPU for headroom ## Memory Calculation Base: 8 GB Per 5,000 EPS: +8 GB Formula: Memory = 8 + (EPS / 625) Example (20,000 EPS): Memory = 8 + (20,000 / 625) = 40 GB Recommendation: Round up to 48 GB ## Disk Calculation (Buffer Storage) Purpose: Queue data during network outages or cloud issues Target: 24 hours of buffer at peak EPS Formula: Disk GB = (Peak EPS × 86,400 × Avg Event Size KB) / (1024 × 1024) Example (20,000 EPS, 1KB avg event): Disk = (20,000 × 86,400 × 1) / 1,048,576 = 1,648 GB Recommendation: 2 TB SSD with 20% free space buffer ## High Availability Considerations - Deploy 2 Broker VMs in active-passive or load-balanced config - Each Broker sized for full load (N+1 redundancy) - Different physical hosts / availability zones

Capacity Cost Optimization

Data Reduction Strategies

Strategy Typical Savings Implementation Risk Level
Filter noisy events at source 20-50% Exclude debug logs, health checks, known-good traffic Medium
Aggregate similar events 10-30% Roll up repeated events into counts Low
Optimize retention tiers 30-50% cost Move older data to cold storage faster Low
Remove duplicate collection 10-20% Audit for same logs from multiple paths Low
Right-size verbosity 20-40% Reduce logging levels where appropriate Medium

Data Reduction Warning

Before filtering or aggregating data, ensure you understand the security implications. Removing logs to save cost can create detection blind spots. Always validate with your detection engineering team that filtered events aren't needed for any correlation rules or hunting queries.

Interview Questions

Capacity planning for a new XSIAM deployment starts with understanding what you're protecting and what data you need to collect. I approach this systematically, working from business requirements down to technical specifications.

First, I build a data source inventory. Work with the infrastructure, network, identity, and cloud teams to catalog every potential log source. For each source, I need to know: what type of logs, how many instances, typical event rates, and whether collection is required (compliance) or desired (security value). Don't guess at EPS - either look at existing SIEM data if you're migrating, or deploy test collectors to measure actual rates.

With the inventory complete, I calculate total daily volume. Sum up the EPS across all sources, multiply by 86,400 seconds, then by average event size. Add 20-30% buffer for growth and variability. This gives you the license sizing requirement and informs retention cost calculations.

For infrastructure sizing, the main decision is Broker VM configuration. I size Brokers based on the EPS they'll handle, with enough CPU and memory to process that volume plus buffer for queuing during network issues. Rule of thumb: 4 vCPU and 16 GB RAM per 5,000 EPS, with SSD storage for 24 hours of buffer. Always deploy in HA pairs - a single Broker failure shouldn't cause data loss.

Finally, I document assumptions and build in review cycles. Initial estimates are rarely perfect. Plan for quarterly capacity reviews where you compare actual consumption to projections and adjust the forward forecast. Build relationships with stakeholders who know about upcoming changes - the infrastructure team planning a cloud migration, HR planning headcount growth, M&A team with acquisition pipeline.

Eighty percent CPU utilization on a Broker VM is in the warning zone - not critical yet, but needs attention before it becomes a problem. My response involves immediate assessment, short-term mitigation, and longer-term planning.

For immediate assessment, I'd look at the utilization pattern. Is 80% the sustained average, or is it spiking during peak hours? Check if this is a recent change or gradual trend. Look at the other resources too - memory, disk I/O, queue depth. If CPU is the only constraint, it's likely a processing bottleneck. If queue depth is also high, we might be approaching the point where data backs up.

Short-term mitigation depends on the cause. If there's a sudden spike from a new data source, maybe we can tune that source's logging level or filter unnecessary events. If it's gradual growth, we need to plan scaling. Check if there are any inefficient collection configurations - sometimes a misconfigured syslog source sends duplicate data or excessive debug logs.

For longer-term planning, 80% utilization means I need to scale within the next quarter. Options include: vertical scaling (add more vCPU to the existing Broker - usually the quickest), horizontal scaling (add another Broker and load balance), or source redistribution (move some data sources to a different Broker if you have multiple). I'd typically choose vertical scaling first if the hypervisor has capacity, since it doesn't require reconfiguring log sources.

Throughout this process, I'd update the capacity plan to reflect actual growth rates. If we hit 80% faster than expected, our projections were optimistic. Adjust the growth rate assumption and recalculate when we'll need the next capacity increase.

Estimating EPS for unknown data sources is part art, part science. I use a combination of vendor documentation, industry benchmarks, and measurement when possible.

Vendor documentation is the first source. Most enterprise products document their logging volume in deployment guides or sizing documentation. Palo Alto publishes EPS estimates for NGFW based on traffic volume, Microsoft documents audit log volumes for various M365 services. These estimates are usually ranges - look at the high end for planning purposes.

Industry benchmarks provide reference points. A typical enterprise firewall generates 500-5,000 EPS depending on traffic. A domain controller produces 50-500 EPS depending on authentication volume. Endpoint agents generate 10-50 EPS per endpoint. These benchmarks come from experience across deployments - they're not exact but they're reasonable starting points.

If possible, measure before committing. Deploy a test collector or enable logging to an existing SIEM for a week. Measure actual EPS during business hours and overnight, capture peak rates, and calculate averages. This real data is much more reliable than estimates. For cloud sources, most providers have dashboards showing API call volumes or event counts that can inform estimates.

When estimating, I also consider what drives the volume. User activity-driven sources (authentication, email) scale with headcount. Traffic-driven sources (firewalls, proxies) scale with network throughput. Application sources scale with transaction volume. Understanding the driver helps predict how the EPS will change as the business grows.

Finally, I always add buffer to estimates - typically 50% for unknown sources. It's better to over-provision initially and optimize later than to under-provision and have capacity problems. As we collect real data, we can refine the estimates and adjust capacity accordingly.

This is one of the fundamental tensions in security operations - more data means better visibility but also higher costs. The key is making informed trade-offs based on security value, not arbitrary cost cutting.

Start by categorizing data sources by security value. Tier 1 sources are essential for detection - endpoint telemetry, authentication logs, firewall denies. These are non-negotiable and should never be filtered for cost reasons. Tier 2 sources add context and enable investigation - DNS queries, proxy logs, application logs. Tier 3 sources are nice-to-have for completeness - debug logs, performance metrics, redundant collection paths.

Cost optimization should focus on Tier 3 first, then careful optimization of Tier 2. For example, do you need verbose DNS logging at full fidelity, or would aggregated statistics meet your needs? Are you collecting the same authentication events from both the DC and Okta? Can debug logging be disabled in production systems?

Retention tiers offer another optimization lever without reducing collection. Hot storage costs significantly more than cold. If you only need 30 days of fast query access but 365 days for compliance, configure aggressive tier movement. You're still collecting everything, just storing older data more efficiently.

I also involve detection engineering in these decisions. Before filtering or aggregating any data source, review it against active correlation rules and hunting queries. What looks like noisy, low-value data might be essential for a specific detection. The detection team can identify which fields they actually need - maybe you can collect summary events instead of verbose logs while preserving security value.

Document every optimization decision with the rationale and potential security impact. This creates accountability and enables reversal if a cost-saving measure creates a detection gap. The goal is conscious trade-offs, not blind cost cutting that creates unknown risk.
