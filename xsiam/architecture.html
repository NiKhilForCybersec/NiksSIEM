Component Deep Dive

Cortex XDR Agent

Endpoint sensor and protection

What It Does

  • Collects telemetry: Process execution, file operations, network connections, registry changes
  • Provides protection: Malware blocking, exploit prevention, ransomware protection
  • Enables response: Remote shell, file retrieval, network isolation, process kill

Where in Console

Settings → Endpoints → Agents (view deployed agents)
Settings → Endpoints → Policy Management → Profiles (configure behavior)
Settings → Endpoints → Agent Installation (download packages)

Key Configuration

Data Collection:Profile → Data Collection → Enable Process/Network/File/Registry collection
Protection Mode:Profile → Malware Protection → Block and Quarantine vs Report Only
Response Actions:Profile → Response Actions → Enable Live Terminal, Isolation, etc.

Data Produced

// Agent data goes to xdr_data dataset dataset = xdr_data | filter agent_hostname = "WORKSTATION-001" | comp count() by event_type // Common event_type values: // PROCESS, FILE, NETWORK, REGISTRY, LOAD_IMAGE, AUTH

Broker VM

On-premises data collection gateway

What It Does

  • Receives logs: Syslog (TCP/UDP), Windows Event Forwarding, SNMP traps
  • Buffers data: Queues logs during network issues
  • Securely uploads: Encrypted transmission to XSIAM cloud

Where in Console

Settings → Data Sources → Broker VM (view status, download OVA)
Settings → Data Sources → Add Data Source → Syslog Collector (configure listeners)

Architecture Placement

[On-Prem Network] [DMZ] [Internet] [XSIAM Cloud] Firewalls ────┐ Servers ──────┼──► Broker VM ──────────────────────────────────► XSIAM Network ──────┘ (10.0.1.50) HTTPS/443 Tenant Devices Syslog TCP/514

Cloud Collectors

Native cloud and SaaS integrations

What It Does

  • Connects to cloud APIs: AWS CloudTrail, Azure Activity, GCP Audit, etc.
  • No on-prem infrastructure: Direct cloud-to-cloud connection
  • Pre-built parsing: Data is automatically normalized

Where in Console

Settings → Data Sources → Cloud Collectors

Available Collectors

AWS:CloudTrail, GuardDuty, Security Hub, VPC Flow Logs
Azure:Activity Log, Entra ID (Azure AD), Defender for Cloud
GCP:Cloud Audit Logs, Security Command Center
SaaS:Okta, Microsoft 365, Salesforce, Workday, Box

Correlation Engine

Detection rule execution

What It Does

  • Runs XQL queries: Continuously executes correlation rules against streaming data
  • Generates alerts: Creates incidents when rule conditions are met
  • Supports multiple rule types: Correlation, BIOC, IOC, Analytics

Where in Console

Rules → Correlation Rules (custom XQL rules)
Rules → BIOC Rules (behavioral indicators)
Rules → Analytics (ML-based detection)
Threat Intel → IOC Rules (indicator matching)

Rule Execution Flow

1. Data Ingested → 2. Parsed & Normalized → 3. Rules Evaluated (every X minutes) → 4. Alert Generated → 5. Incident Created

XSOAR (Built-in)

Security orchestration and automation

What It Does

  • Playbook automation: Auto-enrich, auto-respond, auto-close
  • Integration hub: Connects to 700+ security tools
  • Case management: Track investigations, assign analysts

Where in Console

Automation → Playbooks (view/edit playbooks)
Settings → Integrations (configure 3rd party connections)
Marketplace → Browse (install content packs)

Playbook Trigger Points

Incident Created:Auto-run playbook when new incident appears
Manual:Analyst clicks "Run Playbook" on incident
Scheduled:Run playbook on schedule (e.g., daily IOC import)
API:External system triggers playbook via API

Data Flow: End-to-End

Example: Firewall Log Journey

  1. Generation: Palo Alto NGFW blocks suspicious connection, generates log
  2. Transmission: Firewall sends syslog to Broker VM (10.0.1.50:514)
  3. Receipt: Broker VM receives, queues in local buffer
  4. Upload: Broker VM sends to XSIAM cloud (HTTPS/443)
  5. Parsing: XSIAM applies built-in PANW parser, extracts fields
  6. Normalization: Fields mapped to XDM (src→xdm.source.ipv4)
  7. Storage: Data written to Hot tier (panw_ngfw_traffic_raw dataset)
  8. Detection: Correlation rules check for matches
  9. Alert: Rule triggers, incident created
  10. Automation: XSOAR playbook enriches, notifies analyst
  11. Investigation: Analyst queries data with XQL, investigates
  12. Response: Analyst isolates endpoint, blocks IP at firewall

Interview Questions

XSIAM is a cloud-native platform with five main layers:

  1. Data Collection: XDR agents (endpoints), Broker VMs (on-prem syslog), Cloud Collectors (AWS/Azure/GCP), HTTP Collectors (APIs)
  2. Processing: Parsing engine extracts fields, XDM normalization maps to standard schema, enrichment adds context (TI, geo)
  3. Storage: Hot tier (0-90 days, fast), Warm tier (90-365 days), Cold tier (1-7 years archive)
  4. Detection: Correlation rules (XQL), ML Analytics, IOC matching, BIOC behavioral rules
  5. Response: Incidents, XSOAR playbooks, response actions (isolate, block, remediate)

Key differentiator: Everything integrated in one platform - no separate SIEM, SOAR, EDR, TI platforms to manage.

Complete data flow:

  1. Firewall sends syslog to Broker VM IP (e.g., 10.0.1.50:514 TCP)
  2. Broker VM receives and buffers locally (handles network blips)
  3. Broker VM uploads to XSIAM cloud via HTTPS/443 (compressed, encrypted)
  4. XSIAM cloud receives and routes to parsing engine
  5. Parsing: Built-in parser extracts fields (or custom parsing rule)
  6. XDM normalization: Vendor fields mapped to standard schema
  7. Storage: Data written to appropriate dataset (e.g., panw_ngfw_traffic_raw)
  8. Indexing: Data becomes queryable in XQL

Typical latency: ~30 seconds to 2 minutes from log generation to searchable in XSIAM.
