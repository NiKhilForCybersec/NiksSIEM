Parsing & XDM Normalization

Understanding how XSIAM parses raw logs and maps them to the XDM (XSIAM Data Model) is crucial for effective querying and detection engineering. This guide covers the parsing pipeline, XDM field reference, and custom parser creation.

Data Parsing Pipeline

Understanding how data flows through the parsing pipeline helps troubleshoot issues and optimize detection:

1. Ingestion

Raw logs arrive via API/Syslog/Agent

Stored in raw dataset (vendor_product_raw)
→
2. Classification

XSIAM identifies vendor/product

Uses log format, headers, source IP
→
3. Parsing

Extract fields using parser rules

JSON, regex, or custom patterns
→
4. XDM Mapping

Map extracted fields to XDM schema

src_ip → xdm.source.ipv4
→
5. Storage

Indexed for search & analytics

Both raw + XDM fields available

What is XDM (XSIAM Data Model)?

XDM Explained

XDM is XSIAM's normalization schema - a standard set of field names that all log sources are mapped to. This means you can write ONE query that works across firewalls, endpoints, cloud logs, and any other source.

Why XDM Matters

Without Normalization

Palo Alto: src_ip
Cisco ASA: srcAddress
AWS: sourceIPAddress
Windows: IpAddress

4 different queries needed!

→

With XDM

All sources: xdm.source.ipv4

ONE query works everywhere!

When to Use Raw vs. XDM Fields

Use this decision tree to choose the right field type for your query:

What are you trying to do?
Correlation Rule / Multi-Source Detection
Use XDM Fields xdm.source.ipv4, xdm.event.outcome
Single-Source Investigation
Is the field available in XDM?
Yes → Use XDM (more portable)
No → Use raw field
Vendor-Specific Field (e.g., Palo Alto rule name)
Use Raw Fields rule_name, app_category, threat_name
Debug/Troubleshoot Parsing
Use _raw_log field See original unparsed event

XDM Field Categories

Category Prefix Common Fields Use Case
Source xdm.source.* ipv4, ipv6, host.hostname, port, user.username Where traffic/action originated
Target xdm.target.* ipv4, ipv6, host.hostname, port, user.username Destination of traffic/action
Event xdm.event.* type, outcome, description, original_event_type What happened
Network xdm.network.* protocol, application_protocol, direction Network context
Auth xdm.auth.* auth_method, outcome, privilege_level Authentication events
Observer xdm.observer.* vendor, product, version, name Device that generated the log
Alert xdm.alert.* severity, name, category Security alert context

XDM Field Quick Reference

Source Fields (xdm.source.*)

xdm.source.ipv4 → Source IPv4 address
xdm.source.ipv6 → Source IPv6 address
xdm.source.port → Source port number
xdm.source.host.hostname → Source hostname
xdm.source.user.username → Source username
xdm.source.process.name → Source process name

Target Fields (xdm.target.*)

xdm.target.ipv4 → Destination IPv4 address
xdm.target.port → Destination port number
xdm.target.host.hostname → Destination hostname
xdm.target.user.username → Target username (for auth events)
xdm.target.url → Target URL
xdm.target.file.path → Target file path
xdm.target.file.sha256 → Target file SHA256 hash

Event Fields (xdm.event.*)

xdm.event.type → Event type (NETWORK, AUTH, PROCESS, FILE)
xdm.event.outcome → Outcome (SUCCESS, FAILURE, UNKNOWN)
xdm.event.operation → Operation performed
xdm.event.description → Human-readable description
xdm.event.original_event_type → Original event type from source

Example Field Mappings by Source

Palo Alto NGFW → XDM

// Raw Palo Alto fields → XDM normalized fields src → xdm.source.ipv4 dst → xdm.target.ipv4 sport → xdm.source.port dport → xdm.target.port srcuser → xdm.source.user.username action → xdm.event.outcome (allow=SUCCESS, deny=FAILURE) app → xdm.network.application_protocol rule → xdm.alert.name bytes_sent → xdm.source.sent_bytes bytes_received → xdm.target.sent_bytes

Windows Security Events → XDM

// Windows Event ID 4624 (Successful Logon) → XDM TargetUserName → xdm.target.user.username TargetDomainName → xdm.target.user.domain IpAddress → xdm.source.ipv4 LogonType → xdm.auth.auth_method WorkstationName → xdm.source.host.hostname SubjectUserName → xdm.source.user.username EventID → xdm.event.original_event_type Status → xdm.event.outcome // Logon Type mapping: // 2 = Interactive, 3 = Network, 10 = RemoteInteractive (RDP)

AWS CloudTrail → XDM

// AWS CloudTrail fields → XDM sourceIPAddress → xdm.source.ipv4 userIdentity.userName → xdm.source.user.username eventName → xdm.event.operation eventSource → xdm.observer.name awsRegion → xdm.observer.location errorCode → xdm.event.outcome (null=SUCCESS, else=FAILURE) eventType → xdm.event.type

Using XDM Fields in XQL Queries

Query That Works Across ALL Sources

// Find all failed authentications from any source dataset = xdr_data, panw_ngfw_traffic_raw, microsoft_windows_raw, aws_cloudtrail_raw | filter xdm.event.type = "AUTH" | filter xdm.event.outcome = "FAILURE" | comp failed_attempts = count() by xdm.target.user.username, xdm.source.ipv4 | filter failed_attempts >= 5 | sort desc failed_attempts

This single query detects brute force attacks whether they hit your firewall, Windows servers, or cloud resources!

Network Connections to Specific Port

// Find all connections to port 3389 (RDP) from any source dataset = xdr_data, panw_ngfw_traffic_raw | filter xdm.target.port = 3389 | fields _time, xdm.source.ipv4, xdm.target.ipv4, xdm.source.user.username | sort desc _time | limit 100

Troubleshooting Parsing Issues

When data isn't parsing correctly, use this decision tree:

What's the parsing issue?
XDM fields are empty
Check: Data in _raw_log field?
Yes → Parser not matching (check vendor/product classification)
No → Data not arriving (check ingestion)
Wrong data in XDM fields
Check: Using correct dataset?
Yes → Parser mapping error (review custom parser)
No → Switch to correct dataset name
Partial fields parsed
Log format variation Create custom parser for missing fields

Creating Custom Parsers

Settings → Data Sources → Parsing Rules → + New Parsing Rule

When You Need Custom Parsers

  • Ingesting logs from unsupported vendors
  • Custom application logs
  • Logs with non-standard formats
  • Need to extract additional fields not in default parser

Custom Parser Example (JSON Log)

// Sample raw log (custom application): {"timestamp":"2024-02-15T14:30:00Z","user":"jsmith","action":"login","src_ip":"10.0.1.50","result":"success"} // Parsing rule definition: [PARSER] name = custom_app_auth vendor = CustomCorp product = MyApp format = json [MAPPING] _time = json_extract(raw, "$.timestamp") xdm.source.user.username = json_extract(raw, "$.user") xdm.event.operation = json_extract(raw, "$.action") xdm.source.ipv4 = json_extract(raw, "$.src_ip") xdm.event.outcome = if(json_extract(raw, "$.result") = "success", "SUCCESS", "FAILURE")

Custom Parser Example (Syslog/Regex)

// Sample raw log (legacy syslog): Feb 15 14:30:00 server01 myapp: USER=jsmith ACTION=login SRC=10.0.1.50 STATUS=OK // Parsing rule with regex: [PARSER] name = legacy_app_syslog vendor = LegacyCorp product = LegacyApp format = syslog [EXTRACT] pattern = USER=(?P<user>\S+)\s+ACTION=(?P<action>\S+)\s+SRC=(?P<src_ip>\S+)\s+STATUS=(?P<status>\S+) [MAPPING] xdm.source.user.username = user xdm.event.operation = action xdm.source.ipv4 = src_ip xdm.event.outcome = if(status = "OK", "SUCCESS", "FAILURE")

Interview Questions

XDM - XSIAM Data Model - is the normalization schema that XSIAM uses to map all incoming log sources to a standard set of field names. It's essentially a common language that lets you write detection rules that work across completely different data sources.

The practical impact is huge. Without normalization, if I wanted to detect brute force attacks, I'd need separate rules for Palo Alto (which uses "src" for source IP), Windows (which uses "IpAddress"), AWS CloudTrail (which uses "sourceIPAddress"), and every other source. That's a maintenance nightmare - you're duplicating logic everywhere and hoping you don't miss a source.

With XDM, I write ONE rule using xdm.source.ipv4 for source IP and xdm.event.outcome for success/failure, and it automatically applies to any log source that's been properly mapped. When I onboard a new data source - say, a new cloud provider - my existing detection rules immediately start working on that data without any changes.

The key XDM categories I use constantly are: xdm.source.* for where something came from, xdm.target.* for where it's going, and xdm.event.* for what happened. Those three cover 80% of detection use cases. The mapping happens automatically during ingestion through XSIAM's parsers, or you can create custom parsers for unsupported sources.

My general rule is: use XDM fields for anything that might need to work across multiple sources, and use raw fields only when you need vendor-specific information that doesn't have an XDM equivalent.

For correlation rules and detections, I always use XDM. If I'm building a rule to detect lateral movement, I'll use xdm.source.ipv4 and xdm.target.port rather than the vendor-specific equivalents. That way, whether the traffic is logged by the firewall, the endpoint agent, or a network tap, my rule catches it.

I use raw fields for three scenarios. First, when I need vendor-specific context that doesn't map to XDM - like Palo Alto's "app" field that tells you the specific application identified, or Windows event IDs for specific event types. Second, during initial investigation when I'm exploring a single data source and want to see all available fields. Third, when troubleshooting parsing issues - I'll look at _raw_log to see the original event format.

One gotcha I've learned: some fields might exist in both raw and XDM but with different values due to transformation. For example, xdm.event.outcome might normalize "allow"/"deny"/"drop"/"pass" all to "SUCCESS"/"FAILURE". If I need the original granularity, I'd use the raw field. But for detection purposes, the normalized version is usually what I want.

Let me walk through a concrete example - detecting brute force authentication attacks across firewall, Windows, and cloud logs. The key is designing the rule around XDM fields from the start.

First, I'd identify which XDM fields capture the relevant information across all sources. For brute force, I need: xdm.event.type = "AUTH" to filter to authentication events, xdm.event.outcome = "FAILURE" for failed attempts, xdm.source.ipv4 for the attacking IP, and xdm.target.user.username for the targeted account.

My XQL query would look like: dataset = panw_ngfw_traffic_raw, microsoft_windows_raw, aws_cloudtrail_raw, then filter on those XDM fields, aggregate by source IP and username, and threshold at 5+ failures. The beauty is I can add new datasets to that query trivially - if we onboard Azure AD logs next month, I just add azure_ad_raw to the dataset list.

Before deploying, I'd validate that each source actually populates those XDM fields. Sometimes a parser doesn't map everything, so I'd run a quick query on each dataset to confirm the fields have data. If a source doesn't populate xdm.event.type properly, I might need to use a different filter condition or create a custom parser enhancement.

One nuance: different sources might have different thresholds for what's "normal." A firewall might see thousands of auth events while Windows sees hundreds. I might need to tune the threshold per source or normalize by creating a baseline metric first.

This is a common issue I've dealt with multiple times. The systematic approach is to trace the data through the parsing pipeline.

First, I'd confirm the raw data is actually arriving by querying with minimal filters and looking at the _raw_log field. If I see the raw events but XDM fields are null, that confirms the issue is in parsing, not ingestion. If _raw_log is also empty, the problem is upstream - maybe the data source configuration or Broker VM connectivity.

Assuming raw data is there, next I'd check how XSIAM classified the logs. Go to Settings → Data Sources and find the data source in question. Look at the assigned parser - is it the correct vendor/product? Sometimes logs get misclassified, especially if they're coming through syslog without proper identification. I've seen Cisco logs get classified as "generic syslog" and miss the Cisco parser entirely.

If classification is correct, I'd look at the parser rules themselves. Maybe the log format changed - vendors update their log schemas, and sometimes those changes break existing parsers. I'd grab a sample _raw_log event and compare it against what the parser expects. Common issues are: date format changes, field order changes, or new fields appearing that confuse the regex.

If the default parser just doesn't work for this log variation, I have two options: create a custom parser that handles the specific format, or work with Palo Alto support to get the default parser updated. For a quick fix, custom parser. For long-term sustainability, push for an official update.

I've built custom parsers for several internal applications and niche vendors. The process I follow starts with understanding the log format deeply before writing any parsing rules.

First, I collect 50-100 sample events that represent the full variety of logs this source generates. I'm looking for different event types, error conditions, edge cases. A parser that only works for successful logins but breaks on failed logins isn't useful. I'll save these samples in a test file for validation.

Next, I identify the log format - is it JSON, CSV, syslog, or some custom format? JSON is easiest because I can use json_extract functions. For unstructured formats, I need regex, which requires more careful testing. I usually prototype my regex patterns at regex101.com first because it gives immediate feedback and explains what each part matches.

For the XDM mapping, I think about what security value this data provides. If it's authentication logs, I need xdm.source.ipv4, xdm.target.user.username, xdm.event.outcome at minimum. If it's network logs, xdm.source.ipv4, xdm.target.ipv4, xdm.target.port. I always map _time properly because timestamps are critical for investigation timelines.

After deploying the parser, I validate by running queries that filter on the XDM fields and comparing against the raw data. I spot-check 10-20 events manually to make sure values are mapping correctly. Then I run it for a few days and check for parsing failures - usually there's at least one log variation I didn't anticipate that breaks things.

I've worked with all three, and they're solving the same fundamental problem but with different approaches and trade-offs.

XDM is tightly integrated with XSIAM's parsing and correlation engine. The normalization happens during ingestion automatically, so by the time data is queryable, it's already mapped. This is great for detection engineering because you don't need to think about it - just use xdm.* fields and they work. The downside is you're dependent on Palo Alto's parsers, and if they don't support a source, you need custom work.

Splunk CIM (Common Information Model) is more of a convention than enforced structure. You can use CIM-compliant apps and add-ons, but nothing stops you from ignoring CIM entirely. This flexibility is powerful - you can ingest anything - but it means detection rules need more careful design to handle unmapped data. I've seen Splunk environments where half the data follows CIM and half doesn't, making correlation painful.

Elastic's ECS (Elastic Common Schema) is similar to XDM in philosophy - a standardized schema that data should conform to. The Elastic stack has good tooling (Beats, Logstash) that can normalize on ingest. ECS is very detailed with lots of field options, which can be both powerful and overwhelming.

The biggest practical difference is vendor support. XDM has excellent coverage for Palo Alto products and solid coverage for major third parties. If you're a Palo Alto shop with NGFW, Cortex XDR, and Prisma Cloud, everything just works together through XDM. If you're integrating lots of diverse sources, you'll need more custom parsing work than you might with Splunk's extensive app ecosystem.
