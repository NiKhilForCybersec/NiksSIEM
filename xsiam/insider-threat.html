Insider Threat Investigation

Detection and investigation procedures for insider threats including data exfiltration, unauthorized access, and malicious employee activity using XSIAM behavioral analytics and correlation.

Legal & HR Coordination Required

Insider threat investigations require coordination with HR and Legal before taking action. Ensure proper authorization, document chain of custody, and follow company policy. Privacy laws may restrict monitoring in some jurisdictions.

Insider Threat Categories

Threat Type Motivation Indicators Risk Level
Malicious Insider Financial gain, revenge, ideology Data hoarding, off-hours access, resignation notice Critical
Departing Employee Taking data to new job Mass downloads, USB usage, cloud uploads High
Negligent Insider Carelessness, lack of training Policy violations, accidental sharing Medium
Compromised Insider Account takeover by external actor Unusual patterns, impossible travel Critical
Privilege Abuse Curiosity, unauthorized snooping Accessing data outside job function High

Behavioral Analysis

UEBA Behavioral Queries

# Behavioral Baseline Comparison # ═══════════════════════════════════════════════════════════════ # User Activity Volume vs Peer Group dataset = xdr_data | filter _time > now() - 30d | filter user_department = "[SUBJECT_DEPARTMENT]" | comp count() as total_events, sum(if(event_type = "FILE_READ", 1, 0)) as file_reads, sum(if(event_type = "FILE_WRITE", 1, 0)) as file_writes, sum(action_file_size) as total_bytes by user_name | alter z_score_events = (total_events - avg(total_events)) / stddev(total_events) | alter z_score_bytes = (total_bytes - avg(total_bytes)) / stddev(total_bytes) | filter user_name = "[SUBJECT_USER]" # ═══════════════════════════════════════════════════════════════ # Activity Timeline (Last 90 Days) dataset = xdr_data | filter _time > now() - 90d | filter user_name = "[SUBJECT_USER]" | alter week = format_time("%Y-W%V", _time) | comp count() as weekly_events, count_distinct(action_file_path) as unique_files, sum(action_file_size) as bytes_accessed by week | sort week asc # ═══════════════════════════════════════════════════════════════ # First-Time Access to Sensitive Systems dataset = xdr_data | filter _time > now() - 90d | filter user_name = "[SUBJECT_USER]" | filter event_type in ("FILE_READ", "FILE_OPEN", "APPLICATION_ACCESS") | comp min(_time) as first_access by action_file_path, application_name | filter first_access > now() - 14d // First access in last 2 weeks | sort first_access desc # ═══════════════════════════════════════════════════════════════ # Risk Score Trend dataset = xdr_ueba_raw | filter _time > now() - 90d | filter entity_id = "[SUBJECT_USER]" | fields _time, risk_score, anomaly_type, contributing_factors | sort _time asc

Interview Questions

Departing employees represent elevated risk because they have legitimate access but changing loyalty. My approach balances security with avoiding assumptions of malice.

First, I establish a baseline: I query the past 90 days of activity to understand what's normal for this user - what files they access, what times they work, typical data volumes. Without this baseline, I can't identify anomalies.

Enhanced monitoring setup: I lower alert thresholds for this user specifically. Normal might be flagging >100 downloads/day, but for a departing employee, I want to see >20. I enable alerts for USB usage (any amount), cloud storage uploads to non-corporate destinations, email attachments to personal addresses, and archive file creation.

DLP policy adjustment: If we have DLP, I ensure policies are enforcing (not just alerting) for this user on sensitive data categories. This provides a safety net while investigation continues.

Daily review during notice period: Each day, I run queries comparing their activity to baseline. I'm looking for sudden increases in file access, new systems accessed for the first time, or data staging behaviors like copying files to Desktop or Downloads.

Coordination with HR: I provide daily or every-other-day summaries to HR. If I see concerning activity, they can accelerate the departure or adjust the person's access. On the last day, I coordinate timing so access is revoked exactly when they depart.

The key is proactive monitoring without assuming guilt. Most departing employees are honest - I'm just ensuring we'd catch the minority who aren't.

This is a critical distinction because the response differs significantly. A malicious insider requires HR/legal involvement, while a compromised account is an external attack requiring incident response.

Authentication patterns: Compromised accounts typically show impossible travel, logins from new devices/locations, or access patterns that don't match the user's normal behavior. I check Azure AD/Okta logs - did they log in from Nigeria when they've never been there? That's compromise, not insider threat.

Timing correlation: Does the suspicious activity align with when the user was physically present and logged into their normal workstation? If data exfiltration happened at 3 AM from an IP in another country, but the user badged into the office at 9 AM, it's likely compromise. If it happened from their work laptop during work hours, it's more likely intentional.

Technical sophistication: Malicious insiders using their legitimate access usually don't need to bypass security controls - they just access what they're allowed to access and move it out. Compromised accounts often show additional suspicious behaviors like attempting to access systems beyond the user's normal scope, running enumeration queries, or trying to escalate privileges.

User verification: For ambiguous cases, I work with HR to have the manager casually verify the user's activities. "Hey, were you working late on Project X last night?" If the user has no idea what you're talking about, it's likely compromise.

When in doubt: Treat it as compromise first - reset credentials and revoke sessions. This stops the activity regardless of cause. Then investigate to determine if it was actually the legitimate user. You can always escalate to HR later if evidence points to the actual employee.
