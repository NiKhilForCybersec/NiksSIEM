Incident Properties

INCIDENT PROPERTIES: SEVERITY LEVELS: ├── Informational: Low-risk activity, awareness only ├── Low: Minimal impact, routine investigation ├── Medium: Moderate impact, timely investigation └── High: Critical threat, immediate response required STATUS VALUES: ├── Active: New or under investigation ├── Resolved: Investigation complete, actions taken └── Redirected: Sent to another team/system CLASSIFICATION (upon closure): ├── True Positive: Confirmed malicious activity │ ├── Security test │ ├── Line-of-business application │ ├── Confirmed malicious │ └── Unwanted software ├── False Positive: Legitimate activity incorrectly flagged │ ├── Not malicious │ ├── Not enough data │ └── Other └── Informational: Expected activity DETERMINATION: ├── Multistage attack ├── Malware ├── Phishing ├── Unwanted software ├── Security testing └── Line of business

Phase 1: Triage

TRIAGE WORKFLOW: STEP 1: REVIEW INCIDENT QUEUE Path: security.microsoft.com → Incidents & alerts → Incidents Priority Order: ├── High severity → Immediate ├── Active incidents → Next ├── New incidents → Following └── Low severity → Batch processing STEP 2: INITIAL ASSESSMENT (2-5 minutes) ├── Read incident title and description ├── Check severity and affected assets count ├── Review alert types in incident ├── Check if user/host is VIP or critical └── Look for known false positive patterns STEP 3: TRIAGE DECISION TREE ┌─────────────────────────────────────────────┐ │ Is this a known false positive pattern? │ ├───────────────┬─────────────────────────────┤ │ YES │ NO │ │ │ │ │ │ │ ▼ │ ▼ │ │ Close as FP │ Is immediate action │ │ (Document) │ required? │ │ │ ├── YES → Contain now │ │ │ └── NO → Investigate │ └───────────────┴─────────────────────────────┘ STEP 4: ASSIGN AND ACKNOWLEDGE ├── Assign to self or appropriate analyst ├── Change status to "Active" ├── Add initial notes └── Start investigation timer

Phase 2: Investigation

INVESTIGATION WORKFLOW: STEP 1: REVIEW INCIDENT DETAILS ├── Incident summary ├── All associated alerts ├── Affected devices ├── Affected users ├── Evidence (files, IPs, domains) └── Investigation timeline STEP 2: USE INVESTIGATION TOOLS ┌────────────────────────────────────────────────────────────┐ │ INVESTIGATION VIEW │ │ │ │ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ │ │ Attack │ │ Device │ │ Alert │ │ │ │ Story │ │ Timeline │ │ Timeline │ │ │ │ (Graph) │ │ │ │ │ │ │ └──────────────┘ └──────────────┘ └──────────────┘ │ │ │ │ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ │ │ Evidence │ │ Assets │ │ Advanced │ │ │ │ (Files,IPs) │ │ (Users, │ │ Hunting │ │ │ │ │ │ Devices) │ │ │ │ │ └──────────────┘ └──────────────┘ └──────────────┘ │ │ │ └────────────────────────────────────────────────────────────┘ STEP 3: ANALYZE ATTACK STORY ├── View visual attack graph ├── Understand attack chain ├── Identify initial access point ├── Track lateral movement └── Find all affected entities STEP 4: SCOPE THE INCIDENT // Advanced Hunting - Find related activity let IncidentDevices = datatable(DeviceName:string)["DEVICE01", "DEVICE02"]; let IncidentUsers = datatable(AccountName:string)["user1", "user2"]; DeviceProcessEvents | where Timestamp > ago(7d) | where DeviceName in (IncidentDevices) or InitiatingProcessAccountName in (IncidentUsers) | project Timestamp, DeviceName, FileName, ProcessCommandLine | order by Timestamp STEP 5: GATHER EVIDENCE ├── Collect file hashes ├── Note malicious IPs/domains ├── Screenshot key findings ├── Export relevant hunting results └── Document investigation steps

Phase 3-4: Containment & Remediation

CONTAINMENT ACTIONS: IMMEDIATE CONTAINMENT: ├── Isolate device │ Path: Device page → Actions → Isolate device │ └── Allows MDE communication only ├── Restrict app execution │ Path: Device page → Actions → Restrict app execution │ └── Only Microsoft-signed apps can run ├── Block file │ Path: Evidence → File → Block │ └── Prevents execution across tenant └── Block indicator Path: Settings → Indicators → Add indicator └── Block IP, domain, or file hash INVESTIGATION PACKAGE: ├── Collect investigation package │ Path: Device page → Actions → Collect investigation package │ └── Gathers logs, memory, autoruns, etc. ├── Download and analyze └── Extract additional IOCs REMEDIATION ACTIONS: ├── Run antivirus scan │ Path: Device page → Actions → Run antivirus scan │ └── Full or quick scan options ├── Stop and quarantine file │ Path: Device → File → Stop and quarantine │ └── Terminates process and quarantines ├── Remove persistence │ └── Manual cleanup or automated via IR tools └── Restore from backup if needed REMEDIATION DECISION TREE: ┌───────────────────────────────────────────────────────────┐ │ Can the threat be removed without rebuilding? │ ├─────────────────┬─────────────────────────────────────────┤ │ YES │ NO │ │ │ │ │ │ │ ▼ │ ▼ │ │ Remove malware │ Is device critical? │ │ Clean registry │ ├── YES → Rebuild priority │ │ Restore files │ └── NO → Schedule rebuild │ └─────────────────┴─────────────────────────────────────────┘

Phase 5: Closure

CLOSURE WORKFLOW: STEP 1: VALIDATE REMEDIATION ├── Confirm malware removed ├── Verify no persistence remains ├── Check for reinfection ├── Validate system functionality └── Confirm user can work normally STEP 2: CLASSIFY INCIDENT Classification Options: ├── True Positive │ ├── Determination: (Malware, Phishing, etc.) │ └── Used for: Confirmed threats ├── False Positive │ ├── Determination: (Not malicious, Insufficient data) │ └── Used for: Legitimate activity flagged └── Informational └── Used for: Expected/authorized activity STEP 3: DOCUMENT FINDINGS Incident Comments Should Include: ├── Summary of what happened ├── Root cause (if determined) ├── Scope of impact ├── Actions taken ├── IOCs discovered ├── Lessons learned └── Recommended follow-ups STEP 4: POST-INCIDENT ACTIONS ├── Update detection rules if needed ├── Add tuning/suppression for FPs ├── Share IOCs with threat intel ├── Update runbooks if applicable ├── Schedule follow-up if needed └── Close incident as Resolved STEP 5: METRICS TRACKING Track: ├── Time to detect (TTD) ├── Time to triage (TTT) ├── Time to contain (TTC) ├── Time to remediate (TTR) └── False positive rate

Investigation Queries

// Timeline of all activity on affected device let TargetDevice = "WORKSTATION01"; let TimeWindow = 24h; union DeviceProcessEvents, DeviceNetworkEvents, DeviceFileEvents, DeviceLogonEvents | where Timestamp > ago(TimeWindow) | where DeviceName =~ TargetDevice | project Timestamp, ActionType, FileName, ProcessCommandLine, RemoteIP, LocalIP, AccountName | order by Timestamp // Find all devices with same malware let MaliciousHash = "abc123..."; DeviceFileEvents | where SHA256 == MaliciousHash | summarize Devices = make_set(DeviceName), FirstSeen = min(Timestamp), LastSeen = max(Timestamp) // Check for lateral movement from compromised device let CompromisedDevice = "WORKSTATION01"; DeviceNetworkEvents | where Timestamp > ago(7d) | where DeviceName =~ CompromisedDevice | where RemoteIPType == "Private" | where RemotePort in (445, 135, 3389, 5985, 5986) | summarize Connections = count() by RemoteIP, RemotePort | order by Connections desc // User activity across all devices let SuspiciousUser = "compromised_user"; DeviceLogonEvents | where Timestamp > ago(7d) | where AccountName =~ SuspiciousUser | summarize Devices = make_set(DeviceName), LogonCount = count() by LogonType

Interview Questions

  1. Triage: Review queue, assess severity, assign owner
  2. Investigate: Analyze attack story, scope impact, gather evidence
  3. Contain: Isolate device, block IOCs, restrict execution
  4. Remediate: Remove malware, clean persistence, restore
  5. Close: Classify, document, update detections

Alert: Single detection event from MDE.

Incident: Correlated collection of related alerts grouped by device, user, time, or attack chain. Provides complete attack context.

  • Isolate device: Network isolation (MDE communication only)
  • Restrict app execution: Only Microsoft-signed apps
  • Block file: Prevent execution tenant-wide
  • Block indicator: Block IP/domain/hash via indicators
  • True Positive: Confirmed malicious - malware, phishing, attack
  • False Positive: Legitimate activity incorrectly flagged
  • Informational: Expected activity, security testing

Proper classification improves detection tuning and metrics accuracy.
