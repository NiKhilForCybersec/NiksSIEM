SIEM Platform Comparison

Comprehensive comparison of Palo Alto XSIAM, Microsoft Sentinel, and Splunk Enterprise Security. This guide covers architecture, data collection methods, query languages, detection capabilities, automation, pricing, and real-world deployment scenarios.

Sentinel Data Collection Rule (DCR) Example
// DCR for Windows Security Events - Filter at ingestion
{
  "properties": {
    "dataSources": {
      "windowsEventLogs": [
        {
          "name": "securityEvents",
          "streams": ["Microsoft-SecurityEvent"],
          "xPathQueries": [
            "Security!*[System[(EventID=4624 or EventID=4625 or EventID=4648 or EventID=4672 or EventID=4688 or EventID=4689)]]",
            "Security!*[System[(EventID>=4720 and EventID<=4735)]]"
          ]
        }
      ]
    },
    "destinations": {
      "logAnalytics": [
        {
          "workspaceResourceId": "/subscriptions/.../workspaces/sentinel-workspace",
          "name": "sentinelWorkspace"
        }
      ]
    },
    "dataFlows": [
      {
        "streams": ["Microsoft-SecurityEvent"],
        "destinations": ["sentinelWorkspace"],
        "transformKql": "source | where EventID in (4624, 4625, 4672, 4688) | extend ParsedAccount = extract('Account Name:\\s+(.+)', 1, EventData)"
      }
    ]
  }
}

Contains | filter cmd contains "powershell" | where cmd has "powershell" cmd="*powershell*"
Regex | filter user ~= "admin[0-9]+" | where user matches regex "admin[0-9]+" | regex user="admin[0-9]+"
Count | comp count() as cnt | summarize cnt = count() | stats count as cnt
Group By | comp count() by user | summarize count() by user | stats count by user
Distinct Count | comp count_distinct(user) | summarize dcount(user) | stats dc(user)
Sort | sort desc count | sort by count desc | sort - count
Limit | limit 10 | take 10 | head 10

Side-by-Side Query Examples

Example 1: Brute Force Detection

XQL (XSIAM)

dataset = xdr_data
| filter xdm.event.type = "AUTHENTICATION"
| filter xdm.auth.outcome = "FAILURE"
| filter _time > timestamp_sub(now(), "1h")
| comp count() as failed_count,
       count_distinct(xdm.auth.user) as unique_users
  by xdm.source.ipv4
| filter failed_count > 10
| sort desc failed_count

KQL (Sentinel)

SigninLogs
| where TimeGenerated > ago(1h)
| where ResultType != "0"
| summarize 
    failed_count = count(),
    unique_users = dcount(UserPrincipalName)
  by IPAddress
| where failed_count > 10
| sort by failed_count desc

SPL (Splunk)

index=security sourcetype=WinEventLog:Security
  EventCode=4625 earliest=-1h
| stats count as failed_count,
        dc(user) as unique_users
  by src_ip
| where failed_count > 10
| sort - failed_count

Example 2: PowerShell Encoded Command Detection

XQL (XSIAM)

dataset = xdr_data
| filter xdm.target.process.name contains "powershell"
| filter xdm.target.process.command_line contains "-enc" 
      or xdm.target.process.command_line contains "encodedcommand"
      or xdm.target.process.command_line contains "FromBase64"
| fields _time, xdm.source.host.hostname, 
         xdm.source.user.username,
         xdm.target.process.command_line
| sort desc _time

KQL (Sentinel)

SecurityEvent
| where EventID == 4688
| where NewProcessName has "powershell"
| where CommandLine has_any 
    ("-enc", "-encodedcommand", "FromBase64")
| project TimeGenerated, Computer, 
          Account, CommandLine
| sort by TimeGenerated desc

SPL (Splunk)

index=sysmon EventCode=1 
  Image="*powershell*"
  (CommandLine="*-enc*" OR 
   CommandLine="*encodedcommand*" OR
   CommandLine="*FromBase64*")
| table _time, Computer, User, 
        CommandLine
| sort - _time

Example 3: Data Exfiltration Detection

XQL (XSIAM)

dataset = panw_ngfw_traffic_raw
| filter _time > timestamp_sub(now(), "24h")
| filter action = "allow"
| comp sum(bytes_sent) as total_bytes
  by src_ip, src_user
| alter total_gb = divide(total_bytes, 1073741824)
| filter total_gb > 5
| sort desc total_gb

KQL (Sentinel)

CommonSecurityLog
| where TimeGenerated > ago(24h)
| where DeviceAction == "allow"
| summarize total_bytes = sum(toint(SentBytes))
  by SourceIP, SourceUserName
| extend total_gb = round(total_bytes / 1073741824.0, 2)
| where total_gb > 5
| sort by total_gb desc

SPL (Splunk)

index=firewall action=allow earliest=-24h
| stats sum(bytes_out) as total_bytes
  by src_ip, user
| eval total_gb = round(total_bytes/1073741824, 2)
| where total_gb > 5
| sort - total_gb

Splunk SOAR

# Splunk SOAR Python Playbook
def ip_enrichment_playbook(container):
    # Get IP from artifact
    ip = phantom.get_artifact_field(
        container, "sourceAddress")
    
    # Enrich with VirusTotal
    phantom.act("ip reputation", 
        parameters=[{"ip": ip}],
        assets=["virustotal"],
        callback=check_reputation)

def check_reputation(action, results):
    if results[0]["summary"]["malicious"] > 5:
        # Block IP
        phantom.act("block ip",
            parameters=[{"ip": ip}],
            assets=["palo_alto"])
        
        # Notify team
        phantom.act("send message",
            parameters=[{"message": f"Blocked {ip}"}],
            assets=["slack"])

Detection Rule Conversion Examples

Brute Force Detection - Platform Conversion

XSIAM XQL Version
// XSIAM Correlation Rule: Brute Force Detection
dataset = xdr_data
| filter xdm.event.type = "AUTHENTICATION"
| filter xdm.auth.outcome = "FAILURE"
| filter _time > timestamp_sub(now(), "5m")
| comp 
    failed_count = count(),
    unique_accounts = count_distinct(xdm.auth.user),
    target_accounts = arrayconcat(xdm.auth.user),
    first_seen = min(_time),
    last_seen = max(_time)
  by xdm.source.ipv4
| filter failed_count > 10
| alter attack_type = if(unique_accounts > 3, "Password Spray", "Brute Force")
| alter severity = if(failed_count > 50, "HIGH", "MEDIUM")
Sentinel KQL Version
// Sentinel Analytics Rule: Brute Force Detection
let threshold = 10;
let timeWindow = 5m;
SigninLogs
| where TimeGenerated > ago(timeWindow)
| where ResultType != "0"
| summarize 
    failed_count = count(),
    unique_accounts = dcount(UserPrincipalName),
    target_accounts = make_set(UserPrincipalName),
    first_seen = min(TimeGenerated),
    last_seen = max(TimeGenerated)
  by IPAddress
| where failed_count > threshold
| extend attack_type = iff(unique_accounts > 3, "Password Spray", "Brute Force")
| extend severity = iff(failed_count > 50, "High", "Medium")
Splunk SPL Version
| Splunk Correlation Search: Brute Force Detection
index=security sourcetype="WinEventLog:Security" EventCode=4625 earliest=-5m
| stats 
    count as failed_count,
    dc(user) as unique_accounts,
    values(user) as target_accounts,
    earliest(_time) as first_seen,
    latest(_time) as last_seen
  by src_ip
| where failed_count > 10
| eval attack_type = if(unique_accounts > 3, "Password Spray", "Brute Force")
| eval severity = if(failed_count > 50, "high", "medium")

Interview Questions

Choose XSIAM when:

  • Already have Palo Alto infrastructure (NGFW, Prisma)
  • Want unified SIEM + XDR + SOAR in one platform
  • Need strong native endpoint protection
  • Prefer fully managed cloud solution with no infrastructure

Choose Sentinel when:

  • Microsoft-centric environment (M365, Azure AD, MDE)
  • Want to leverage existing M365 E5 investment
  • Need cost-effective consumption-based pricing
  • Azure cloud-first strategy

Choose Splunk when:

  • Complex multi-vendor environment
  • Need on-premises or air-gapped deployment
  • Require maximum customization and flexibility
  • Have mature SPL expertise in-house
  • Need platform for security AND IT operations

XSIAM: Cloud-native, single-tenant SaaS. Everything runs in Palo Alto's cloud. Combines SIEM + XDR + SOAR natively. Data flows from agents/Broker VMs Cloud processing Data Lake Analytics.

Sentinel: Built on Azure Log Analytics. Cloud-only but within your Azure tenant. Modular - you add capabilities through connectors and solutions. Relies on Azure Monitor Agent (AMA) for collection.

Splunk: Distributed architecture with separate tiers (forwarders indexers search heads). Can be deployed on-prem, cloud, or hybrid. Most flexible but also most complex to manage. ES is an app layer on top of core Splunk.

XQL (XSIAM):

  • Pipe-based syntax similar to KQL
  • Uses filter instead of where
  • Uses comp for aggregation (like summarize)
  • Newest, least documentation/community

KQL (Sentinel):

  • Pipe-based, reads left-to-right
  • Case-sensitive operators
  • Strong time-series functions
  • Also used in MDE Advanced Hunting

SPL (Splunk):

  • Oldest, most mature
  • Most flexible and powerful
  • Largest community and documentation
  • Can be complex for advanced queries

XSIAM:

  • Per-endpoint pricing + data volume
  • Includes XDR, SOAR, ASM, TI
  • Typically premium pricing

Sentinel:

  • Per GB ingested (pay-as-you-go or commitment)
  • Some Microsoft data is free
  • Additional costs: retention, Logic Apps, Defender licenses

Splunk:

  • Per GB indexed per day (workload pricing available)
  • ES is additional license
  • SOAR, UBA are separate products
  • Typically highest total cost

XSIAM:

  • Cortex XDR Agent: Endpoint telemetry (full EDR)
  • Broker VM: On-prem syslog, WEF, CEF collection
  • Cloud Collectors: API-based SaaS integration
  • HTTP Collector: Custom webhooks

Sentinel:

  • Azure Monitor Agent (AMA): Windows/Linux logs
  • Data Connectors: Microsoft and 3rd party
  • DCRs: Data Collection Rules for transformation
  • HTTP Data Collector API: Custom logs

Splunk:

  • Universal Forwarder: Lightweight agent
  • Heavy Forwarder: On-prem parsing and routing
  • HTTP Event Collector: API ingestion
  • Technology Add-ons: Vendor-specific collection

High-level migration approach:

  1. Assessment: Inventory all data sources, detection rules, dashboards, and integrations
  2. Azure Setup: Create Log Analytics workspace, enable Sentinel, configure retention
  3. Data Onboarding: Start with Microsoft data (easiest), then add 3rd party via AMA and connectors
  4. Rule Conversion: Convert SPL correlation searches to KQL analytics rules
  5. Automation: Rebuild Splunk SOAR playbooks as Logic Apps
  6. Parallel Run: Run both SIEMs simultaneously, validate detection parity
  7. Cutover: Disable Splunk alerts, enable Sentinel, decommission

Key challenges:

  • SPL to KQL translation (different syntax)
  • CIM to ASIM mapping differences
  • Splunk props.conf parsing to DCR transforms
