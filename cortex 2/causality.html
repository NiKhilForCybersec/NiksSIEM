Understanding Causality

Causality Analysis automatically reconstructs the complete attack chain by linking related events across time and showing the parent-child relationships between processes, files, and network connections.

Causality Chain Elements

Element Description Investigation Value
Causality Group Owner (CGO) The root process that initiated the activity chain Identifies initial access point
Actor Process Parent process that spawned the current action Shows execution context
Action Process The process performing the suspicious activity Identifies the threat tool
Target Object Files, registry keys, network connections affected Shows impact and IOCs
Causality ID Unique identifier linking all related events Query for complete chain

XQL Queries for Causality Analysis

Find Complete Attack Chain

// Get all events in a causality chain
dataset = xdr_data 
| filter causality_id = "abc123def456"
| sort asc _time
| fields _time, 
         event_type,
         causality_actor_process_image_name,
         action_process_image_name,
         action_process_command_line,
         action_file_path,
         action_remote_ip
| limit 500

Find Root Cause (CGO)

// Identify the Causality Group Owner
dataset = xdr_data 
| filter action_process_causality_id = "abc123def456"
| filter causality_actor_causality_id = null  // CGO has no parent causality
| fields agent_hostname,
         action_process_image_name,
         action_process_image_path,
         action_process_command_line,
         action_process_username,
         _time
| dedup action_process_image_name

Trace Process Lineage

// Follow process tree from suspicious child to root
dataset = xdr_data 
| filter event_type = ENUM.EVENT_TYPE_PROCESS
| filter action_process_image_name = "mimikatz.exe"
| fields agent_hostname,
         // Direct parent
         actor_process_image_name as parent,
         actor_process_command_line as parent_cmd,
         // Grandparent (if available)
         causality_actor_process_image_name as grandparent,
         // The suspicious process
         action_process_image_name as child,
         action_process_command_line as child_cmd,
         action_process_causality_id
| limit 100

Map Attack Timeline

// Create timeline of attack phases
dataset = xdr_data 
| filter causality_id = "abc123def456"
| alter phase = case(
    event_type = ENUM.EVENT_TYPE_PROCESS and 
    action_process_image_name in ("outlook.exe", "chrome.exe"), "Initial Access",
    event_type = ENUM.EVENT_TYPE_PROCESS and 
    actor_process_image_name contains "Office", "Execution",
    event_type = ENUM.EVENT_TYPE_FILE_CREATE and
    action_file_name contains ".exe", "Payload Delivery",
    event_type = ENUM.EVENT_TYPE_REGISTRY_SET_VALUE and
    action_registry_key_name contains "Run", "Persistence",
    event_type = ENUM.EVENT_TYPE_OPEN_REMOTE_PROCESS_HANDLE and
    target_process_name = "lsass.exe", "Credential Access",
    event_type = ENUM.EVENT_TYPE_NETWORK and
    action_remote_port in (4444, 8080, 443), "C2 Communication",
    default, "Other"
)
| fields _time, phase, event_type, action_process_image_name, action_file_name, action_remote_ip
| sort asc _time

Living Off The Land (LOLBins)

// Detect abuse of legitimate Windows binaries
dataset = xdr_data 
| filter event_type = ENUM.EVENT_TYPE_PROCESS
| filter action_process_image_name in (
    "certutil.exe", "bitsadmin.exe", "mshta.exe",
    "regsvr32.exe", "rundll32.exe", "msiexec.exe",
    "wmic.exe", "cmstp.exe", "msconfig.exe"
)
| filter action_process_command_line contains "http" or
         action_process_command_line contains "-decode" or
         action_process_command_line contains "/i:" or
         action_process_command_line contains "scrobj"
| fields agent_hostname,
         action_process_image_name as lolbin,
         action_process_command_line,
         actor_process_image_name as parent,
         action_process_causality_id,
         _time
| limit 100

Lateral Movement Chain

// Track lateral movement between systems
dataset = xdr_data 
| filter event_type = ENUM.EVENT_TYPE_PROCESS

Q: How do you use Cortex XDR's causality analysis to investigate an alert?

A: When I receive an alert, I start by opening the Causality View which shows a graphical representation of the entire attack chain. I identify the Causality Group Owner (CGO) which tells me how the attack started, whether it was a phishing email, drive-by download, or compromised service. I then trace the process tree from the initial access through each phase - execution, persistence, credential access, lateral movement, etc. Each node in the chain gives me IOCs like file hashes, command lines, and network connections. I use XQL queries with the causality_id to pull all related events and build a complete timeline. This helps me understand not just what happened, but how the attacker operated and where I need to focus remediation efforts.

Q: What's the difference between the actor process and the action process?

A: The actor process is the parent that initiated or spawned the action. The action process is the one actually performing the suspicious activity. For example, if Word opens and runs a macro that launches PowerShell, Word is the actor process and PowerShell is the action process. Understanding this relationship is critical for root cause analysis. If I see PowerShell doing something malicious, I need to know what launched it to understand the full attack chain. In Cortex XDR, the Causality Group Owner is the ultimate root that started the entire chain, while each subsequent process has its own actor-action relationship.

Extended Best Practices

Operational Excellence

  • Documentation Standards: Maintain comprehensive runbooks for all procedures. Include step-by-step instructions, expected outcomes, and rollback procedures.
  • Change Management: Test all changes in non-production environments first. Document changes and obtain approvals before production deployment.
  • Monitoring Strategy: Implement proactive monitoring with clear thresholds. Set up alerting for critical metrics before they become problems.
  • Regular Reviews: Conduct periodic audits of configurations, performance metrics, and security posture. Schedule quarterly reviews at minimum.
  • Continuous Training: Keep team skills current with platform updates. Allocate time for learning new features and best practices.
  • Knowledge Sharing: Document lessons learned and share across the team. Build institutional knowledge through wikis and team sessions.

Security Hardening

  • Principle of Least Privilege: Grant minimum necessary permissions for each role. Review and revoke unnecessary access regularly.
  • Comprehensive Audit Logging: Enable detailed audit trails for all administrative actions. Retain logs according to compliance requirements.
  • Data Protection: Encrypt sensitive data in transit and at rest. Implement data classification and handling policies.
  • Access Reviews: Conduct regular reviews of user access and permissions. Implement automated access certification workflows.
  • Secrets Management: Use secure vaults for credential storage. Rotate secrets regularly and avoid hardcoding credentials.
  • Network Security: Implement network segmentation and firewall rules. Monitor for unauthorized network access attempts.

Performance Optimization

  • Query Optimization: Review and optimize frequently-used queries. Use indexed fields and appropriate time ranges.
  • Capacity Planning: Monitor resource utilization trends. Plan for growth and scale proactively.
  • Caching Strategies: Implement caching for frequently-accessed data. Configure cache TTLs appropriately.
  • Index Management: Ensure proper indexing for search performance. Review and optimize index configurations.
  • Data Lifecycle: Implement appropriate retention policies. Archive or delete data according to requirements.
  • Load Distribution: Distribute workloads across available resources. Implement load balancing for high availability.

Comprehensive Troubleshooting

Systematic Troubleshooting Methodology

  1. Define the Problem: Clearly articulate what is not working as expected. Gather specific error messages, timestamps, and affected components.
  2. Gather Information: Collect relevant logs, metrics, and configuration details. Document recent changes that might be related.
  3. Analyze Data: Review collected information for patterns and anomalies. Compare against baseline behavior and known-good configurations.
  4. Develop Hypothesis: Form theories about potential root causes. Prioritize based on likelihood and impact.
  5. Test Solutions: Implement fixes in a controlled manner. Test one change at a time to isolate the effective solution.
  6. Document Resolution: Record the root cause and solution. Update runbooks and knowledge base for future reference.

  • Multiple security tools with siloed data
  • Alert fatigue from excessive false positives

  • Phased rollout starting with critical data sources
  • Custom detection rules based on threat model
  • Automated playbooks for common scenarios
  • Integration with ITSM and CMDB systems

Scenario 2: MSSP Multi-Tenant Operations

Organization Profile: Managed security provider serving 100+ SMB customers

Challenges:

  • Need for strict data isolation between customers
  • Scalable operations with limited staff
  • Standardized service delivery across customers
  • Flexible SLA-based prioritization

Implementation Approach:

  • Multi-tenant architecture with customer isolation
  • Shared detection content library with customization
  • Centralized analyst console with customer context
  • Automated reporting and SLA tracking

Results: 3x increase in customers per analyst, consistent service quality, improved customer satisfaction

Challenges:

  • HIPAA compliance requirements for PHI protection
  • Legacy medical devices with limited security logging
  • Need for 6-year log retention
  • Limited IT security staff

Implementation Approach:

  • Comprehensive EHR and clinical application monitoring
  • PHI access detection rules
  • Long-term archival storage configuration
  • Pre-built compliance dashboards and reports

Results: Audit-ready compliance posture, early detection of PHI access violations, reduced audit preparation time by 80%
