Detection Tuning Overview

Effective detection tuning balances security coverage with operational efficiency. The goal is to reduce noise (false positives) while maintaining visibility into actual threats. CrowdStrike provides multiple tuning mechanisms across different detection layers.

  • Maintain Coverage - Preserve detection capability
  • Optimize Performance - Reduce sensor resource usage

  • Stale Exclusions - Old exclusions no longer needed

Exclusion Pattern Syntax

Understanding pattern matching syntax is critical for creating effective exclusions that are neither too broad nor too narrow.

Path Wildcards

PatternMeaningExampleMatches
* Any characters in filename C:\Temp\*.exe All .exe files in C:\Temp
** Any subdirectory depth C:\Users\**\Downloads\* Downloads in all user profiles
? Single character app?.exe app1.exe, appA.exe

Path Pattern Examples

# Specific file - Most precise
C:\Program Files\CustomApp\app.exe
# All executables in directory - Medium precision
C:\Program Files\CustomApp\*.exe
# All files in directory - Lower precision
C:\Program Files\CustomApp\*
# All subdirectories and files - Lowest precision (avoid if possible)
C:\Program Files\CustomApp\**
# User profile paths with wildcard
C:\Users\*\AppData\Local\CustomApp\app.exe
# Multiple file types
C:\CustomApp\*.exe
C:\CustomApp\*.dll
# Environment variable style (NOT supported - use explicit paths)
# WRONG: %PROGRAMFILES%\CustomApp\*
# RIGHT: C:\Program Files\CustomApp\*
#        C:\Program Files (x86)\CustomApp\*

Command Line Pattern Matching

# IOA Command Line Exclusions
# Contains - Match substring anywhere
Operator: Contains
Value: /silent /norestart
Matches: Any command line containing this string
# Starts With - Match beginning
Operator: Starts With
Value: C:\Windows\System32\msiexec.exe /i
Matches: MSI installations
# Regex Pattern - Complex matching (where supported)
Operator: Regex
Value: .*\\msiexec\.exe.*/i.*{[A-F0-9-]{36}}.*
Matches: MSI with product GUID
# Case Sensitivity
- File paths: Case-insensitive on Windows
- Command line arguments: Case-sensitive for regex
- Recommendation: Match case of actual usage

False Positive Analysis

Systematic false positive analysis ensures you create appropriate exclusions based on evidence rather than assumptions.

FP Analysis Workflow

1

Identify Pattern

Find recurring alerts using Event Search analytics

2

Investigate

Examine process tree, command line, parent process

3

Validate Legitimacy

Confirm software is authorized and behavior expected

4

Determine Scope

Identify minimum exclusion scope needed

5

Document & Approve

Record justification and obtain approval

6

Implement

Create exclusion with proper configuration

7

Monitor & Validate

Verify FP rate decreased, no new blind spots

Event Search Queries for FP Analysis

# Top alerting files by count (last 7 days)
#event_simpleName=DetectionSummaryEvent
| stats count by FileName, SHA256HashData, DetectName
| sort -count
| head 50
# Analyze specific detection pattern
#event_simpleName=DetectionSummaryEvent DetectName="SuspiciousPowerShellUsage"
| stats count by ComputerName, FileName, CommandLine, UserName
| sort -count

| where HostCount > 10
| sort -count
# ML detection analysis

Q: How do you approach tuning detections to reduce false positives while maintaining security coverage?

A: I follow a systematic process. First, I use Event Search to identify the highest-volume detections and analyze patterns - looking at file paths, process relationships, and affected hosts. For each potential FP, I validate that the software is authorized and the behavior is expected. I create the most specific exclusion possible - preferring SHA256 hashes over paths, and single rules over global exclusions. Every exclusion requires documentation with business justification and approval. After implementation, I monitor to verify the FP rate decreased. I conduct quarterly reviews to remove obsolete exclusions and update hashes for new software versions. The goal is minimal exclusions with maximum specificity.

Q: What's your strategy for enabling prevention (blocking) mode safely?

A: I use a phased rollout approach. Start with detect-only mode for 2-4 weeks to collect baseline data. Analyze detections and create necessary exclusions for legitimate software - SCCM, development tools, backup agents are common culprits. Then enable blocking for low-FP-risk categories first: ransomware and credential theft have low false positive rates and high impact. Next, enable ML blocking at moderate sensitivity. Finally, enable script-based and exploit prevention after application compatibility testing. Throughout, I use different policies for different host types - servers get more conservative settings than workstations. I maintain a rollback plan and monitor closely after each change.

Q: How do you handle a situation where a critical application is being blocked by CrowdStrike?

A: Immediate response: If production is impacted, I'd first verify the alert isn't a true positive - even legitimate applications can be compromised. Assuming it's a false positive, I create a hash-based exclusion for the specific file version as a quick fix. For longer-term resolution, I investigate the detection - what triggered it, is there a less-risky exclusion approach. I work with the application team to understand the behavior pattern. If it's a vendor issue, I report it to CrowdStrike as a potential ML false positive. I document the incident, create proper exclusions with approval, and add the application to monitoring for future version updates. I also evaluate if policy settings are too aggressive for that host type.

Q: What are the risks of over-exclusion and how do you mitigate them?

A: Over-exclusion creates blind spots attackers can exploit. Adversaries specifically target commonly-excluded paths like C:\Windows\Temp or backup software directories. Mitigation strategies: Use hash-based exclusions instead of path-based where possible since attackers can't easily replicate legitimate file hashes. Apply exclusions to the smallest scope - specific host groups rather than global. Document everything and require approval. Set expiration dates on temporary exclusions. Conduct quarterly reviews to remove unneeded exclusions. Monitor excluded paths with other controls if possible. Never create sensor visibility exclusions without exhausting other options. The principle is: exclude the minimum necessary to resolve the false positive.

Q: How do you measure detection tuning effectiveness?

A: Key metrics I track: False positive rate - percentage of alerts closed as FP, target under 5%. Alert volume trend - should be manageable for the team size. Mean time to triage - tuning should reduce investigation time. Prevention block rate - percentage of true threats blocked vs detected. Exclusion count - should be minimal and justified. I also track coverage using MITRE ATT&CK mapping to ensure tuning hasn't created gaps in technique detection. I create dashboards showing these metrics over time and review them weekly. Sudden increases in any metric trigger investigation - new software deployed, configuration change, or potential attack. The goal is optimal signal-to-noise ratio.
