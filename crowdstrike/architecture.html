Key Design Decisions

  • Host groups by risk: Trading floor, executives, and standard users have different policies
  • Prevention tuning: Trading apps have exclusions to prevent false positives during market hours
  • RTR tiered access: Full RTR for IT, respond-only for SOC, read-only for helpdesk
  • Overwatch priority: Executive endpoints flagged for priority MDR attention
  • Container protection: DaemonSet deployment for EKS with runtime protection

Interview Questions

A: "The Threat Graph is CrowdStrike's cloud-native graph database that correlates over a trillion events weekly from all Falcon customers. It's important because it enables: (1) Cross-customer threat intelligence - if an attack is seen at one customer, all customers benefit immediately; (2) Behavioral pattern detection - can identify attack chains even with never-before-seen malware; (3) Real-time correlation - processes, files, network connections, and user activity are linked as nodes in a graph, making it easy to trace lateral movement; (4) Indicators of Attack (IOAs) - behavior-based detection rather than just IOC hash matching. This is why CrowdStrike can detect fileless attacks and living-off-the-land techniques that signature-based AV misses."

A: "IOCs (Indicators of Compromise) are atomic indicators - file hashes, IP addresses, domain names. They're reactive - you need to see the threat first to create the signature. IOAs (Indicators of Attack) are behavioral patterns - 'PowerShell spawned from Word document, made network connection, then dropped executable.' IOAs detect the behavior regardless of the specific malware. CrowdStrike prioritizes IOAs because attackers can easily change file hashes but the attack technique stays the same. For example, a credential theft IOA might look for: lsass.exe memory access + unusual parent process + credential dump tool behavior, regardless of whether it's Mimikatz, a custom tool, or a new variant."

A: "For large-scale deployment: (1) Use existing tools - SCCM, Intune, GPO, or your MDM for macOS. CrowdStrike provides MSI/PKG packages with silent install options; (2) Pre-create host groups and assign sensors to groups using installation tokens or tags; (3) Start with detect-only mode to identify false positives before enabling prevention; (4) Phase the rollout - IT first, then low-risk groups, then high-risk (trading floor last); (5) Create exclusions proactively for known applications; (6) Monitor the Sensor Health dashboard for connectivity issues; (7) Use the API for deployment tracking and compliance reporting. For containerized environments, deploy as DaemonSet in Kubernetes with appropriate resource limits."
