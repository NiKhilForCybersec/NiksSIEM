Alert Lifecycle States: NEW │ Alert created, awaiting triage │ ▼ UNDER_INVESTIGATION │ Analyst actively working on alert │ ├──► RESOLVED_TRUE_POSITIVE │ Confirmed malicious, incident created │ ├──► RESOLVED_FALSE_POSITIVE │ Confirmed benign, rule tuning needed │ ├──► RESOLVED_DUPLICATE │ Same as existing alert/incident │ └──► RESOLVED_OTHER Other resolution (informational, etc.)

Causality Chain Analysis

The Causality Chain shows the process tree and relationships leading to the alert.

Alert Details → Causality View

Example Causality Chain: explorer.exe (User shell) │ └── outlook.exe (Email client) │ └── WINWORD.EXE (Opened attachment) │ └── cmd.exe (Macro execution) │ └── powershell.exe ← ALERT TRIGGERED │ ├── Network: 185.x.x.x:443 └── File: C:\Users\...\payload.exe Key Analysis Points: 1. What initiated the chain? (explorer = user action) 2. What's the attack vector? (Word document from email) 3. What actions did the alert process take? 4. Is there C2 communication? 5. Were additional payloads dropped?

Causality Investigation Steps

  1. Identify the Causality Group Owner (CGO) - top of chain
  2. Trace the execution path to the alert
  3. Examine child processes of the alert process
  4. Check network connections at each stage
  5. Look for file drops and persistence

Response Actions

ActionWhen to UseImpact
Isolate Endpoint Confirmed compromise, active threat Blocks all network except XSIAM
Kill Process Malicious process running Terminates specific process
Quarantine File Malicious file identified Moves file to quarantine
Block Hash Known malicious file Prevents execution globally
Run Script Custom remediation needed Execute remediation script
Collect Package Need forensic data Gathers logs and artifacts

Response Action Paths

Isolate Endpoint: Alert Details → Host → Actions → Isolate Block Hash: Alert Details → File → Actions → Add to Block List Run Script: Alert Details → Host → Actions → Run Script Select script → Set parameters → Execute Collect Investigation Package: Alert Details → Host → Actions → Collect Investigation Package Wait for collection → Download package

Alert Resolution

Resolution Types

ResolutionWhen to UseFollow-up Actions
True Positive Confirmed malicious activity Create incident, containment actions
False Positive Benign activity, detection error Add exclusion, tune rule
Duplicate Already covered by another alert Link to existing incident
Informational Legitimate but worth noting Document, no action needed

Resolution Documentation

Good Resolution Note Example: "Alert investigated and confirmed as false positive. - Process: PowerShell script is part of SCCM deployment - Parent: sccm-agent.exe (legitimate) - Network: Connection to internal SCCM server (10.0.5.20) - User: SYSTEM account (expected for SCCM) - Action: Added exclusion for SCCM PowerShell executions - Exclusion ID: EXC-2024-001"

Managing Exclusions

Settings → Exclusions → Create Exclusion

Exclusion Types

Exclusion Scopes: Process Path Exclusion: - Exclude specific executable paths - Example: C:\Program Files\App\legitimate.exe Process + Command Line: - Exclude based on process and arguments - Example: powershell.exe with -File "C:\Scripts\approved.ps1" Hash Exclusion: - Exclude specific file hashes - Use for known-good files Signer Exclusion: - Exclude by code signing certificate - Example: Signed by "Microsoft Corporation" Host Exclusion: - Exclude specific hosts from rule - Use for test/lab systems

Exclusion Best Practices

  • Be specific: Narrow exclusions to minimize gaps
  • Document reason: Explain why exclusion was added
  • Track in tickets: Reference FP ticket number
  • Review regularly: Audit exclusions quarterly
  • Test impact: Verify exclusion works as expected

Creating Incidents

When to Create Incident

  • Confirmed true positive requiring investigation
  • Multiple related alerts indicating attack
  • Activity requiring response actions
  • Need to track remediation progress

Incident Creation Options

From Single Alert: Alert Details → Actions → Create Incident - Auto-links alert to new incident - Carries over alert details From Multiple Alerts: Alert Queue → Select alerts → Link to Incident - Create new incident - OR link to existing incident Automatic Incident Creation: Configured in detection rules - Auto-create on high-severity alerts - Auto-group related alerts

XQL for Alert Metrics

// Alerts by severity over time dataset = xdr_alerts | filter insert_timestamp >= now() - 7d | comp count() by severity, bin(insert_timestamp, 1d) // False positive rate by rule dataset = xdr_alerts | filter insert_timestamp >= now() - 30d | comp count() as total, count(resolution = "RESOLVED_FALSE_POSITIVE") as false_positives by alert_name | alter fp_rate = round(false_positives * 100.0 / total, 2) | sort desc fp_rate // Mean time to close dataset = xdr_alerts | filter resolution_timestamp != null | alter close_time_hours = timestamp_diff(resolution_timestamp, insert_timestamp, "HOUR") | comp avg(close_time_hours) as avg_close_hours by alert_name

Interview Questions

  1. Severity: Critical and High first
  2. Asset criticality: Servers before workstations
  3. Alert source: High-fidelity sources first
  4. Related activity: Multiple alerts on same host
  5. Time sensitivity: Active threats over old alerts
  1. Check command line: Is it encoded? What does it do?
  2. Review parent process: How was PowerShell launched?
  3. Check user context: Expected for this user?
  4. Network connections: Any external communication?
  5. File operations: Did it drop any files?
  6. Historical: First time on this host/network?

  • Tune detection rules: Add conditions to reduce noise
