Digital Forensics

Essential digital forensics techniques for SOC analysts. Learn evidence collection, artifact analysis, and investigation procedures to support incident response.

Forensics Fundamentals

Forensic Principles

Core Forensic Principles: 1. PRESERVE EVIDENCE - Don't modify original data - Work on copies when possible - Document all actions 2. CHAIN OF CUSTODY - Track who handled evidence - Document transfers - Maintain integrity 3. DOCUMENTATION - Record every action - Screenshot findings - Note timestamps 4. REPRODUCIBILITY - Steps should be repeatable - Use validated tools - Document methodology
⚠️ Important: In legal/HR investigations, coordinate with Legal before collecting evidence. Improper collection can make evidence inadmissible.

Evidence Collection

Order of Volatility

Collect in Order of Volatility (Most → Least): 1. CPU registers, cache 2. Memory (RAM) ← Collect FIRST 3. Network state 4. Running processes 5. Disk contents 6. Remote logging 7. Physical configuration 8. Archival media Memory is CRITICAL - lost on power off!

Memory Analysis with Volatility

# Volatility 3 Commands # Identify OS profile vol -f memory.raw windows.info # List processes vol -f memory.raw windows.pslist vol -f memory.raw windows.pstree # Tree view # Find hidden/injected processes vol -f memory.raw windows.malfind # Network connections vol -f memory.raw windows.netscan # Command line arguments vol -f memory.raw windows.cmdline # DLLs loaded by process vol -f memory.raw windows.dlllist --pid 1234 # Registry analysis vol -f memory.raw windows.registry.hivelist # Extract files from memory vol -f memory.raw windows.filescan vol -f memory.raw windows.dumpfiles --pid 1234

What to Look For in Memory

Memory Analysis Checklist: SUSPICIOUS PROCESSES: □ Unknown process names □ Processes with no parent □ Duplicate system processes □ Processes in wrong locations INJECTED CODE: □ malfind results (RWX memory) □ Hollowed processes □ Injected DLLs NETWORK: □ Unexpected connections □ Connections to bad IPs □ Hidden connections PERSISTENCE: □ Registry run keys in memory □ Service configurations □ Scheduled tasks

Timeline Analysis

Building a Timeline

Timeline Sources: FILE SYSTEM: - $MFT timestamps (MACB) - USN Journal changes - $LogFile entries REGISTRY: - Last write times - UserAssist (ROT13 encoded) EVENT LOGS: - Security events - System events - Application events APPLICATION: - Browser history - Email timestamps - Recent documents EDR/SIEM: - Process events - Network connections - Alert timestamps

MACB Timestamps

NTFS Timestamps: M - Modified: Content changed A - Accessed: File opened (may be disabled) C - Changed: Metadata changed ($MFT updated) B - Birth: File created $STANDARD_INFORMATION (user-visible): - Easily modified by attackers - Shown in Explorer $FILE_NAME (hidden): - Harder to modify - More reliable for forensics Timestomping Detection: - Compare $SI and $FN timestamps - $FN Born > $SI Modified = suspicious

Interview Questions

Order of volatility ranks evidence by how quickly it disappears:

  1. CPU registers/cache (nanoseconds)
  2. RAM/Memory (power dependent)
  3. Network state (seconds)
  4. Running processes (changes constantly)
  5. Disk (persistent but modifiable)

Why it matters: Collect most volatile evidence first. Memory contains processes, credentials, and malware that are lost when powered off.

  • Compare $STANDARD_INFORMATION and $FILE_NAME timestamps
  • $FN Born time > $SI Modified = suspicious
  • Timestamps before OS installation = suspicious
  • Use tools like MFTECmd to parse both attributes

Attackers modify $SI timestamps but $FN is harder to change.

  • Prefetch: Run count, last 8 execution times
  • Amcache: SHA1 hash, first execution
  • ShimCache: Execution flag, timestamps
  • UserAssist: GUI program execution count
  • BAM/DAM: Background Activity Moderator
  • SRUM: Resource usage per application
