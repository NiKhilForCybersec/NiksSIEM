Alert Triage Methodology

A systematic approach to efficiently evaluate, prioritize, and disposition security alerts. Effective triage is critical for managing alert volume while ensuring real threats are not missed.

What is Alert Triage?

Alert triage is the process of rapidly assessing incoming security alerts to determine:

  • Validity: Is this a true positive or false positive?
  • Severity: How critical is this if confirmed?
  • Priority: How urgently does this need attention?
  • Action: What should happen next?

Triage Goals

  • Minimize Mean Time to Detect (MTTD)
  • Maximize true positive identification
  • Reduce analyst fatigue from false positives
  • Ensure high-priority alerts get immediate attention
  • Create consistent, repeatable decisions

5-Step Triage Process

1

Initial Assessment (30 seconds)

Read alert name, severity, affected asset. Does this look legitimate at first glance? Check if similar alerts are already being investigated.

2

Context Gathering (2-5 minutes)

Who/what triggered the alert? What's the asset criticality? What's the user's role? Is this during business hours? Any recent changes?

3

Enrichment & Correlation (2-5 minutes)

Check threat intel for IOCs. Look for related alerts. Query additional data sources. Check historical activity for this entity.

4

Decision (1 minute)

Based on evidence: True Positive, False Positive, or Needs Investigation. Assign priority if escalating.

5

Document & Disposition (1-2 minutes)

Record findings in ticket. Close if FP or escalate if TP. Identify tuning opportunities.

Key Triage Questions

About the Alert

  • What exactly triggered this alert?
  • What detection rule/signature fired?
  • What's the alert severity and confidence level?
  • What MITRE ATT&CK technique is this mapped to?

About the Entity

  • What asset triggered this? (endpoint, server, user)
  • What's the asset's business criticality?
  • Who is the user? What's their role?
  • Is this expected behavior for this entity?

About the Context

  • When did this occur? Business hours?
  • Are there related alerts around the same time?
  • Any change requests or maintenance windows?
  • Has this entity triggered similar alerts before?

About the Indicators

  • Are there known-bad IOCs (hashes, IPs, domains)?
  • Does threat intel provide context?
  • What's the reputation of external IPs/domains?

Alert-Type Specific Triage

Malware Detection Alert

Questions to answer: 1. What malware was detected? (name, family, type) 2. Was it blocked or just detected? 3. What was the delivery mechanism? (email, web, USB) 4. Did it execute or attempt to execute? 5. What's the malware's capability? (RAT, ransomware, infostealer) 6. Are there other hosts with the same hash? Key lookups: - Hash reputation (VirusTotal, internal threat intel) - File path (suspicious locations?) - Process tree (how did it get there?) - Network connections (C2 communication?)

Authentication Alert (Brute Force, Failed Logins)

Questions to answer: 1. What account is being targeted? 2. What's the source IP? Internal or external? 3. How many failures? Over what time period? 4. Was there a subsequent success? 5. Is this a service account or user account? 6. Any other accounts targeted from same source? Key lookups: - User's normal login patterns - Source IP reputation and geolocation - Other activity from this source - Account privilege level

Suspicious Process Execution

Questions to answer: 1. What process executed? Is it normally seen in environment? 2. What's the parent process? (unexpected parents are suspicious) 3. What's the command line? (encoded commands, suspicious args) 4. What user ran it? Expected for their role? 5. What network connections did it make? 6. What files did it create/modify? Key lookups: - Process reputation - Normal usage patterns in environment - MITRE ATT&CK mapping - Similar executions on other hosts

Data Exfiltration Alert

Questions to answer: 1. What data is potentially being exfiltrated? 2. What's the destination? (cloud storage, external IP) 3. How much data? Is this unusual for this user/system? 4. What application/process is transferring data? 5. Is this authorized? (approved cloud service, backup) 6. What's the user's role? Access to sensitive data? Key lookups: - User's normal data transfer patterns - Destination reputation - DLP policy violations - Recent file access activity

Lateral Movement Alert

Questions to answer: 1. What type of lateral movement? (RDP, SMB, WMI, PsExec) 2. Source and destination systems? 3. What account is being used? 4. Is this expected admin activity? 5. What happened after the lateral movement? 6. Are there multiple destination systems? Key lookups: - Account's normal behavior - Change request for admin activity - Destination system criticality - Activity on destination systems

Common False Positive Patterns

Alert TypeCommon FP CausesHow to Verify

Suspicious PowerShell SCCM deployments, admin scripts, monitoring tools Check parent process, verify with IT ops

Platform-Specific Triage Queries

Splunk - Get Alert Context

// Get user's recent activity index=* user="$SUSPECT_USER$" earliest=-24h | stats count by index, sourcetype, action | sort - count // Get host's process activity index=endpoint host="$SUSPECT_HOST$" earliest=-1h | table _time, process_name, process_command_line, parent_process_name, user // Get network connections for host index=firewall src_ip="$SUSPECT_IP$" OR dest_ip="$SUSPECT_IP$" earliest=-1h | table _time, src_ip, dest_ip, dest_port, action, bytes

Microsoft Sentinel - Get Alert Context

// Get user's recent activity union SigninLogs, AuditLogs, SecurityEvent | where TimeGenerated > ago(24h) | where UserPrincipalName == "$SUSPECT_USER$" or Account == "$SUSPECT_USER$" | summarize count() by Type, bin(TimeGenerated, 1h) // Get host's process activity DeviceProcessEvents | where TimeGenerated > ago(1h) | where DeviceName == "$SUSPECT_HOST$" | project TimeGenerated, FileName, ProcessCommandLine, InitiatingProcessFileName // Get related alerts SecurityAlert | where TimeGenerated > ago(24h) | where Entities contains "$SUSPECT_HOST$" or Entities contains "$SUSPECT_USER$"

Interview Questions

  1. Initial Assessment: Note malware name, affected host, detection action (blocked vs detected)
  2. Context: Check host criticality, user role, was this expected (pen test, admin tool)?
  3. Enrichment: Hash reputation lookup, check process tree, look for related network activity
  4. Key Questions:
    • How did the malware get there? (email, web, USB)
    • Did it execute or just land on disk?
    • Any C2 communication?
    • Same hash on other hosts?
  5. Decision: If blocked and no execution, likely contained. If executed, escalate to IR.
  1. Check for active attacks: Ransomware, active C2, data exfiltration get immediate attention
  2. Asset criticality: Critical business systems before workstations
  3. Alert severity: Critical > High > Medium > Low
  4. Time sensitivity: Some alerts become stale (failed logins from hours ago)
  5. Group related alerts: Multiple alerts on same host = potential incident

Formula: Priority = Severity × Asset Criticality × Time Sensitivity

  • Track false positives: Document every FP with root cause
  • Submit tuning requests: Work with detection engineering to improve rules
  • Automate enrichment: Use SOAR to pre-populate context
  • Create runbooks: Standard procedures reduce cognitive load
  • Rotate analysts: Mix high-volume and low-volume alerts
  • Review metrics: Track FP rate by rule, prioritize tuning
  • Pattern recognition: Quickly identify normal vs abnormal
  • Technical knowledge: Understanding of attacks, protocols, systems
  • Tool proficiency: Efficient use of SIEM, EDR, threat intel
  • Attention to detail: Catch subtle indicators
  • Time management: Know when to escalate vs dig deeper
  • Documentation: Clear, thorough notes for handoffs
  • Continuous learning: Stay current on threats and TTPs
