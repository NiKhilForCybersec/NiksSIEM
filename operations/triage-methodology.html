Alert Triage

Systematic approach to evaluating security alerts. Learn to quickly assess validity, severity, and required response for each alert.

Triage Process

1

Read the Alert

Understand what triggered: rule name, description, severity, affected entities

2

Check Context

Who/what is affected? Asset criticality? User role? Time of activity?

3

Gather Evidence

Review raw logs, related events, historical activity for entities

4

Make Decision

True positive? False positive? Needs more investigation?

5

Take Action

Close FP, escalate TP, or continue investigation as needed

Triage Questions

WHO

Which user? Service account? Admin? Executive? Expected for this activity?

WHAT

Which system? Server or workstation? Critical asset? What action occurred?

WHEN

Business hours? After hours? Weekend? During maintenance window?

WHERE

Source IP/location? Internal or external? Expected geography?

Quick Validation Checks

Common Triage Checks
□ Is this a known false positive pattern?
□ Is the user/system on an exception list?
□ Was there a change request or maintenance scheduled?
□ Does the user's job function explain the activity?
□ Is this a test/lab system?
□ Are there related alerts for the same entities?
□ What's the historical baseline for this activity?
□ Does threat intel support this as malicious?
□ Can I correlate with other data sources?
□ What would an attacker gain from this action?

Interview Questions

  1. Context check: Does the activity make sense for this user/system?
  2. Historical baseline: Is this normal behavior for this entity?
  3. Known patterns: Does this match documented FP patterns?
  4. Correlation: Are there supporting indicators of malice?
  5. Verification: Can you confirm with the user or system owner?

Document your reasoning - if the same FP occurs repeatedly, it should be tuned.
